## 1. Day 6：通知、消息队列与监控告警

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容


- 主题：异步解耦、重试幂等、监控指标与告警体系

1. 先定范围

支持 `iOS/Android Push + SMS + Email`，软实时，支持用户退订，量级示例是日千万级推送。

1. 高层架构从“单机”到“解耦”

初版有单点和性能瓶颈，改进版引入 `多通知服务器 + 缓存/DB拆分 + 按渠道队列 + workers + 第三方通道`。

1. 核心链路

业务服务调通知 API → 通知服务做校验和模板渲染 → 写队列 → worker 消费 → 调 APNS/FCM/Twilio/SendGrid。

1. 可靠性设计

通知数据持久化（日志表），失败重试，去重（事件ID），接受“至少一次”而非幻想“绝对恰好一次”。

1. 运营能力

模板、用户通知偏好、速率限制、认证安全、队列监控、打开/点击事件追踪。

**面试怎么回答（结构）**

1. 先讲目标

“这是多渠道异步通知平台，目标是高可用、可扩展、可观测，不丢通知。”

1. 讲架构主干

“API层、通知编排层、分渠道消息队列、worker层、第三方通道层。”

1. 讲关键取舍

“采用 MQ 做削峰解耦；语义上至少一次投递，靠幂等去重；按渠道隔离，避免单通道故障扩散。”

1. 讲治理闭环

“用户偏好检查、限频、失败重试告警、事件追踪和队列积压监控。”

**你可以直接背的 60-90 秒版本**

“我会把通知系统设计成多渠道异步平台，支持 Push、短信和邮件。入口由通知 API 接收请求，做参数校验、用户偏好判断和模板渲染后，把通知事件写入按渠道拆分的消息队列。各渠道 worker 独立消费并调用第三方通道，例如 APNS、FCM、短信和邮件服务。这样做的核心价值是解耦和隔离，某个渠道故障不会拖垮全链路。可靠性上采用通知日志持久化、失败重试和事件ID去重，语义上接受至少一次并通过幂等控制重复。治理上补充限频、认证、队列积压监控和打开点击追踪，形成可观测可运营的闭环。”

### 分布式系统链路监控

链路监控的核心目标是三件事：
- 调用链路可视化；
- 链路性能监控；
- 链路故障排查。

常见实现思路是做一个统一的链路采集框架，让每一次服务调用都经过这个框架：
- 采集调用的起点、终点、耗时、异常信息；
- 把这些数据持久化；
- 再通过可视化界面展示整条调用链路、各节点耗时和异常点。

常见链路追踪体系包括：
- Google `Dapper`
- 阿里 `鹰眼`
- 大众点评 `CAT`
- Twitter `Zipkin`
- `SkyWalking`

面试里可以补一句：
**链路监控不是单纯记日志，而是要能把一次跨服务请求完整串起来，回答“这次请求经过了哪些服务、卡在了哪一跳、为什么失败”。**
