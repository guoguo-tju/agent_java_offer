## 1. Day 3：一致性哈希与 KV 存储

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容


主题：分片、重平衡、读写路径、复制与故障恢复

**1. 一致性哈希与虚拟节点（先讲分片）**

1. 把哈希空间看成一个环（0 到 2^32-1）。
2. 节点（机器）和 key 都映射到环上。
3. key 顺时针找到第一个节点作为归属节点。
4. 扩容/缩容时，只迁移受影响区间的数据，不会全量搬迁。
5. 虚拟节点（vnodes）做法：每台物理机映射多个点到环上。
6. 价值：数据更均匀、热点更少、节点增减更平滑。
7. 进一步可加“加权虚拟节点”：高配机器分配更多 vnode。

**2. 高可用 KV（再讲写入、复制、读取）**

1. 数据模型`key -> value + version + timestamp`。
2. 分区：用一致性哈希决定 key 所在分区。
3. 复制：每个 key 写到 N 个副本（比如主副本+后续副本）。
4. 写路径：客户端到协调节点，协调节点并行写 N 副本，达到 W 个成功即返回。
5. 读路径：协调节点读 N 副本中的 R 个，做版本合并后返回最新值。
6. 常见条件`R + W > N` 可提高读写一致性概率。
7. 故障处理：副本宕机时先写可用副本（hinted handoff），恢复后补数据。
8. 后台修复：anti-entropy（如 Merkle tree）+ read repair。

**3. 可用性与一致性取舍（追问高频）**

1. 追求强一致（偏 CP）：写入要多数派确认，网络分区时宁可拒写，保证不读脏数据。
2. 追求高可用（偏 AP）：优先接受读写请求，允许短暂不一致，后续异步收敛。
3. 场景决策：

- 交易、余额、库存扣减：优先一致性。  
- 缓存、Feed、推荐、计数：优先可用性和低延迟。

**4. 你可以直接背的面试口述（90秒）**

“我会先用一致性哈希做分片，节点和 key 都映射到环上，key 顺时针落到第一个节点。为避免数据倾斜，每台机器配置多个虚拟节点。这样扩缩容时只迁移局部区间。  

在 KV 高可用上，我采用 N 副本复制，写入走协调节点并行写副本，达到 W 个确认返回；读取拉 R 个副本并做版本合并，结合 read repair 保证收敛。  

一致性和可用性的取舍上，我按业务分级：交易链路用更强一致策略，内容链路用最终一致换低延迟和高可用。核心是把取舍显式化，并用监控和修复机制兜底。”
