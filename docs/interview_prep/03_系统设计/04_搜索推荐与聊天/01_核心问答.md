  - 追问准备：“倒排、分词、召回、排序如何拆层？”
  - 
- 爬虫系统: 
  - 爬虫系统的本质，是一个带调度能力的数据采集系统
  - 核心链路是 Seed URL 进入 Frontier调度队列，Downloader 拉取页面，Parser 解析内容和新链接，再经过 URL/内容去重、过滤，回到 Frontier。
  - 其中最关键的是 URL Frontier。它不是简单 FIFO，而是整个调度中心，要同时解决三个问题：优先级、礼貌性和新鲜度。优先级上，我会按页面重要性和更新频率来安排抓取顺序；礼貌性上，会按 host 维度拆队列，同一 host 串行抓取，并加抓取间隔；新鲜度上，对更新频繁的重要页面提高重抓频率。  
  另外系统还要重点处理去重、DNS 缓存、robots.txt、失败恢复和 spider trap。总结来说，爬虫系统最核心的不是‘抓得快’，而是‘调度得稳、过滤得准、恢复得动’
  - 去重: 
    - URL去重: 防止重复 URL 进入 Frontier; 常用 Bloom Filter / HashSet / Redis Set
    - 内容去重: 防止不同 URL 对应相同内容, 常用哈希值、checksum
- 搜索系统:

搜索系统主链路可以压缩成四个字：**召回、粗排、精排、展现**。

1. 写入链路（离线/准实时）
  - 文档采集进来
  - 解析、清洗、分词、抽字段
  - 建倒排索引
  - 生成排序特征
  - 索引写入搜索引擎

这条链路是“把数据变成可搜索”。

1. 查询链路（在线）

用户输入 query 后：

- Query 理解：分词、纠错、同义词扩展
- 召回：从倒排索引快速拿到候选文档
- 排序：按 BM25/相关性模型/点击率模型综合排序
- 返回前做过滤、去重、高亮、分页  
这条链路是“把 query 变成结果页”。

1. 召回怎么理解

召回阶段的目标不是最精准，而是**别漏掉可能相关的候选**。  
常见方式：

- 倒排索引召回
- 同义词扩召回
- 拼写纠错召回
- 向量召回（语义检索）  
搜索系统里，召回更看覆盖率和延迟。

1. 排序怎么理解

排序阶段才真正决定“谁排前面”。  
常见排序信号：

- 文本相关性
- 热度
- 点击率/转化率
- 时效性
- 个性化特征  
可以分粗排和精排：  
粗排先快速筛一遍，精排再用更贵的模型细算。

1. 自动补全怎么讲

自动补全和全文搜索不完全一样。  
它更强调**前缀匹配 + 高频词优先 + 低延迟**。  
常见实现：

- Trie / 前缀树
- 倒排 + 前缀索引
- Redis ZSet 存热点 query  
排序通常结合：前缀命中、历史频次、时效性、个性化。

## 2. Day 7：信息流与聊天系统

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容


- 主题：Feed 推拉模型、在线状态、消息存储与同步

信息流系统设计，按“朋友圈/微博首页”这个场景，用最通俗的方式讲。

**先定目标**

1. 用户发一条动态后，粉丝能较快看到。
2. 首页打开要快。
3. 高峰期不能崩。
4. 允许“几秒内慢慢同步”，但不能乱。

**系统分成5个角色**

1. 发帖服务：负责接收“发动态”。
2. 关系服务：知道“谁关注了谁”。
3. 消息队列：做缓冲，防止瞬间流量把系统打挂。
4. 推送工人：从队列取任务，分发到各粉丝首页。
5. 首页服务：用户打开首页时，快速拿到内容。

---

**一、发帖流程（写入）**

1. 用户发帖，先把帖子存到帖子库。
2. 生成一条“分发任务”放入消息队列。
3. 推送工人拿到任务，查粉丝列表。
4. 把“这条帖子编号”写进粉丝的首页列表。
5. 粉丝下次打开首页，就能直接看到。

这就是“写扩散”：发帖时先分发，读的时候更快。

---

**二、看首页流程（读取）**

1. 用户打开首页。
2. 首页服务先读“我的首页列表”（只存帖子编号）。
3. 再根据编号批量取帖子正文、作者信息、点赞数。
4. 组合后返回给前端。

这里“首页列表只存编号”很关键，省内存、速度快。

---

**三、为什么还要读扩散**

如果是普通用户，粉丝不多，写扩散很合适。  

但如果是大V，发一条要写几百万粉丝首页，写入压力太大。

所以大V用“读扩散”：

1. 发帖时先不全量分发。
2. 粉丝打开首页时，再临时拉取并合并大V内容。

**最终做法：混合**

1. 普通用户：写扩散（读快）。
2. 大V用户：读扩散（写稳）。

---

**四、怎么保证稳定**

1. 幂等：同一条分发任务重复执行，也只写一次。
2. 重试：第三方或下游失败，自动重试。
3. 死信队列：多次失败先隔离，避免堵主链路。
4. 监控：重点看“队列堆积、首页延迟、分发成功率”。

---

**面试可直接说（60秒）**

“我会把信息流推送拆成发帖链路和首页读取链路。发帖先落库，再把分发任务写入消息队列，由分发服务异步把帖子编号写进粉丝首页列表；用户读首页时先拿编号，再批量取正文和作者信息。核心取舍是写扩散和读扩散：普通用户走写扩散保证首页快，大V走读扩散避免写放大，采用混合模式。系统通过幂等、重试、失败隔离和监控保证高并发下稳定运行。”

如何设计一个聊天系统?

先确认范围：支持 1 对 1、最多 100 人群聊、在线状态、多端登录、离线推送。  

通信层(客户端和后台)我会选 WebSocket 做长连接，因为聊天是双向实时场景。登录注册等非实时接口走 HTTP 即可。  

架构上分成：

聊天服务. 聊天服务是有状态的，因为要websocket维护长连接；通过服务发现给用户分配最合适的聊天节点，避免单机过载

在线状态服务. 5秒钟的这个心跳机制来判断这个用户是在线还是离线。对，这个也不要太实时，不然在地铁就信号抖动的地方会频繁上下线，对用户体验不好。

存储层.  聊天消息我更倾向 KV 存储，原因是写入量大、低延迟要求高、可水平扩展。消息主键不能只用时间戳，我会引入可排序的 message_id，保证同会话内顺序。用户关系方面，用户资料和关系链可放关系库；

通知服务. 就是去发送消息，在线通 知直接将在线这条消息同步给接收方对应的那个 聊天chat 节点。离线通知就给它写到离线通知的 kv 存储里, 再去调用APS 或者 Firebase 这些原生的push接口通知

流程是：

双方在线:  发送方写入聊天服务，生成 message_id，先落消息存储，再投递给接收方在线节点；

如果接收方离线，Worker -> 离线推送服务: 发离线推送.  B上线后 -> 拉取增量消息(message_id > 本地最大ID)

多端同步, 用每台设备自己的 cur_max_message_id 拉增量，保证手机和电脑最终一致。  

群聊方面，小群可以给每个成员维护收件箱副本，读取简单、延迟低(写扩散)；大群则要避免全量复制，可改为按需拉取或分层广播 (读扩散)。最后补上重试、死信、幂等和监控，保证消息不丢、可追踪、可恢复。

### 原文内容

#### 2.5.1 ES基础概念

ES 术语：
- `Index`：相当于数据库的 Table
- `Document`：相当于数据库的一行记录
- `Field`：相当于数据库的 Column
- `Mapping`：相当于数据库的 Schema
- `DSL`：相当于数据库的 SQL
- 中文分词器常用 IK

倒排索引：
- key 存储：ES 还有一层 `Term Index`，只存储部分词前缀，并常驻内存；在内存中通常以 `FST` 形式保存，节省内存。
- value 存储：`PostingList` 会用 `FOR` 压缩文档 ID，并用 `Roaring Bitmaps` 支持交并集运算。

#### 2.5.2 ES数据写入与查询

ES 数据写入：
- 协调节点负责路由。
- 数据先写内存缓冲区，每隔 1s 刷到文件系统缓存，所以 Query 是近实时的。
- 为防节点宕机，ES 还会写 `translog`，但 translog 落盘也有时间窗口，因此节点异常时存在短暂数据丢失风险。
- 主分片写成功后并行复制到副本，副本完成后返回 ack。

ES 查询：
- `Get`（按 ID 查）是实时的，会先查 translog；
- `Query`（按 query 匹配）是近实时的，主要查 segment。
- `QUERY_THEN_FETCH` 分两阶段：
  - Query Phase：每个分片先返回局部 topN doc id；
  - Fetch Phase：协调节点合并排序后，再向目标分片拉取完整文档。

#### 2.5.4 ES深分页

ES 深分页的问题本质是：每个分片都要先返回 `from + size` 条候选，再由协调节点做全局合并排序，页数越深，排序和内存成本越高。

常见解法：
- 业务上限制最大页数；
- `from/size` 只用于浅分页；
- 批处理任务用 `scroll`；
- 实时高并发深翻页用 `search_after`，用上一页最后一条记录作为游标。

### 微信朋友圈这样的社交系统，应该如何设计？

一个比较通俗的拆法是：
1. **发布链路**
- 用户点击发朋友圈时，客户端先把内容缓存到本地，立刻给用户成功反馈；
- 图片/视频先就近上传到 CDN；
- 客户端把图片地址、文字、可见范围等元数据提交给后端；
- 后端写入“朋友圈发布表”。

2. **索引层 / 相册层**
- 发布表写成功后，再往相册表写一条索引，指向真正的发布内容；
- 这样用户查看自己历史内容时，可以先走索引，再回源到发布表。

3. **时间线分发**
- 后台离线任务或异步任务，根据可见范围，把这条朋友圈写入好友的时间线表；
- 好友刷朋友圈时，本质上查的是时间线表，而不是全局扫发布表。

4. **读取链路**
- 时间线里拿到帖子索引；
- 再根据图片 URL 去 CDN 拉内容；
- 有新内容时可做红点提醒。

### 朋友圈权限控制怎么做？

权限本质上是“关系 + 可见范围 + 用户配置”的联合判断。

一种可讲的工程化思路是：
- 用户和好友之间的关系、朋友圈权限设置、总可见范围这些信息，都需要在服务端持久化；
- 同时客户端会缓存一部分权限数据，用于本地快速判断和展示优化；
- 每次刷新朋友圈时，再结合当前用户与发帖人的关系、帖子本身的可见范围，做一次最终判定。

一句话说：
**发布时控制可分发范围，读取时再做一次权限校验，形成双层控制。**

### 如何设计高并发的朋友圈点赞系统架构？

点赞和评论通常是独立数据，不建议和主帖子强耦合。

常见做法：
1. **Redis 存点赞集合**
- 每条朋友圈对应一个 Redis Set；
- Set 里放给这条朋友圈点赞的用户 ID；
- 点赞数量可以直接用 `SCARD`，点赞列表可用 `SMEMBERS` 或分页拉取。

2. **好友可见点赞**
- 如果“只有好友才能看见某个点赞”，可以把“当前用户的好友集合”和“这条朋友圈的点赞用户集合”做交集判断；
- 工程上可以通过 `SISMEMBER` 之类方式快速判断是否可见。

3. **重复点赞幂等**
- 如果用数据库存点赞关系，可以用 `(user_id, post_id)` 联合唯一键保证一个人只能点一次；
- 如果用 Redis Set，本身天然去重，重复添加不会产生第二条记录。

面试里可以总结为：
**朋友圈主链路是“发帖异步分发到时间线”，点赞链路是“独立存储、缓存优先、关系裁剪可见性、幂等防重”。**
