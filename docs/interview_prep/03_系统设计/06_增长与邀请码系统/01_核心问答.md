## 1. Day 4：分布式 ID 与短链系统

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容


- 主题：全局唯一 ID、短链编码、热点与重定向
- 当日产出：
  - 对比口述：UUID / 号段 / Snowflake
  - 追问准备：“短链冲突和高并发写入如何处理？”

可以。你这天可以按“ID系统 + 短链系统 + 热点治理”三段来答。

**一、分布式 ID：UUID、号段、Snowflake 怎么选**

1. UUID

- 优点：无中心、生成简单、全局唯一性好。  
- 缺点：字符串长、无序，做 DB 主键会导致索引离散、页分裂多。  
- 适用：不追求有序、量不大或离线场景。

1. 号段模式（Segment/Leaf）

- 思路：从 DB 一次批量申请一段 ID（如 10000 个），本地内存递增发放。  
- 优点：趋势递增、性能高、实现简单。  
- 缺点：依赖号段服务和 DB；要处理号段预取、服务高可用。  
- 适用：业务系统主键、订单号等“希望有序且高性能”的场景。

1. Snowflake

- 思路：时间戳 + 机器ID + 序列号（常见 41+10+12）。  
- 优点：本地生成、趋势递增、吞吐高、长度固定。  
- 难点：时钟回拨、机器ID分配、跨机房部署。  
- 解法:  持久化 lastTimestamp，服务重启时先校验，防止重启后时钟更小。或者是把时间上报给 Zookeeper 之类的中间件，让他们来校验和管理每个机器的这个时钟。
- 适用：高并发在线系统主流方案。

面试一句话：  

“要无依赖选 UUID；要数据库友好选号段；要高并发低延迟通常选 Snowflake，但要重点治理时钟回拨和机器号冲突。”

---

**二、短链系统：生成、冲突、跳转链路**

1. 生成方案

- 自增ID(号段/雪花算法)转62进制（最常见）`id -> base62 short code`，短、稳定、可控。  
- 插入DB数据 （依赖唯一索引兜底防短码冲突）,写入冲突时重试（重新生成或加盐）。 
- 写入redis加缓存

生产常见：主方案用自增ID+Base62，别名短链做去重哈希辅助。

1. 跳转链路（读路径）

客户端 -> CDN/网关 -> 短链服务 -> Redis 查短码映射(code -> long_url) -> 命中返回 301/302 到长链。  

未命中再查 DB，回填缓存（设置 TTL）。  

埋点异步化：点击日志入 MQ，避免阻塞跳转主链路。

1. 301/302 怎么选

- 301：长期不变，利于 SEO 和缓存。  
- 302：可控性高，便于后续改目标链接。

很多业务默认 302，营销活动常用。

---

**三、热点 key 与缓存穿透（追问高频）**

1. 热点 key

- 现象：某短码爆红，请求集中打单 key。  
- 治理：本地缓存 + Redis，多副本读、热点预热、限流熔断、热点隔离。

1. 缓存穿透（查不存在短码）

- 参数校验（非法短码直接拦）。  
- Bloom Filter 预判不存在。  
- 空值缓存（短 TTL）防止反复回源。

1. 缓存击穿/雪崩顺带提

- 击穿：热点失效瞬间并发回源 -> 互斥重建。  
- 雪崩：大量 key 同时失效 -> TTL 抖动 + 限流降级。

## 2. 补充原问：邀请码系统设计（五阶段版）

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容

如果让我设计一个邀请码系统，我会先把它当成一个增长中台能力，而不是单次活动接口。因为它后面可能会服务注册拉新、交易返佣、活动裂变等多个场景。所以我会把核心链路拆成五段：邀请码生成、邀请码下发、邀请码验证、邀请关系绑定、邀请奖励发放。这里我会坚持一个原则：**绑定关系必须强一致，奖励发放可以最终一致**，这样既能保证业务正确性，也能把高并发压力拆开。

先说**生成阶段**。这一层核心是保证邀请码唯一、可读、可控。我一般会做两种模式：一种是活动开始前批量预生成，另一种是用户触发时实时生成。预生成适合大活动，能减少高峰时实时生成压力；实时生成适合长尾场景，节省库存。邀请码 code 的生成，首先用雪花算法生成一个全局唯一的 ID，且是自增的。然后再通过 Base62 的编码转换成字符串+数字，生成一个短的 8 位的码，便于传播。生成结果先落权威存储，再把热点数据同步到 Redis，这样后面查码时会快很多。

第二个是**验证阶段**。这里是所有请求进入系统后的第一道关键校验。我一般会先过 API Gateway，做鉴权、限流、黑名单、参数合法性检查。真正验码的时候优先查 Redis，因为邀请码是否存在、是否过期、是否被禁用、是否达到使用次数上限，这些都适合放缓存里快速判断。只有缓存没命中或者缓存不确定的时候，才回源数据库查权威状态。除此之外，这一层我还会把风控前置，比如设备异常、IP 异常、账号异常、短时间高频尝试，这些如果命中风险规则，就不进入后面的绑定逻辑。

第三个是**绑定阶段**，这是整个系统最核心的地方。这一层我一定会做成强一致。请求进入绑定逻辑之后，我会先做幂等校验，比如每次请求都带 `request_id`，防止用户重复点击或者客户端超时重试导致重复绑定。真正写绑定关系时，我会依赖数据库事务和唯一约束来保证“同一个被邀请人在同一个场景下只能绑定一次”。

第四个是**发奖阶段**。发奖我不会放在主链路同步执行，因为这样会拖慢接口，也会把库存、账户、消息通知这些下游系统耦合进来。所以我的做法是，绑定成功后只做一件事：发一条“绑定成功事件”到 MQ，比如 Kafka 或 RocketMQ。下游奖励服务异步消费，按规则决定是发现金、发积分、发优惠券还是发返佣。这里也必须做幂等，因为 MQ 可能会重复投递，所以消费方要保证同一个奖励事件只生效一次。

如果面试官再问我“这套设计为什么能扛住高并发”，我会补一句：**前面三段主要靠缓存、限流和风控削峰；绑定阶段靠幂等、事务和唯一约束保证正确性；发奖阶段靠 MQ 异步解耦压力。** 所以整个系统的设计思路其实就是八个字：**前链路做短，后链路做异步。**
