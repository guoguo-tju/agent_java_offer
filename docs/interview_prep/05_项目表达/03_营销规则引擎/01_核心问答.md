活动营销规则引擎（基于Aviator表达式）
背景：交易所要高频地推出各种活动来促活和拉新, 活动规则要圈选人群包、配置任务条件、以及完成任务后动作，配置大活动的复杂度和维护成本较高。所以构想出一个轻量级、可视化营销决策编排引擎，核心目标是让运营同学10-30分钟就能完成一个复杂活动的配置并上线,不需要研发干预.

(可配合架构图或流程图讲解)
技术选型以 Aviator 表达式引擎 为核心，整个系统采用 “配置层-存储层-缓存层-执行层” 四层闭环架构，实现运营自助编排 + 秒级热更新.

活动配置流程:
运营侧在管理台配置活动,包括活动基本信息->圈定人群包->任务条件->奖品配置, 在前端低代码画布上拖拽节点实时拼接出条件表达式. 点击"保存"前端提交完整的json决策树,到后端校验(比如:循环监测/节点合法性/版本对比), 然后进行规则解析Parser解析成树结构,序列化并存入DB, 然后推送给Redis更新缓存, 并且发MQ同步到决策服务重新预编译Aviator表达式; 整个"DB+Redis+MQ"的热更新链路, 实现活动上线的秒级更新和预热. 如果是更新活动会带有version字段,每次更新mysql和redis时verson+1.数据一致性方面: db和缓存之间有兜底机制,每30s一次后台job主动核对版本确保数据一致性. 

实时决策执行流程:
当用户侧出现行为事件时(如注册/交易/入金等), 后端通过Kafka实时接受用户事件中携带用户上下文信息Map, 决策系统根据activityId从Redis中获取到最新的决策树, 通过决策引擎递归遍历决策树:每个条件节点执行Aviator表达式, 预编译后的Expression执行对象, 字节码级别执行很快,根据true/false走到分支, 最后到达叶子动作结点Action, 执行对应的动作(如发奖/通知/发送MQ), 一次决策流程完成. 单次性能提示明显从50ms降低至5ms, 支撑万级别的TPS规则判断

总结一下: 前端低代码画布可以直接拖拽节点表达业务意图，运营同学能直观看懂“在什么条件下触发什么动作”，减少沟通损耗。对研发侧而言，决策树也便于做循环检测、节点合法性检查和版本比对，治理成本明显低于脚本堆砌方式, 树结构支持复杂活动的扩展。同时有一套热更新的机制, 通过预编译优化实时链路性能, 整体系统兼顾了灵活性、可解释性和工程可维护性。


问:为什么选决策树模型？
活动规则质就是逐层判断并进入不同分支，最终落到某个叶子动作节点。用决策树建模后，前端低代码画布可以直接拖拽节点表达业务意图，运营同学能直观看懂“在什么条件下触发什么动作”，减少口头沟通损耗。对研发侧而言，树结构也便于做循环检测、节点合法性检查和版本比对，治理成本明显低于脚本堆砌方式。树结构支持复杂活动的扩展，整体系统兼顾了灵活性、可解释性和工程可维护性。

问:为什么选 Aviator？
我们在选型时并不是直接锁定 Aviator，而是对 SpringEL、阿里的QLExpress、Groovy 脚本方案和 Drools 做过横向评估。
Groovy 脚本, 灵活性高，规则多了后不好维护, 以及性能抖动问题；
Drools 规则能力强，但体系偏重，学习成本高,更适合复杂规则网络
SpringEL、阿里的QLExpress都可用，但在我们场景下，最终选择 Aviator 的原因是：支持预编译，能将解析成本前移，显著降低线上时延抖动；与现有 Java 服务、版本化发布、链路集成成本低。单次决策时延从约50ms降到约5ms，说明该选型兼顾了业务灵活性与工程稳定性。及我们团团队内之前有人用过类似Aviator表达式的一些实践的先例

问：如何处理多实例漂移？
答：多实例漂移本质是“同一时刻不同实例持有不同规则版本”，会直接导致用户决策结果不一致。我的处理策略是三层防线：第一层是单调版本校验，实例只接受更高版本更新，拒绝乱序和回退消息，确保版本推进方向唯一；第二层是幂等刷新机制，同版本重复下发不会重复生效，避免广播重试造成抖动；第三层是周期性对账任务，以数据库最新版本为基准，每30秒检查实例内存版本和 Redis 版本，一旦发现落后即触发补偿刷新。我们还把关键版本变更打到监控中，观察发布时延、实例收敛时间和漂移实例数。通过这套机制，即使消息系统偶发延迟，也能在可控时间窗口内完成版本收敛，确保线上行为一致。

问: 遇到过的事故？
答：一次典型事故是规则发布后少量实例没有及时刷新到最新版本，导致同一用户请求在不同实例命中不同规则分支，出现行为不一致。复盘发现根因是发布链路存在时序竞争：某些实例先消费到广播但刷新失败，后续又因版本判断不严出现重复或乱序处理。我们的修复分三步：第一，强化单调版本校验，实例只接受更高版本，拒绝回退。第二，刷新逻辑改为幂等，即使重复消息也不会破坏状态。第三，引入30秒周期对账，主动发现并修复落后实例。事故后我们补齐了发布收敛监控和告警阈值，确保类似问题能在早期被发现并快速止损。

问：规模再涨10倍怎么做？

答：规模再涨10倍时，我会从“读扩展、算扩展、发布扩展、治理扩展”四个方向推进。读扩展上，对 Redis 做分片与热点隔离，结合本地缓存和异步预热，减少冷启动抖动。算扩展上，执行层做多实例横向扩容，按活动或租户维度拆分流量，避免单池竞争。发布扩展上，引入发布流控和分批广播，控制版本收敛节奏，防止全量同时刷新引发瞬时尖峰。治理扩展上，完善容量模型与压测基线，提前评估规则复杂度对CPU和时延的影响，并以自动告警和自愈机制兜底。核心原则是先保证一致性和稳定性，再追求极限吞吐，避免“扩容后更不稳定”的反效果。

# 活动营销规则引擎 - 20条追问

1. 问：为什么选决策树模型？

答：我选择决策树模型的核心原因是，它和营销活动规则的业务形态天然同构。活动规则通常由“人群圈选、条件判断、动作执行”三类环节组成，本质就是逐层判断并进入不同分支，最终落到某个叶子动作节点。用决策树建模后，前端低代码画布可以直接拖拽节点表达业务意图，运营同学能直观看懂“在什么条件下触发什么动作”，减少口头沟通损耗。对研发侧而言，树结构也便于做静态校验、循环检测、节点合法性检查和版本比对，治理成本明显低于脚本堆砌方式。更重要的是，树结构既支持复杂活动的扩展，又能保持执行链路稳定可控，兼顾了灵活性、可解释性和工程可维护性。

1. 问：为什么选 Aviator？

答：我们在选型时并不是直接锁定 Aviator，而是对 SpEL、QLExpress、Groovy 脚本方案和 Drools 做过横向评估。
Groovy 脚本, 灵活性高，规则多了后不好维护, 以及性能抖动问题；
Drools 规则能力强，但体系偏重，学习成本高,更适合复杂规则网络

1. 问：规则变更如何传播？

答：规则变更传播采用“版本推进 + 缓存刷新 + 广播通知 + 预编译落地”的闭环链路。具体流程是：运营在管理台提交规则后，后端先做结构与表达式校验，校验通过再写入数据库并生成新版本；随后将最新版本同步到 Redis，作为在线快速读取入口；再通过 MQ 广播规则变更事件给各决策实例，实例收到消息后拉取新配置并完成预编译，替换内存中的旧版本对象。为了确保全链路一致性，我们把版本号作为唯一对齐标准，所有实例只接受“高于当前版本”的更新，天然具备幂等能力。即使某次广播延迟或丢失，也会被周期性对账任务修复，从而保证最终一致和发布可追踪。

1. 问：为什么数据库（DB）和 Redis 都有版本号？

答：数据库和 Redis 同时维护版本号，是为了把“事实一致性”和“服务一致性”分层治理。数据库是规则的事实源，负责持久化、审计追踪和变更回放；Redis 是在线决策的高性能读取层，负责低时延服务。当只有数据库有版本号而缓存没有版本标识时，实例无法快速判断缓存是否陈旧，容易在高并发下出现旧规则被误用。双版本机制能让每个实例在读取时先比对版本，再决定是否执行、刷新或回源数据库。这样做的价值在于：发布链路出现抖动时可快速定位断点，实例行为可解释，回滚路径可控，同时把缓存从“黑盒加速层”变成“可验证的服务层”，显著降低多实例场景下的灰度与回滚风险。

1. 问：如何处理多实例漂移？

答：多实例漂移本质是“同一时刻不同实例持有不同规则版本”，会直接导致用户决策结果不一致。我的处理策略是三层防线：第一层是单调版本校验，实例只接受更高版本更新，拒绝乱序和回退消息，确保版本推进方向唯一；第二层是幂等刷新机制，同版本重复下发不会重复生效，避免广播重试造成抖动；第三层是周期性对账任务，以数据库最新版本为基准，每30秒检查实例内存版本和 Redis 版本，一旦发现落后即触发补偿刷新。我们还把关键版本变更打到监控中，观察发布时延、实例收敛时间和漂移实例数。通过这套机制，即使消息系统偶发延迟，也能在可控时间窗口内完成版本收敛，确保线上行为一致。

1. 问：为什么要预编译？

答：预编译的核心价值是把高成本操作前置，把线上路径做“瘦身”。如果每次请求都在线解析表达式并生成可执行对象，CPU 消耗和尾延迟都会明显上升，系统在高峰时更容易被解释执行拖垮。我们把表达式在发布阶段统一解析、编译并缓存，线上只拿预编译对象执行，避免重复解析和语法树构建，从根上减少无效开销。预编译还有两个附加收益：第一，发布期可提前暴露语法错误和非法函数调用，把风险拦截在上线前；第二，执行性能更稳定，便于容量评估和SLA管理。最终体感是请求路径更短、抖动更小、吞吐更高，这也是我们把单次决策时延从约50ms优化到约5ms的关键抓手之一。

1. 问：性能提升主要来自哪里？

答：性能提升不是单点优化，而是“缓存命中 + 预编译执行 + 数据结构稳定化”叠加的结果。第一，规则主数据走 Redis，减少数据库访问，把配置读取时延从毫秒级波动收敛到更稳定区间。第二，Aviator 表达式做预编译，线上请求不再反复解析表达式文本，CPU 时间显著下降。第三，执行引擎按决策树递归遍历，只执行必要分支，不做无效节点计算，降低平均执行步数。结合这些措施，系统在高并发活动场景下能稳定维持低延迟，单次决策从约50ms降到约5ms，同时保持规则变更的秒级生效能力。

1. 问：如何做兼容性保障？

答：兼容性保障的重点是“新规则可上线，旧规则不被破坏，历史能力可平滑演进”。我们采用 schema 版本化管理规则结构，发布时先按目标版本做字段合法性和必填项校验，再做表达式语义校验，确保配置可执行。对新增节点类型或函数能力，遵循向后兼容原则：旧版本规则在新执行器上保持原语义，新功能通过显式版本门控开启，避免全量规则被动受影响。执行阶段通过版本标签绑定解析器策略，保证同一条规则在不同时间点结果可复现。上线流程中配套灰度、回滚和对账机制，一旦发现兼容性问题可按版本快速回退到稳定态。这样既支持业务快速迭代，也避免“升级一次、全局受扰”的平台风险。

1. 问：MQ 广播失败怎么办？

答：MQ 广播失败是规则平台必须正视的常态故障，不应依赖“永不失败”的理想假设。我们的处理原则是“可重试、可幂等、可补偿、可观测”。首先，发布事件带版本号并采用幂等消费，允许重复投递，不会造成重复生效。其次，对发送失败和消费失败都配置重试策略，短期抖动可自动恢复。再次，设计对账补偿任务定期比对数据库、Redis和实例内存版本，发现实例未更新时主动触发拉取和预编译，弥补广播链路丢失。最后，把发布成功率、消费延迟、积压深度和实例收敛时长接入监控告警，确保问题可见且可定位。这样即使 MQ 出现局部故障，也不会长期影响规则一致性和业务可用性。

1. 问：Redis 数据过期或滞后怎么办？

答：Redis 作为高性能读层，确实可能因过期、淘汰或更新延迟出现数据滞后。我们的策略是“运行时版本校验 + 回源加载 + 主动修复”。在执行请求时，实例先读取 Redis 的规则和版本号，并与本地已加载版本做比较：若 Redis 缺失或版本落后，则触发回源数据库读取最新规则，并立即完成预编译和内存替换，同时把最新版本回写 Redis。这样可确保单次请求不会长期依赖过期缓存。发布链路之外再配合周期对账任务，持续扫描并修复缓存滞后，避免问题累积。通过这套机制，Redis 即便短时异常也只会影响极小时间窗口，不会演变成全局一致性问题，线上决策准确性仍可保障。

1. 问：规则安全如何保障？

答：规则安全不只是“防注入”，而是覆盖配置入口、执行引擎和运行资源的全链路治理。首先在发布前做结构校验，拒绝非法节点、缺失字段和循环依赖。其次对表达式执行能力做最小权限控制，只开放白名单函数，禁止高风险调用，避免运营误配触发危险操作。第三，限制表达式复杂度、节点数量和执行深度，防止超复杂规则拖垮执行线程。第四，动作节点引入幂等约束和参数校验，避免重复发奖、错误下发。最后，通过审计日志记录“谁在何时发布了什么版本”，并在监控侧跟踪异常命中率与执行超时。这样安全策略既不牺牲运营效率，又能在平台层面控制规则失控风险。

1. 问：为什么不直接脚本执行？

答：直接脚本执行看似灵活，实则在中台场景下治理成本极高。第一，脚本自由度过高，语义难标准化，运营和研发之间很难形成统一抽象，长期会演化为“只有作者自己看得懂”的规则黑盒。第二，脚本缺乏结构化约束，发布前难做自动化校验，线上一旦出问题定位和回滚都很慢。第三，脚本执行安全边界难控制，容易引入函数滥用、资源失控和审计缺失。相比之下，决策树 + 表达式引擎方案把灵活性限制在可治理范围内：配置可视化、能力可白名单、行为可追踪、版本可回滚。对于高频变更的营销业务，这种“受约束的灵活”比“无限制的灵活”更适合长期演进。

1. 问：如何防止坏规则上线？

答：防止坏规则上线，我们采用“静态校验 + 预执行验证 + 灰度发布 + 快速回滚”四步机制。静态校验阶段检查树结构合法性、节点依赖、表达式语法和函数权限，先把明显错误挡在入口。预执行验证阶段使用样本数据回放关键路径，观察命中率和动作分布是否符合预期，避免逻辑偏差直接进入生产。发布阶段走灰度策略，先在小流量实例启用新版本，监控时延、错误率和关键业务指标，再逐步放量。若出现异常，通过版本化机制可在分钟级回退到稳定版本，并由对账任务推动全实例收敛。这样可以把规则发布风险从“线上排雷”前移到“上线前可验证”，显著降低事故概率和影响面。

1. 问：平台化代价是什么？

答：平台化不是零成本，主要代价体现在系统复杂度和治理成本上升。为了让运营具备自助能力，我们需要建设配置层、存储层、缓存层、执行层的完整闭环，还要配套版本发布、对账补偿、监控告警和审计体系，研发投入明显高于一次性业务开发。同时，平台要兼顾灵活性与安全性，意味着要设计规则边界、函数白名单、复杂度限制等治理策略。短期看，这会增加架构和维护负担；但中长期收益更大：活动上线不再强依赖研发，规则调整速度显著提升，且执行稳定性可度量、可回滚、可复盘。对高频营销场景而言，这种投入是把“人力瓶颈”转化为“系统能力”的必要过程。

1. 问：如何定义成功？

答：我定义这个项目成功，不是只看技术指标，而是看“业务效率、系统性能、工程可控”三条线是否同时达标。业务效率上，运营能在10-30分钟内独立完成活动配置与发布，研发从重复接需求中释放出来。系统性能上，单次决策时延从约50ms降到约5ms，并在高并发下保持稳定。工程可控上，规则发布具备版本追踪、灰度放量、异常回滚和对账修复能力，实例行为一致性可监控可验证。最终成功标准是：活动迭代更快、线上风险更低、团队协作更顺畅，而不是只追求单次压测数字。只有这三条线闭环，平台价值才真正成立。

1. 问：遇到过的事故？

答：一次典型事故是规则发布后少量实例没有及时刷新到最新版本，导致同一用户请求在不同实例命中不同规则分支，出现行为不一致。复盘发现根因是发布链路存在时序竞争：某些实例先消费到广播但刷新失败，后续又因版本判断不严出现重复或乱序处理。我们的修复分三步：第一，强化单调版本校验，实例只接受更高版本，拒绝回退。第二，刷新逻辑改为幂等，即使重复消息也不会破坏状态。第三，引入30秒周期对账，主动发现并修复落后实例。事故后我们补齐了发布收敛监控和告警阈值，确保类似问题能在早期被发现并快速止损。

1. 问：50ms 到 5ms 怎么解释？

答：从约50ms到约5ms的优化，本质是把“在线解释执行”改造成“离线编译 + 在线快速执行”。原先每次请求都需要做表达式解析、语法树构建和执行准备，CPU开销高且抖动大。优化后，我们在规则发布时就完成表达式预编译，线上请求只做上下文填充和预编译对象执行，去掉了重复解析成本。同时，规则主数据放在 Redis 并配合版本缓存，减少配置读取路径的抖动。再加上决策树按命中路径执行，避免无关分支计算，整体执行步数下降。三者叠加后，平均时延和尾延迟都显著收敛，5ms 不是单点魔法，而是架构层改造后的自然结果。

1. 问：如何防止表达式滥用？

答：防止表达式滥用的关键是“能力收敛 + 成本约束 + 审计可追溯”。我们先做函数白名单，只开放与业务决策相关、可控且无副作用的函数，禁止任意调用外部能力。然后设置复杂度阈值，例如表达式长度、嵌套深度、函数调用次数和节点规模上限，防止运营配置出难以维护或高耗时规则。发布前做语义校验和风险扫描，发现超限或高风险表达式直接拒绝。运行时再配合超时保护和异常监控，避免单条坏规则拖垮实例。最后通过版本审计记录发布人、发布时间和变更内容，出现问题可快速追责与回滚。这套机制既保留业务灵活性，也把风险稳定地压在平台边界内。

1. 问：规模再涨10倍怎么做？

答：规模再涨10倍时，我会从“读扩展、算扩展、发布扩展、治理扩展”四个方向推进。读扩展上，对 Redis 做分片与热点隔离，结合本地缓存和异步预热，减少冷启动抖动。算扩展上，执行层做多实例横向扩容，按活动或租户维度拆分流量，避免单池竞争。发布扩展上，引入发布流控和分批广播，控制版本收敛节奏，防止全量同时刷新引发瞬时尖峰。治理扩展上，完善容量模型与压测基线，提前评估规则复杂度对CPU和时延的影响，并以自动告警和自愈机制兜底。核心原则是先保证一致性和稳定性，再追求极限吞吐，避免“扩容后更不稳定”的反效果。
