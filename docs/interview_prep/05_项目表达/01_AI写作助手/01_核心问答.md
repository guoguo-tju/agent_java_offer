# AI写作助手（LangChain + LangGraph）项目讲稿

## 讲解方式

**项目名称：**  

个性化写作智能体

**责任描述：**从0到1构建写作智能体 , 基于 LangGraph 将“风格提炼→RAG素材检索→定长创作→AI初审+人工二审→多比例封面生成”全流程标准化，解决素材分散与风格不稳问题，长文产出由2-3小时缩短至约10分钟。

**系统架构:** 前后端采用 React + FastAPI + LangChain + Chroma + SQLite + SQLModel 架构**技术描述：**  

1. 使用 LangGraph 将写作流程编排为状态机“风格提取->素材检索→风格约束写作→去AI味审核→人工二审编辑→封面生成”多阶段流程，实现写作链路标准化与可复用。
2. 基于 LangChain + Chroma 构建 RAG 检索增强链路，支持写作过程按主题动态召回素材，提高内容相关性与信息密度。
3. 前后端采用 React + FastAPI + LangChain + Chroma + SQLite + SQLModel 架构。
4. 围绕流式SSE生成与DB存储的一致性问题优化，收敛 ORM 会话生命周期,  把流式生成拆成‘任务主记录 + 分片chunk幂等落库 + 结束态收口’，并把 ORM session 改成批次短事务，从而在 SSE 中断、重试、并发下仍保持可恢复的一致性。
5. 项目在个人生成创作场景持续迭代验证，已形成通用的闭环能力，项目已开源至 GitHub 。

一. 大模型（LLM）调用失败怎么处理？

答：错误分为可重试（超时、限流、瞬时网络）和不可重试（参数错误、内容违规、上下文超限）。

可重试错误走指数退避并限制最大次数. 

不可重试直接失败并返回可读错误码。若主模型持续失败，会按策略降级到备用模型或缩短上下文执行保底版本。

所有调用都记录 ，便于复盘与成本治理。这样可以把“偶发失败”控制在业务可接受范围内。

二. 问：最难的工程问题是什么？

答：最难的是流式生成场景下的“一致性边界”问题：

前端先看到流式对话分片结果，但后端涉及 SQLModel 会话提交、向量检索调用、状态更新三套动作，任何一步失败都可能出现半成功状态。

早期版本把会话持有得太长，导致连接占用和异常回滚范围过大

解决方案：

第一层是事务收窄，把流式生成拆成‘任务主记录 + 分片chunk幂等落库 + 结束态收口，并把 ORM session 改成批次短事务

第二层是异常恢复，引入 task 状态机与结束态收口机制，失败时标记并可重试补偿。这样做后，单次失败只影响当前分片，不会污染整条链路，稳定性和可恢复性明显提升，也更容易做并发扩展。

二.一.问：测试怎么做？

答：测试策略分三层：

单测验证纯业务逻辑，如路由条件、重试上限、评分阈值；

接口回归验证端到端链路，覆盖正常流、降级流、失败流和人工介入流；

Prompt 回归使用固定样本集做快照比对，监控风格漂移和结构退化。

对流式接口我还会做并发与中断测试，验证重连、取消、续写是否一致。

三. 问:业务价值体现在哪？

答：把原本碎片化的创作动作产品化为标准流程：素材检索、风格约束改写、质量审核、人工复核、封面生成形成闭环。

对个人或团队来说，收益体现在两个维度：

效率上，长文产出时间从小时级降到半小时级；

质量上，输出风格更稳定、可复审、可回放，不再依赖单次临场 Prompt。

从工程视角看，流程资产可复用、可迭代，后续新增场景只需扩节点和规则，边际成本更低。

四. 问：和普通聊天版（chatbot）有啥区别？

答：chatbot 的核心是展示能力，生产化系统的核心是稳定交付。这个项目和普通聊天 demo 的区别在于：

第一，有显式状态机和可追踪执行轨迹，不是一次性对话；

第二，支持质量门控和人工复核，不把最终质量完全交给模型随机性；

第三，有持久化与版本化，结果可回放、可审计；

第四，具备异常处理、重试和降级策略。

换句话说，我做的是“可维护的后端系统”，而不是只在单次演示里看起来聪明的原型。

五. 问：如果上生产怎么扩展？

答：生产扩展会从存储、执行、观测三方面推进。

存储层将 SQLite 升级到 PostgreSQL，并引入 Redis 做短期状态缓存，向量检索可迁移到 生产专用向量库。

执行层把长耗时节点异步化，接入任务队列与 Worker 池，支持水平扩展和限流。

观测层补齐 Metrics、Logs、Traces，并统一 traceId 打通 API、工作流、模型调用链路。

再加模型路由、成本配额监控和熔断策略，才能在高并发下稳定运行并可控运营。

六. 问：这个项目中如何防止模型陷入逻辑死循环？
一个是限制了工具的最大调用轮数以及对调用工具设超时。然后一个是在那个主编审核，以及审核完重写的这个回环里，限制一些限制最大的步数。

七.面试官：文本切分和 Embedding 模型用的是什么？为什么选这个？
文本切分，使用的是按 token 去切分，把大文本切成小文本。一般 token 是在256~512个 token 因为一般这个 token 它的嵌入模型，做 embedding 的效果会比较好。同时文本切分的时候，还要去兼顾它那个上下文的语义。比如说它是一一整段的，一整段的文本就如果就按段段落区分。如果是如果是如果是代码块的话，就要把整段代码块也截取下来。然后还有一个文文本的重叠度，一般会预留10%~20%的那个文本重叠的部分，就是为了防止它切的时候语义断层，到时候检索出来效果会比较差。
Embedding 模型用的硅基流动的 BEG-M3 模型。因为是可以免费调用的。对中文的支持效果也比较好。

八. 长记忆机制的介绍? 存储时如果出现前后不一致的情况如何解决?
长记忆就是把“跨会话仍有价值的信息”沉淀下来，让 Agent 下次还能用.
长期记忆分三类，一类是事实记忆，比如说用户的画像、偏好、账号的配置，这些就存到一些结构化的存储，比如说 MySQL 或者 Redis 里。二类是行业知识，就比如说跨对话的记忆、文档的片段，还有一些行业的经验。这个使用向量数据库检索，就是与 RAG 的流程类似。还有一类就是过程记忆，比如说是任务轨迹或者工具调用的结果，失败的原因。这个可以记到事件或者日志的记录里，支持复盘和恢复。

存储记忆是前后不一致，可能就是需要先定义真相源。主要是以结构化的数据库为主，以向量库 只是辅助检索的副本。然后每条记忆带版本号、更新时间、源头还有重要性，读取时按最新版本优先加高置信优先。并且定时的跑 job 把主主库的 job 和向量库的哈希做对比，发现漂移的时候自动重建向量重建向量库的索引分片。保证整个记忆是可追溯、可修复、可回滚的。

九. 长期记忆与短期记忆的压缩方案
短期记忆压缩（会话内）:
短期记忆的压缩主要是通过滑动窗口保保存最近 K 轮的原文，比如说6~10轮。6~10轮窗口之外的内容会进行结构化的摘要。然后当上下文投肯，超投肯预算的时候，进行一次触发上下文的压缩。压缩完之后，优先注入最近的原文和摘要，保证整个上下文的连续性。

长期记忆压缩（跨会话）:
先抽取，把原始的对话对称分成三类。一类是事实性的，一类是偏好性的，一类是事件事件性的。然后针对每一类做分别的存储，事实性的存到结构化的数据库里，然后知知识性的存到向量与向量库里，然后对于事件性的就是存到一些日志系统里。然后再按周或者月做聚类的摘要，低价值的记忆要降级或者过期。然后每条记忆带 Version  Update  Source 和和关键度。然后读取时按照意图做动态的注入，根据 Version 和置信度等做 Top K 的检索。

# AI写作助手 - 20条追问

1. 问：为什么选 LangGraph 而不是纯链式流程（chain）？

答：这个项目不是单步问答，而是“检索-改写-审核-回退-人工介入-再生成”的多分支流程。纯 chain 更适合线性任务，一旦出现条件路由、循环重试和中断恢复，维护成本会明显上升。LangGraph 的优势是把节点、边、状态转移显式建模，分支条件可审计、可测试，失败节点可单点重试，后续接入 Checkpointer 也更自然。对工程落地来说，图编排的可控性和可观测性明显优于纯链式串联。

1. 问：LangChain 在项目里负责什么？

答：LangChain 在项目里主要承担三层职责：第一是模型层抽象，统一接入不同 LLM 供应商并屏蔽差异；第二是提示词与输出层，使用模板化 Prompt 和结构化解析器保证输入输出可控；第三是检索增强执行层，把 Retriever、重写器、上下文组装串成可复用链路。简单说，LangGraph 管流程编排，LangChain 管单节点内的“模型调用与工具组合”，二者分工清晰，便于后续替换模型和复用能力模块。

1. 问：核心状态有哪些？

答：我把状态分为四类：内容状态、控制状态、质量状态和追踪状态。内容状态包括素材片段、草稿正文、封面提示词；控制状态包括当前节点、分支路由、重试计数、是否人工介入；质量状态包括审核分、AI味评分、结构完整性分；追踪状态包括 taskId、版本号、时间戳和错误码。这样设计的好处是流程控制与业务数据解耦，既支持中断恢复，也支持复盘定位“卡在哪个节点、因为什么分支决策进入重写”。

1. 问：检索增强生成（RAG）是怎么接入的？

答：RAG 链路分三步：先离线入库，再在线召回，最后上下文装配。离线阶段会对素材做清洗、分块、向量化并写入 Chroma，同时保留来源、主题、时间等元数据。在线阶段按用户主题和风格标签做向量检索，可叠加元数据过滤，控制召回范围。生成阶段不会原样拼接全部片段，而是做去重与长度预算，再把高相关片段注入 Prompt。这样既提升事实相关性，也降低上下文噪声和幻觉风险。

1. 问：为什么最小可行版本（MVP）用 SQLite + Chroma？

答：MVP 阶段的目标是快速验证流程闭环，而不是先做复杂运维。SQLite 适合本地单机快速迭代，事务语义清晰、迁移成本低，足够支撑任务、审核、版本等结构化数据。Chroma 则覆盖向量检索能力，满足素材召回核心诉求。两者组合能在最小成本下跑通“结构化状态 + 语义检索”双栈。等到并发和数据规模上来，再平滑迁移到 PostgreSQL + pgvector 或外部向量库，不会推翻现有分层设计。

1. 问：如何防止流程无限循环？

答：我采用“硬阈值 + 质量收敛 + 人工兜底”三层防护。硬阈值层面，每个可回退节点都设置最大重试次数，超过即强制退出自动链路；质量收敛层面，若多次重写后评分提升低于阈值，判定为不收敛，避免无效迭代；人工兜底层面，自动分支失败会路由到 HITL 节点，由人工确认后继续或终止。此外所有循环节点都记录迭代轨迹，便于后续分析“为什么循环、在哪步循环”，支持策略调优。

1. 问：提示词漂移怎么控？

答：我把 Prompt 当作“可发布资产”管理，而不是临时文本。具体做法是模板版本化、变量 schema 固化、上线前回归评测三件事。模板版本化保证可追溯，出现质量回退可以快速回滚；变量 schema 固化防止上下文字段漂移导致输出不稳定；回归评测使用固定样本集比对风格一致性、结构完整性和事实引用率。这样可以把“提示词调参”从拍脑袋试错变成可验证的工程流程，显著降低线上输出波动。

1. 问：输出质量如何评估？

答：质量评估采用“规则指标 + 模型评审 + 人工抽检”的组合策略。规则指标关注可量化项，如字数区间、段落结构、禁用词命中、引用完整性；模型评审负责语义维度，如逻辑连贯性、风格贴合度、AI味强弱；人工抽检用于校准模型评审偏差，重点覆盖高风险主题。最终会输出统一评分卡并绑定 taskId，作为后续路由依据：达标直接通过，不达标进入重写或人工节点。这样既保证效率，也保留质量兜底。

1. 问：为什么做流式接口（API）？

答：流式接口的核心价值不是“炫技”，而是降低用户等待焦虑并提升交互可控性。长文生成耗时天然较长，若只返回最终结果，用户在几十秒内没有反馈容易误判为失败。SSE 可以让前端先拿到首包并持续看到增量结果，同时同步进度事件（检索中、改写中、审核中），增强透明度。工程上我还把流式分片与任务状态对齐，支持中断、重连和取消操作，确保体验层和执行层语义一致，而不是单纯文本流输出。

1. 问：大模型（LLM）调用失败怎么处理？

答：我先做错误分层，再做分层恢复。错误分为可重试（超时、限流、瞬时网络）和不可重试（参数错误、内容违规、上下文超限）。

可重试错误走指数退避并限制最大次数. 

不可重试直接失败并返回可读错误码。若主模型持续失败，会按策略降级到备用模型或缩短上下文执行保底版本。

所有调用都记录 ，便于复盘与成本治理。这样可以把“偶发失败”控制在业务可接受范围内。

1. 问：最难的工程问题是什么？

答：最难的是流式生成场景下的“一致性边界”问题：前端先看到分片结果，但后端涉及 SQLModel 会话提交、向量检索调用、状态更新三套动作，任何一步失败都可能出现半成功状态。早期版本把会话持有得太长，导致连接占用和异常回滚范围过大；并且向量操作与事务提交耦合后，排障很难定位。这个问题不是单点 bug，而是链路级工程问题，涉及会话生命周期、幂等写入和失败补偿策略，需要整体重构而不是局部修补。

1. 问：如何解决？

答：解决方案分三层：第一层是职责解耦，把编排控制、内容生成、持久化写入拆成独立服务函数，避免跨层共享长会话；第二层是事务收窄，改为“短事务 + 分片幂等写入”，每次只提交最小必要状态；第三层是异常恢复，引入 task 状态机与结束态收口机制，失败时标记并可重试补偿。这样做后，单次失败只影响当前分片，不会污染整条链路，稳定性和可恢复性明显提升，也更容易做并发扩展。

1. 问：如何保证层次清晰？

答：我用分层和契约两条线保证清晰度。分层上，API 层只处理入参校验和响应协议；Service 层负责业务编排和规则决策；Repository/Model 层只负责数据读写与实体映射；Workflow 层专注状态转移与节点执行。契约上，层间统一使用 DTO 和明确的错误码，不透传底层实现细节。这样即使替换 LLM、切换向量库或升级存储，也主要影响局部层，不会牵动整条链路，代码可维护性和可测试性都更高。

1. 问：业务价值体现在哪？

答：价值不只是“能生成文章”，而是把原本碎片化的创作动作产品化为标准流程：素材检索、风格约束改写、质量审核、人工复核、封面生成形成闭环。对个人或团队来说，收益体现在两个维度：效率上，长文产出时间从小时级降到半小时级；质量上，输出风格更稳定、可复审、可回放，不再依赖单次临场 Prompt。从工程视角看，流程资产可复用、可迭代，后续新增场景只需扩节点和规则，边际成本更低。

1. 问：测试怎么做？

答：测试策略分三层：单测验证纯业务逻辑，如路由条件、重试上限、评分阈值；接口回归验证端到端链路，覆盖正常流、降级流、失败流和人工介入流；Prompt 回归使用固定样本集做快照比对，监控风格漂移和结构退化。对流式接口我还会做并发与中断测试，验证重连、取消、续写是否一致。这样能同时覆盖“代码正确性”和“模型行为稳定性”，避免只测接口通却忽略生成质量回退。

1. 问：和普通聊天演示版（demo）有啥区别？

答：demo 的核心是展示能力，生产化系统的核心是稳定交付。这个项目和普通聊天 demo 的区别在于：第一，有显式状态机和可追踪执行轨迹，不是一次性对话；第二，支持质量门控和人工复核，不把最终质量完全交给模型随机性；第三，有持久化与版本化，结果可回放、可审计；第四，具备异常处理、重试和降级策略。换句话说，我做的是“可维护的后端系统”，而不是只在单次演示里看起来聪明的原型。

1. 问：如果上生产怎么扩展？

答：生产扩展会从存储、执行、观测三方面推进。存储层将 SQLite 升级到 PostgreSQL，并引入 Redis 做短期状态缓存，向量检索可迁移到 pgvector 或专用向量库。执行层把长耗时节点异步化，接入任务队列与 Worker 池，支持水平扩展和限流。观测层补齐 Metrics、Logs、Traces，并统一 traceId 打通 API、工作流、模型调用链路。再叠加模型路由、成本配额和熔断策略，才能在高并发下稳定运行并可控运营。

1. 问：人在回路（HITL）体现在哪？

答：HITL 不是“最后人工看一眼”，而是流程级可插拔节点。系统会在低分审核、风格冲突或事实风险场景下自动路由到人工节点，人工可以选择重写、局部编辑或直接通过。关键是人工介入后不会打断全局状态：系统会记录版本差异和操作轨迹，并从当前节点继续后续流程，而不是整单重跑。这种设计把“AI效率”和“人工把关”统一在一条链路里，既保证产能，也保证发布质量与可审计性。

1. 问：下一步怎么优化？

答：下一步优先做 Checkpointer 和评测闭环。Checkpointer 可以把关键节点状态持久化，支持中断恢复、断点续跑和失败重放，特别适合长文本与多分支流程。评测闭环方面，会引入离线样本集与线上反馈联动，形成“版本发布-质量监控-策略回滚”的闭环机制。同时优化模型路由策略，根据任务类型动态选择模型与上下文长度，在保证质量的前提下降低 token 成本。目标是把系统从“能用”提升到“长期可运营”。

1. 问：这个项目对面试价值？

答：这个项目的面试价值在于它覆盖了 AI 工程落地的完整链路，而不只是会调接口。我不仅实现了 LangChain/LangGraph 的基础使用，还处理了真实工程问题：流程状态建模、流式与存储一致性、重试与降级、质量评测、人工介入和可观测性。面试官通常关注“能不能把 AI 能力做成可维护系统”，这个项目可以直接证明我具备从原型到工程化的能力，也能体现我对架构边界、稳定性和长期演进的理解。

## 补充：更完整的项目口述版

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容

项目是一个从 0 到 1 搭建的个人写作智能体。

之前的写作过程比较的非标，一个是自己去各种网站上去看和收集素材，再自己手写文字，再生成封面图，去用微信公众号的后台排版工具进行排版，那排版工具比较难用. 总结下问题: 一是之前 写一篇长文大概要好几个小时，效率很低。二是收集素材多网站上找,素材比较分散,没法好的统一管理; 三是长文写作很容易风格不稳定、结构散、用AI写的话AI 味重

我把整套写作流程标准化后，融合成一个写作智能体，单篇长文产出时间从数小时缩短到了约 30 分钟。

系统能力主要有四块：  
第一，写作风格建模, 基于过往文章做 12 维风格建模，沉淀个人写作风格；  
第二，素材库, 支持素材收集, 用的标准的rag链路, 长文本的切片以及embedding向量化, 存入向量数据库中, 在写作时可以通过 RAG 检索，写作时可按主题、标签召回素材；  
第三，智能写作, 按字数和风格生成文章，经过 AI 主编审稿 + 人工二审，降低“AI 味”, 这里用LangGraph做图编排来控制, 以及checkpoint的小事务重试  
第四，支持根据正文自动生成封面，支持 2.35:1（公众号） 等多比例和多风格提示词切换。  
第五，智能排版，实现多种排版样式的公众号html格式的一键排版导出

技术架构上，前端是 React，后端是 Python FastAPI；流程编排基于 LangGraph，模型链路基于 LangChain，向量检索使用 Chroma，结构化数据先用 SQLite 做本地存储, 嵌入模型用的bge-m3。

工程上最难的问题是流式生成场景的一致性。  
用流式SSE的话,前端会先看到流式分片，但后端任一步失败都可能造成“半成功”状态，同时长事务还会占用数据库连接。  
我的解决方案有两点：  
第一，收窄事务边界，把流程拆成“主任务记录 + 分片 chunk 落库 + 结束态收口”，并把 DB 的长事务缩短为短事务。  
第二，引入异常恢复机制，基于状态机做失败标记和重试。这样单次失败只影响当前分片，不会污染整条链路，稳定性和可恢复性都明显提升。

如果扩展到生产级，我会重点推进三件事：  
存储层升级到 PostgreSQL + Redis + 生产级向量库 Milvus / PgVector；  
执行层长耗时的任务引入异步任务队列和线程池，同时做水平扩展与限流；  
观测层补齐 Metrics、Logs、Traces，并打通全链路，叠加模型路由、成本监控和熔断策略。
