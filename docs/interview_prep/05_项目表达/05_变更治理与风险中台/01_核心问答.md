### 1.2 某大型支付科技平台技术风险中台

某大型支付科技平台技术风险中台，收口全站所有的变更，提供了一个标准的变更 pipeline 能力。即一个变更进行，必须满足变更三板斧的要求：可灰度、可监控、可回滚。

#### 变更三板斧

- 可灰度：比如要推送一个配置，不能一把梭，要通过分阶段逐步扩大流量的方式来验证。
- 可监控：在变更分阶段执行过程中，触发对应的变更防御规则。防御规则可以理解为一段脚本，由各域的 SRE 来编写。比如执行过程中/执行完成发现监控的报警数上升，就将变更进行熔断，不进行下一个批次，由人为确定有无风险。
- 可回滚：在变更分阶段执行过程中发现问题，要具备一键回滚的能力，回滚到安全的版本。

#### 具体实现（流程图）

1. 用户通过统一变更平台提交变更单，进入标准提单接口与 pipeline 流程。
2. 从灰度核心根据变更对象读取分批策略，同时从风险评估平台获取影响面和风险等级信息（不同风险等级对应不同批次暂停策略）。
3. 有了上述的信息后构造变更计划工单，然后开启审批流进行审批。
4. 审批完成之后开始执行流。
5. 执行流开始先进行调用变更核心（变更防御的平台）整体前置校验（主要判断变更准入，比如饭点不能进行变更，比如该变更对象有未执行完成的变更，不能进行）。
6. 变更准入通过后可以分批执行，每批次也分为前置校验、执行、后置校验。前置校验主要做一些准入判断，后续校验主要看变更之后有没有异常。前后置校验都要调用变更核心来执行上面的防御规则。变更核心提供防御能力相关的 SPI，可以实现防御能力可插拔。SPI 已经接入的防御能力：资损核对能力 / 灰度引流能力 / 监控能力 / 自定义巡检能力等。
7. 在执行步骤通过调用 API 总线，也是通过 SPI 集成各业务（Saas、Paas、Iaas）的执行能力，执行能力可插拔。
8. 如果执行过程中某个前后置校验没有通过，可以通过错误码平台来执行相应的预案。比如推送 ip 失败，重试三次后可以系统跳过这个步骤继续下面的推送，一定程度上降低人的干预，进一步可以做到变更无人值守。
9. 同时还建设了消息中心、答疑机器人等配套能力。

#### 架构亮点

横向的 pipeline 流程 + 纵向的切面的变更防御体系。

#### 技术亮点

数据强一致性设计。  
自研状态机对变更单状态的强管控。  
用户定制化审批流程、执行流程。  
可插拔的原子执行能力。  
变更无人值守能力。  
宕机自愈机制。
