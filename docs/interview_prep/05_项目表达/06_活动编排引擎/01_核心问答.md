### 原文内容

这个项目的背景是，交易所需要高频推出活动来做促活和拉新。每次配置活动都要处理复杂规则，包括人群圈选、任务条件、完成后的发奖动作。大活动场景下，配置复杂度和维护成本高，效率也偏低。

我当时先把这个项目抽象成三类核心问题。

第一类是业务自助化与交付敏捷性问题，核心是如何把活动规则能力从研发实现升级为运营可自助编排，缩短活动上线和调整周期。

第二类是复杂配置可维护性问题，把复杂、易变的活动规则从硬编码中解耦出来，沉淀成可配置、可复用、可持续扩展的通用规则模型。

第三类是实时决策效率与系统承载问题，核心是如何让规则链路在高并发场景下仍然保持低时延执行和稳定吞吐。

所以我主导建设了一套轻量级、可视化的营销决策编排引擎，核心目标是让运营同学在 10 到 30 分钟内自主完成复杂活动配置并上线，减少研发介入。

技术上，我以 Aviator 表达式为核心，整体架构分为配置层、存储层、缓存层、执行层四层。  
离线配置流程是：运营在活动管理台通过低代码画布配置活动信息、人群包、任务条件、奖品，前端拖拽节点实时拼接条件表达式。点击保存后，前端提交完整 JSON 决策树。后端先做校验，包括节点合法性、循环检测、版本比对；然后通过 Rule Parser 将 JSON 解析为树结构，序列化存入 DB，再推送 Redis 刷新缓存，同时发 MQ 通知决策服务重载并预编译表达式。  
DB + Redis + MQ 形成热更新链路，可支持活动秒级上线和预热。活动更新带 version 字段，每次更新 version +1。数据一致性方面，DB 与在线缓存之间有 30 秒离线核对任务，确保 JSON 规则一致。

在线实时决策流程是：用户发生注册、交易、入金等行为时，后端接收上游实时事件，事件里带用户上下文 map。系统根据 activityId 从 Redis 获取最新决策树，递归遍历执行。每个条件节点执行 Aviator 表达式，返回 true 或 false，并进入对应分支；最终到达叶子动作节点，执行发奖或通知等动作。  
因为表达式做了预编译，执行是字节码级，单次决策从 50ms 降到 5ms，可支撑万级 TPS 的规则判断。

为什么选决策树模型：活动规则本质是逐层判断、分支流转、最终命中叶子动作，和决策树天然匹配。决策树也能直接映射低代码画布的 JSON 嵌套结构，业务可读性更强；同时便于做循环检测、节点校验和版本对比，治理能力优于脚本堆叠方式，也支持复杂活动扩展。

为什么选 Aviator：我们做过横向评估，候选包括 SpEL、QLExpress、Groovy、Drools。Groovy 灵活但规则量大后可维护性差，且运行时有性能抖动；Drools 规则能力强但体系较重、学习和维护成本高；SpEL 与 QLExpress 在本场景可用。最终选 Aviator 的核心原因是支持预编译，能把解析成本前移，降低执行时延波动，并且与 Java 服务链路集成更顺畅；团队也有相关实践经验，落地风险更低。

如何处理多实例漂移：我的方案是三层防线。  
第一，单调版本校验，实例只接受更高版本，拒绝乱序和回退。  
第二，幂等刷新，同版本重复下发不重复生效，避免广播抖动。  
第三，周期对账，以 DB 最新版本为准，每 30 秒核对内存实例与 Redis 版本，发现落后立刻补偿更新。  
另外把版本更新打到监控，观察发布时延、实例收敛时间、漂移实例数。即使有偶发消息延迟，也能在可控时间内收敛。

一次典型事故是：规则发布后，少量实例未及时刷新到新版本，同一用户在不同实例命中不同分支，出现行为不一致。修复做了三步：  
第一，强化单调版本校验，只接收更高版本，拒绝回退。  
第二，刷新逻辑改为幂等，重复消息不破坏状态。  
第三，引入 30 秒周期对账，主动发现落后并补偿。  
同时补齐发布收敛告警和阈值，保证问题早发现、快止损。

如果规模再涨 10 倍，我会从读扩散、写扩散、发布扩散、治理扩散四个方向推进。  
读扩散：Redis 分片与热点隔离，叠加本地缓存和异步预热，降低冷启动抖动。  
写扩散：执行层多实例扩容，按活动或租户拆流，避免单线单池竞争。  
发布扩散：引入发布流控和分批广播，控制版本收敛节奏，避免全量同时刷新造成尖峰。  
治理扩散：完善容量模型和压测基线，提前评估规则复杂度对 CPU 时延的影响，并用自动告警与自愈兜底。
