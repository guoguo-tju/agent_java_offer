## 1. RPC与注册中心（6）

### 1. 问：Dubbo、Spring Cloud、gRPC、Thrift 怎么选？

答：优先按“协议性能、生态成熟度、团队技术栈”选型。Java 内网高性能调用常见 Dubbo/gRPC；跨语言统一协议常用 gRPC/Thrift；若团队已重度 Spring 生态且治理组件齐全，Spring Cloud 落地成本最低。  
追问：如果核心链路从 HTTP 改 gRPC，网关、监控和鉴权要改哪些点？

### 2. 问：RPC 一次调用的完整链路是什么？

答：调用方先通过代理拦截本地接口，序列化请求并做服务发现、负载均衡，再经网络传输到服务端；服务端反序列化后执行业务逻辑并返回结果，客户端再做反序列化、超时控制和重试决策。  
追问：链路里最常见的性能瓶颈在哪三处，如何快速定位？

### 3. 问：服务注册中心的核心职责是什么？

答：核心是“服务实例元数据维护 + 健康检查 + 发现路由”。提供者注册并周期上报心跳，消费者订阅实例变更并更新本地路由表。注册中心本身要保障高可用、弱依赖和故障时可降级读本地缓存。  
追问：注册中心短时不可用时，消费者如何避免全链路雪崩？

### 4. 问：RPC 的超时、重试、熔断怎么配？

答：先按下游 P99 延迟设超时，再设置有限次重试并加抖动退避，避免放大故障；对非幂等写请求默认不自动重试。熔断要按错误率和慢调用触发，并配合半开探测恢复，防止长期黑洞。  
追问：为什么“重试次数越多越危险”，在高峰时如何止损？

### 5. 问：序列化协议取舍看什么？

答：看四点：性能、兼容、可调试、跨语言。Protobuf 体积小性能高且跨语言强；JSON 可读性好但体积大；Hessian/Java 原生序列化易用但版本兼容和安全风险要额外治理。生产上更偏 Protobuf。  
追问：线上灰度升级时，如何做字段向前/向后兼容？

### 6. 问：RPC 调用失败的排查顺序是什么？

答：先看调用维度监控：超时率、错误码、重试率；再分层排查 DNS/注册发现、网络连接池、线程池和下游依赖。最后结合 TraceId 还原单次请求路径，区分“流量突增、代码回归、下游抖动”三类根因。  
追问：同样是超时，如何快速判断是网络抖动还是线程池打满？

高频连问路径：

1. 选型对比 -> 链路细节 -> 瓶颈定位
2. 注册发现 -> 心跳机制 -> 故障降级
3. 超时重试 -> 重试风暴 -> 熔断恢复

## 2. 网关与流量治理（6）

### 1. 问：网关的职责边界是什么？

答：网关负责统一入口治理：鉴权、路由、限流、灰度、审计和协议转换；不应承载重业务逻辑。原则是“控制面集中、业务面下沉”，避免网关演化成超级单体，导致发布风险和性能瓶颈同时上升。  
追问：哪些逻辑必须放网关，哪些必须下沉到业务服务？

### 2. 问：动态路由一般怎么实现？

答：常见做法是路由配置中心化存储，网关实例通过订阅推送或定时拉取刷新本地路由表，刷新过程走版本号与原子替换，避免半更新状态。路由规则要支持回滚，且变更全链路可审计。  
追问：如何保证路由热更新时“无中断 + 不丢请求”？

### 3. 问：网关如何扛住 10w QPS？

答：核心是无状态横向扩容 + 轻量过滤链 + 连接池优化。数据面用 Nginx/Netty 低开销处理，热点校验本地缓存化；慢逻辑异步化。再配合 CPU 绑核、零拷贝、限流和熔断，保障高峰时延可控。  
追问：在网关里做 JWT 鉴权时，如何降低验签 CPU 消耗？

### 4. 问：网关鉴权与签名校验怎么设计？

答：先做身份鉴别（JWT/OAuth2/API Key），再做权限判定（资源级 ACL/RBAC），最后校验重放防护（时间戳+nonce+签名）。敏感接口要有细粒度审计日志，确保“谁在何时访问了什么资源”可追踪。  
追问：如果 token 被盗用，网关层有哪些快速止血手段？

### 5. 问：限流、熔断、降级在网关怎么配合？

答：限流负责入口削峰，熔断负责隔离故障下游，降级负责快速返回兜底结果。三者顺序通常是“先限流再转发，转发前后看熔断状态，失败后走降级”。治理指标要按租户、接口、区域三维拆分。  
追问：如何避免把所有流量都打成 429 导致业务全面不可用？

### 6. 问：灰度发布在网关层怎么落地？

答：基于请求标签做流量分组，如用户ID、Header、地域或版本号，先小流量导入新版本并实时观测错误率和延迟。灰度策略必须可一键回切，且与配置变更解耦，防止发布与路由事故叠加。  
追问：灰度过程中出现“部分用户会话漂移”如何解决？

高频连问路径：

1. 职责边界 -> 动态路由 -> 热更新一致性
2. 高并发优化 -> 限流熔断 -> 降级策略
3. 灰度发布 -> 监控阈值 -> 一键回滚

## 6. Service Mesh与Istio（6）

### 1. 问：为什么需要 Service Mesh？

答：Mesh 把重试、限流、熔断、可观测和 mTLS 等通用治理能力从业务代码中抽离，降低多语言系统治理成本。业务团队聚焦领域逻辑，平台团队统一治理策略，减少重复造轮子和实现不一致。  
追问：在什么规模下引入 Mesh 才更划算？

### 2. 问：Sidecar 模式的收益与代价是什么？

答：收益是能力标准化、流量可编程、可观测统一；代价是资源开销增加、链路复杂度上升、排障层次变深。Sidecar 适合治理需求强且服务数量大的系统，小规模系统可能得不偿失。  
追问：如何量化 Sidecar 带来的时延和成本增量？

### 3. 问：Istio 的核心能力有哪些？

答：核心是流量治理（路由、灰度、熔断）、安全（mTLS、身份认证、授权）和可观测（指标、日志、追踪）。控制面下发策略，数据面执行策略。实际落地要严格做版本治理和配置审计。  
追问：为什么 Istio 配置错误会导致全局性故障？

### 4. 问：Mesh 和网关的边界怎么划分？

答：网关主要处理南北向流量（外部到内部），Mesh 主要处理东西向流量（服务间通信）。两者都可做流量治理，但职责不同：网关更偏入口安全与聚合，Mesh 更偏服务间治理一致性。  
追问：同一条流量在网关和Mesh都限流会有什么问题？

### 5. 问：引入 Mesh 后排障方式有什么变化？

答：排障从“应用日志单点定位”升级为“应用 + Sidecar + 控制面”三层联动。要同时看业务指标、代理日志、配置下发状态和证书链路。很多问题不是代码 bug，而是治理配置或策略冲突。  
追问：最常见的 Mesh 误配置导致故障有哪些？

### 6. 问：哪些场景不建议上 Mesh？

答：服务规模小、治理诉求弱、团队运维能力不足时不建议上 Mesh。因为它会引入额外复杂度与成本。若仅需少量治理能力，可先用 SDK/网关方案，等规模和多语言需求上来再演进。  
追问：如果不引入 Mesh，如何先做“轻量治理”过渡？

高频连问路径：

1. 引入动机 -> 收益代价 -> 适用边界
2. Istio能力 -> 配置治理 -> 故障案例
3. 网关边界 -> 责任划分 -> 冲突消解

### Dubbo调用底层实现

消费者侧通常会经历这些层次：
1. **动态代理**：调用方面向接口编程，底层通过代理对象拦截方法调用。
2. **负载均衡**：从服务列表里选一台机器，例如 random、consistent hash 等策略。
3. **协议层**：决定走 dubbo / rmi / hessian / http 等协议。
4. **网络通信**：底层常基于 Netty / Mina 做 NIO 异步通信。
5. **序列化**：把请求编码成二进制后发出去。

服务端则做反序列化、协议解析、网络收包，再通过代理调到真正的本地服务实现。

Dubbo 常见集群容错策略：
- `failover`：失败自动切换并重试，常用于读操作；
- `failfast`：失败立刻返回，常用于非幂等写操作；
- `failsafe`：失败忽略，适合日志这类非关键调用；
- `failback`：失败后后台重发，适合消息通知类；
- `forking`：并行调用多个 provider，一个成功就返回，换时延但更耗资源。

Dubbo 的 SPI 机制本质是：运行时按配置动态装配接口实现类。比如协议、序列化、代理、负载均衡这些扩展点，都可以通过 SPI 替换。

服务治理重点：
- 自动生成调用链路；
- 统计访问量、时长、可用率；
- 监控失败率并报警；
- 做服务分层，避免循环依赖；
- 配置超时、重试和降级策略。

### 原文内容

### 如何设计一个 RPC 框架

最小闭环通常包括：
1. **动态代理**：把本地接口调用代理成远程调用；
2. **注册中心**：保存服务注册信息，并支持服务发现；
3. **负载均衡**：从多个 provider 中选路；
4. **协议与序列化**：定义请求格式和编解码；
5. **网络通信框架**：通常用 Netty 实现高并发连接处理。

### Dubbo 和 Spring Cloud 的差异

- Dubbo 更偏 **高性能 RPC 框架**，基于 TCP，调用更轻量；
- Spring Cloud 更偏 **整套分布式系统解决方案**，组件更全，和 Spring 生态结合更深；
- 关键差异在于：一个偏“RPC 框架”，一个偏“治理生态 + HTTP 体系”。

### 注册中心比较：Eureka / ZooKeeper

1. **ZooKeeper**
- 偏 `CP`，强调一致性；
- 服务列表变更可通过 watcher 快速通知客户端；
- 缺点是服务实例很多时，推送压力会很大。

2. **Eureka**
- 偏 `AP`，优先保证可用；
- 节点间 peer 同步，消费者周期性拉取注册表；
- 默认配置下变更感知偏慢，但可通过缩短缓存同步、心跳和拉取周期来优化。

3. **大规模注册中心优化思路**
- 服务注册表做分片存储；
- 节点主从备份，保证高可用；
- 消费者通过代理层按需拉取部分注册信息；
- 避免所有节点保存全量数据导致同步风暴。

### 网关

网关的职责通常包括：
- 动态路由、负载均衡；
- 灰度发布；
- 统一鉴权；
- 统一限流、熔断、降级；
- 响应缓存；
- 对外暴露统一入口。

常见选型：
- `Kong`：基于 Nginx / OpenResty，插件丰富，抗高并发强；
- `Zuul`：Java 技术栈友好，但高并发能力一般；
- `自研网关`：用 Java / Servlet / Netty 自己构建，灵活度最高，但成本也最高。

动态路由常见做法：
- 路由关系存配置中心或数据库；
- 网关实例通过订阅或定时刷新加载本地路由表；
- 发布配合灰度策略和回滚开关。

如果面试官追问“网关如何扛 10w QPS”，可以继续强调：
- 无状态水平扩容；
- Nginx/Netty 低开销处理链；
- 本地缓存热点鉴权数据；
- 限流熔断前置；
- 机器规格和前置 LVS/Nginx 负载均衡配合。

### ZooKeeper

ZooKeeper 是典型的分布式协调系统，常见用途包括：
- 分布式锁；
- 集群元数据存储；
- Master 选举；
- 分布式协调与通知。

集群角色通常有三种：
- `Leader`：负责写；
- `Follower`：同步数据并提供读；
- `Observer`：只读，不参与选举。

客户端和 ZooKeeper 之间会建立 TCP 长连接，也就是一个 `session`。只要客户端在 `sessionTimeout` 内重新连回任一台 ZK 节点，就可以延续原来的 session；否则临时节点会失效。

ZK 的数据模型是 `znode` 树：
- **持久节点**：适合放元数据；
- **临时节点**：连接断开就消失，适合分布式锁、服务在线状态、协调通知。

Watcher 是 ZK 非常核心的能力：客户端可以对 znode 注册监听，节点变化时收到回调通知。

### ZAB 协议

ZAB（ZooKeeper Atomic Broadcast）可以理解成“带顺序保证的主从同步协议”：
- Leader 收到事务请求后，先写本地日志；
- 再把事务 proposal 按顺序同步给 follower；
- follower 先写磁盘日志，再返回 ack；
- Leader 收到过半 ack 后发出 commit；
- follower 收到 commit 后才真正让新数据可见。

这里的关键点：
- proposal 会带全局递增的 `zxid`，保证顺序；
- 采用过半写机制；
- Leader 崩溃后，只要超过半数节点存活，就能重新选主并恢复服务。

从一致性角度看，ZooKeeper 不是“严格强一致”，更准确地说是**顺序一致性优先、最终收敛**。如果业务强依赖强一致读，可以主动调用 `sync()`。

### 注册中心选型与优化

1. **ZooKeeper vs Eureka**
- ZK 偏 `CP`，一致性更强、变更感知快；
- Eureka 偏 `AP`，可用性更强，但默认感知较慢；
- Eureka 可以通过缩短缓存同步、心跳、拉取周期来把感知延迟从几十秒压到几秒。

2. **大规模注册中心优化**
当服务实例规模上万时，常见优化思路是：
- 注册表做分片存储，不让每台机器存全集；
- 每个分片再做主从备份保证高可用；
- 消费者通过代理层按需拉取局部注册信息；
- 避免每次变更都在全集上广播和同步。

3. **Redis 能不能当注册中心**
可以，但更像工程折中方案：
- Provider 把实例信息写到 Redis Hash，并设置 TTL；
- 周期性刷新过期时间作为心跳；
- Consumer 周期性扫描服务前缀拉取列表；
- 多集群部署时再通过路由逻辑分摊写入和读取压力。

适合轻量级、内部可控场景；如果要求强一致注册发现和丰富 watcher 语义，ZooKeeper / etcd 仍然更标准。
