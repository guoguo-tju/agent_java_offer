## Redis（8）

### 1. Redis 为什么快？

核心原因是基于内存、单线程事件循环减少锁竞争、并使用高效数据结构（哈希表、跳表、压缩结构）。再叠加 I/O 多路复用和合理 pipeline，单机可实现很高吞吐和低延迟。

### 2. redis中的数据类型, 用于计数的HyperLogLog , 用于支持存储地理位置信息的Geo.

1. `HyperLogLog`（去重计数）

- 作用：做“基数统计”，比如 UV、独立设备数，结果是近似值。
- 优势：内存非常省（每个 key 大约固定 12KB），适合海量去重计数。
- 代价：有误差（标准误差约 `0.81%`），不能拿来做精确计费。
- 常用命令：`PFADD`、`PFCOUNT`、`PFMERGE`。

1. `Geo`（地理位置）

- 作用：存经纬度并做“附近的人/店/骑手”查询。
- 底层：基于 `Sorted Set` + `GeoHash` 编码。
- 常用命令：`GEOADD`、`GEODIST`、`GEOSEARCH`（新版推荐）。
- 场景：外卖附近商家、打车最近司机、门店距离排序。

面试一句话：
`HyperLogLog` 解决“海量去重计数省内存”，`Geo` 解决“经纬度存储和附近检索”。前者重统计，后者重空间查询。

### 3. 从海量key里查出某一固定前缀的key

1.低频排查/运维场景
用 SCAN，不要用 KEYS。海量 key 会卡 Redis 主线程。
正确方式：
SCAN 0 MATCH prefix:* COUNT 1000
特点：非阻塞、渐进遍历，但结果不是强一致快照，可能有重复/漏读（业务要容忍）。

2.高频业务查询场景
给key前缀单独建立set/zset建二级索引,写入业务 key 时，同时写二级索引集合. 查询时走 Set/ZSet，不扫全库；再配合 Lua 原子更新、脏索引清理，在海量 key 下稳定运行

### 4. redis的哨兵机制

解决主从模式master宕机后服务不可用的问题, 是一个独立的进程,监控多个master-slave集群,通过多哨兵投票确认故障并自动选主切换，保证服务在主节点故障后可恢复.
高可用用 Sentinel；要横向扩容/分片用 Redis Cluster;

### 5. Redis Cluster是什么

Redis Cluster 是 Redis 官方的分布式方案，核心目标是同时解决横向扩容和高可用, 更适合大数据量和高并发场景. 将不同的key通过某个规则分散存储在多个redis节点上,即数据分片.redis-client采用无中心结构,每个节点都和其他节点连接,采用Gossip协议传播信息和发现新的节点.

如何将不同的key均匀放在不同的redis集群节点? 
一致性哈希算法: 对2^32取模,将哈希值空间组成虚拟的圆环.按顺时针组织,选用服务的ip或者主机名作为关键字,来进行hash,来确定每台服务器在hash环上的位置, key 顺时针找到第一个节点作为归属, 新增或下线一个节点时，只影响它相邻区间的数据，迁移量显著降低. 
哈希环的数据倾斜问题:在节点很少时会因为节点分布不均匀而造成大量数据集中缓存到某一台服务器上, 为每台服务器计算虚拟节点,均匀分布到环上,多了一步虚拟节点到实体节点的映射,数据定位算法不变.这样相对较少的数据节点也能实现数据的均匀分布.实际应用中将虚拟节点设置为32.

### 6. 缓存击穿如何处理？

击穿是热点 key 失效瞬间并发打到 DB, 治理核心是把“多线程重建”收敛成“单线程重建”，其余线程要么等、要么快速返回. 常用方案是互斥重建（分布式锁）:只有一个线程去查库并回填缓存，其他线程等待或快速失败重试、热点永不过期配合主动更新

### 7. 缓存穿透如何处理？

缓存穿透是指请求查询的数据本身不存在，导致每次都 miss 缓存并打到 DB，容易被恶意流量放大。

可举个例子：
查用户 user (line 99999999)，该 ID 不存在。若不处理，10000 并发会都打到 DB。加上三层后：非法 ID 先被拦，合法但不存在的 ID 多数被布隆挡住，少量漏过的请求查库后写入空值，后续直接命中空缓存，不再压 DB。

治理一般做三层防线：
  1.入口参数校验
  先拦非法请求，比如负数 ID、超长字符串、明显异常格式。这层成本最低，能先挡掉一批无效流量。
  2.布隆过滤器（Bloom Filter）把主键集合放进布隆过滤器。查询前先判定：若“肯定不存在”直接返回，不查 Redis/DB。布隆过滤器要跟随数据增量更新，避免“新数据刚写入却被误拦”.
  3.空值缓存（Null Cache）当 DB 查不到时，把空结果也写入缓存，设置短 TTL（如 30s~120s）。防止同一个不存在 key 被反复打 DB。空值 TTL 要短，避免数据刚创建时被旧空缓存影响太久

### 8. 布隆过滤器是什么?

布隆过滤器可以理解为“省内存的存在性判断器”。生产里 Bloom 通常放在“昂贵查询之前做第一层粗筛”，核心价值是用很小内存挡掉大量确定不存在的请求，典型就是缓存穿透治理和海量去重场景。

1. 它是什么

- 用来判断元素“可能存在 / 一定不存在”。  
- 结论特性：  
- 返回不存在：一定不存在。  
- 返回存在：可能存在（有误判）。

1. 实现原理

- 底层是一个位数组（bit array）+ 多个哈希函数。  
- 插入元素时：用 `k` 个哈希函数算出 `k` 个位置，把这些 bit 置为 1。  
- 查询元素时：再算这 `k` 个位置。  
- 只要有一个位是 0 => 一定不存在。  
- 全是 1 => 可能存在。

1. 为什么会误判

- 不同元素可能把同一批 bit 置为 1，导致“本不存在的元素查询时恰好全是 1”。

1. 优缺点

- 优点：极省内存，查询/写入都很快，适合海量 key 预过滤。  
- 缺点：有误判；标准布隆不支持精确删除（可用 Counting Bloom 改进）。

1. 常见场景

- 缓存穿透防护（先判 key 是否可能存在）。  
- 爬虫 URL 去重。  
- 黑名单/白名单预判。

### 9. 缓存雪崩如何处理？

缓存雪崩是“同一时刻大量缓存失效”或“Redis整体不可用”，导致请求集中回源 DB，引发级联故障。治理要分预防和止血两层。

预防层（减少同一时刻失效）

1. TTL 随机抖动：

- 不要都设 30 分钟，改成 `30min + rand(0~5min)`。  
- 作用：把失效时间打散，避免同秒回源峰值。

1. 热点永不过期 + 主动更新：

- 对核心 key 不设 TTL，由 MQ/配置变更主动刷新。  
- 作用：把“被动过期”改成“主动可控更新”。

1. 多级缓存：

- Redis 前再加本地缓存（如 Caffeine），顶住短时流量尖峰。  
- Redis 抖动时本地缓存可当缓冲层。

止血层（雪崩发生时保护 DB）

1. 限流与熔断：

- 对回源接口做 QPS 限流，超过阈值快速失败或排队。  
- DB/下游超时时触发熔断，避免线程池被打满。

1. 降级兜底：

- 返回默认值、旧值、本地快照，优先保证可用性。
- 核心场景可做“只读降级”，暂时关闭非关键写操作。

一句话：  
雪崩处理不是单点技巧，而是“打散过期 + 多级缓存 + 限流熔断 + 降级兜底”组合拳。

### 10. 大 key 风险？

大key的本质问题: 一次命令处理数据太多，会放大 Redis 单线程模型的时延风险；治理关键是“提前发现、结构拆分、分页访问、异步删除”.
大 key 会导致单次操作阻塞事件循环、网络包过大、主从数据迁移和复制耗时长，还会造成集群槽位热点。治理方案: 1.拆分 key（核心）; 2.避免全量读写,用分页命令：HSCAN/SSCAN/ZSCAN、ZRANGE 分页，避免一次全拉; 3.删除用异步

### 11. RDB 与 AOF 取舍？

RDB 是全量快照存储，文件紧凑、恢复快，但可能丢失最近一段数据；AOF 记录写命令存储，数据更完整但文件更大、恢复更慢。生产常用混合策略：RDB 做基线，AOF 做增量保障。

### 12. 分布式锁如何安全释放？

Java 推荐直接用 Redisson（可重入、续期、解锁安全、）.比手写 SET NX PX 更常用:
RLock 支持可重入：同一线程可以重复加锁，不会自己把自己锁死。
自动续期（Watchdog）：业务没执行完时自动延长过期时间，降低“锁提前失效”风险。
解锁安全：内部用 Lua 校验锁持有者，避免误删别人的锁。
等锁更高效：基于 Pub/Sub 通知，减少无脑轮询 Redis。
API 完整：公平锁、读写锁、联锁、多锁等开箱即用。

### 13. 为什么不把所有查询都塞到 Redis？

Redis 适合高频热点和低延迟读，不适合无限扩张成“全量数据库”。全塞进去会带来高内存成本、复杂一致性和失效治理问题，冷热分层和按访问特征选型更可持续。

#### 2.7.1 Redis Hash扩容

Redis 的 hash 结构扩容：
- 扩容：当 hash 内部的元素比较拥挤时（hash 碰撞比较频繁），就需要进行扩容。扩容需要申请新的两倍大小的数组，然后将所有的键值对重新分配到新的数组下标对应的链表中（rehash）。如果 hash 结构很大，比如有上百万个键值对，那么一次完整 rehash 的过程就会耗时很长。这对于单线程的 Redis 来说压力很大，所以 Redis 采用了渐进式 rehash 的方案。它会同时保留新旧两个 hash 结构，在后续的定时任务以及 hash 结构的读写指令中，将旧结构的元素逐渐迁移到新结构中。这样就可以避免因扩容导致的线程卡顿现象。
- 缩容：Redis 的 hash 结构不但有扩容还有缩容，从这一点出发，它要比 Java 的 HashMap 更完整。缩容原理和扩容一致，只不过新的数组大小要比旧数组小一倍。

#### 2.7.2 Redis限流实现

Redis 实现限流：
- 基于 `zset` 实现：限流最核心的是滑动窗口。可以把每次请求都写入一个 `zset`，其中 `value` 保持唯一（可用 UUID 生成），`score` 使用当前时间戳。
- 判断当前时间窗口内的请求数时，只需要统计两个时间戳之间的元素数量。
- 这种方案的优点是窗口平滑、精度高，适合做“最近 N 秒最多允许 M 次请求”的限流规则。
