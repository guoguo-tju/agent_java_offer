## 并发/JUC（8）

### 1. `synchronized` vs `ReentrantLock`？

`synchronized` 语法简单、JVM 优化成熟，适合大多数互斥场景。`ReentrantLock` 提供可中断、超时、公平锁和条件队列，适合需要精细控制并发行为的复杂场景。

可以按“原理 + 场景”这样答：

1. `synchronized` 底层原理

- JVM 内置监视器锁（Monitor）。  
- 对象头里有 `Mark Word`，记录锁状态（无锁/偏向/轻量/重量）。  
- 竞争低时走轻量路径（CAS + 自旋）；竞争高会膨胀成重量级 Monitor，线程阻塞/唤醒由 JVM/OS 协作完成。  
- `monitorenter/monitorexit` 保证互斥和内存可见性（happens-before）。

1. `ReentrantLock` 底层原理

- 基于 JUC 的 AQS（AbstractQueuedSynchronizer）。  
- 用 `state` 表示锁占用，CAS 抢锁。  
- 抢不到就进入 CLH 变种等待队列`LockSupport.park/unpark` 挂起唤醒。  
- 支持可重入、公平/非公平、可中断、超时、多 `Condition`。

1. 生产场景怎么选

`synchronized` 常见场景：  

- 方法级/代码块级简单互斥。  
- 业务内小临界区，逻辑简单，优先可读性和低心智负担。  
- 框架里很多简单线程安全封装。

`ReentrantLock` 常见场景：  

- 需要 `tryLock(timeout)` 防止长时间阻塞。  
- 需要 `lockInterruptibly()` 支持取消任务。  
- 需要公平锁避免饥饿（少量场景）。  
- 需要多个条件队列（生产者消费者多条件唤醒）。  
- 需要结合 AQS 能力做更精细并发控制。

一句话：  

`synchronized` 是 JVM 原生监视器锁，适合“简单可靠”`ReentrantLock` 是 AQS 可编排锁，适合“复杂控制”。

### 2. `volatile` 保证什么？

volatile 保证写入对其他线程可见，并通过内存屏障约束指令重排保证顺序性。它不保证复合操作原子性，所以 i++ 这类读改写仍需锁或原子类

`可以把 volatile 记成两件事：**可见性 + 有序性**，不管原子性。`

`

1. 可见性`

`一个线程写 volatile 变量后，其他线程能尽快看到最新值。`  

`原因是写入会刷新到主内存，读取会从主内存重新取值，而不是长期用线程本地缓存。`

`

1. 有序性（禁止特定重排）`

`volatile 通过内存屏障建立 happens-before：`  

`- 对同一个 volatile 变量，线程 A 的写 happens-before 线程 B 的读。`  

`- 在 volatile 写之前的普通写，不会被重排到它之后。`  

`- 在 volatile 读之后的普通读，不会被重排到它之前。`  

`所以常用于“发布标志位”。`

`

1. 不保证原子性`

`

1. 典型使用场景`

`- 停止标志volatile boolean running。`  

`- 双重检查单例里的 volatile instance（防止指令重排导致半初始化对象可见）。`  

`- 状态位通知（一个线程写，多个线程读）。AQS中的state`

`一句话：`  

`volatile 让“看得见、顺序对”，但“改得准”还得靠锁或原子类。`

### 3. 什么是 AQS？

AQS 是 JUC 的同步器框架，核心是 state + CLH 等待队列 + CAS + park/unpark唤醒，把并发控制从‘每个锁自己造轮子’变成‘只实现状态规则，其余排队唤醒交给框架

1. 它解决什么问题

把“线程竞争资源、排队等待、被唤醒重试”这套通用流程封装起来，业务只要定义“资源怎么拿、怎么放”。

1. 核心组成

- `state`：一个 `int` 状态位，表示同步状态（例如锁是否被占用、剩余许可数）。  
- 等待队列：CLH 变种双向链表，抢不到资源的线程排队。  
- CAS：原子修改 `state`，减少锁竞争。  
- `park/unpark`：挂起与唤醒线程`LockSupport`）。

1. 工作流程（独占锁视角）
2. 线程先 CAS 抢 `state`。
3. 抢到就进入临界区。
4. 抢不到就入队并 `park` 挂起。
5. 持有线程释放资源后 `unpark` 队头线程。
6. 被唤醒线程再尝试 CAS 抢锁。
7. 你要实现 AQS 只需重写什么

- `tryAcquire` / `tryRelease`（独占）  
- `tryAcquireShared` / `tryReleaseShared`（共享）

AQS 负责排队、阻塞、唤醒，你只定义规则。

1. 基于 AQS 的常见组件

- ReentrantLock（独占）
  - 规则：同一时刻只允许一个线程持有锁，可重入。
  - state：重入次数。
  - 额外规则：公平/非公平、可中断、超时获取。
- Semaphore（共享许可）
  - 规则：最多 N 个线程同时通过。
  - state：剩余许可数。
  - 获取就是 state-1，释放就是 state+1。
- CountDownLatch（一次性门闩）
  - 规则：计数没到 0 前，等待线程都阻塞；到 0 一次性放开。
  - state：剩余计数。
  - 特点：一次性，不可重置（要重置用 CyclicBarrier/Phaser）。
- ReentrantReadWriteLock（读写分离）
  - 规则：读可并发，写独占；可重入。
  - state：高低位分别编码读锁计数和写锁计数。
  - 额外规则：读写互斥策略、公平性策略。

### 4. 线程池关键参数？

关键参数是 corePoolSize、maximumPoolSize、workQueue、keepAliveTime、rejectionHandler。调优要结合任务类型（CPU 密集/IO 密集）和 SLA，避免“队列过大掩盖问题”。

1. 过程: 线程池初始时是没有线程的。首先任务过来会去判断当前线程池里面的线程数是否大于核心线程数，如果小于则新建线程执行任务。执行完任务后的核心线程线程会阻塞，等待任务队列中有任务到来。如果任务过来发现当前线程池里面的线程数等于核心线程数，就将任务放在任务队列中，等待有线程来处理。如果任务队列放满了，就去判断当前线程池里面的线程数是否小于最大线程数，如果小于就新建普通线程执行任务，执行完毕之后会等待空闲时间之后 , 会销毁普通线程；如果等于最大线程数(此时任务队列也放满了)，就执行拒绝策略 .
2. 线程池的核心参数配置
  1. core：核心worker数量，线程池会一直维持这个数量的worker，哪怕没有任务。
  2. queue：如果core用完了会先将task放到queue中。
  3. max：如果queue也满了才会再次创建新的worker直到这个max限值。
  4. reject：如果max都满了，那就执行对应的拒绝策略了。
  5. keepalive：超过core数量时，而queue又为空，这个时候多余的woker会等待空闲时间后就被回收掉。

### 5. 为什么常用有界队列？

无界队列看似平滑，实则在高峰会无限堆积任务，导致内存和时延失控。

有界队列能把压力显式暴露出来，配合拒绝策略和降级更符合生产治理思路。

### 6. 死锁如何排查？

死锁排查核心是用 jstack 还原“谁持有什么锁、在等什么锁”的闭环；修复重点是统一锁顺序和减少锁内慢操作

### 7. `CompletableFuture` 常见坑？

常见问题是链路里混入阻塞操作（`join/get`）占满公共线程池，造成线程饥饿和级联超时。实践上要分离线程池、避免在回调中阻塞，并设置超时和异常兜底。

### 8. 异步任务如何降级？

1. 先做任务分级

把异步任务分成三类：  

- 核心必需（失败要影响主流程）  
- 重要可降级（失败可用默认值）  
- 非核心可丢弃（埋点、推荐补充信息）

没有分级就谈不了降级策略。

1. 每个分支必须有超时

异步不等于无限等待。  

给每个分支设置独立超时（如 100ms/300ms），超时立刻返回 fallback，避免拖慢主链路。  

`completeOnTimeoutorTimeout` 都可以。

1. 线程池隔离

不同异步任务用不同线程池，防止一个慢依赖把全局异步池占满。  

同时配置有界队列和拒绝策略（常用 `CallerRuns` 或快速失败），把压力显式化。

1. 熔断 + 限流

当某下游错误率/超时率升高时，快速熔断，短时间直接走降级结果；  

并对高并发流量做限流，避免雪崩放大。

1. 监控与验收

必须监控：线程池队列长度、拒绝次数、熔断状态。  

降级不是事故才开，而是平时就可灰度演练。

面试一句话收口：  

“异步降级的本质是把失败隔离在分支内，不让非核心能力反向拖垮主流程。关键手段是超时、隔离、熔断、限流和可用兜底。”

### 9.CAS是什么

CAS（Compare-And-Swap）是并发里的原子指令思想：

1. 是什么

给一个内存位置 `V`，预期旧值 `A`，新值 `B`：  

- 如果 `V == A`，就把 `V` 改成 `B`（成功）  
- 否则不改（失败）

整个比较+写入是原子的，无需加锁。

1. 在 Java 里的位置

- `AtomicInteger/AtomicLong/AtomicReference` 底层核心就是 CAS。  
- `AQS`、`ConcurrentHashMap`、线程池状态流转等大量用到 CAS。  
- `synchronized` 是悲观锁思路，CAS 是乐观并发思路。

1. 应用场景

- 计数器（高并发自增）  
- 无锁状态机（任务状态 INIT->RUNNING->DONE）  
- 无锁数据结构（栈/队列部分实现）  
- 减少锁竞争，提高吞吐

1. 常见问题

- ABA 问题：值 A->B->A，CAS 以为没变。  
解决：`AtomicStampedReference`（带版本号）。  
- 自旋开销：CAS 失败会重试，冲突高时 CPU 空转。  
- 只保证单变量原子性，多变量一致性仍要锁或事务。

一句话：  
CAS 是 Java 并发里“无锁原子更新”的基础能力，适合高并发下做轻量同步，但高冲突和复杂一致性场景不能只靠它。

## 10. 补充原问：异步任务/视频生成任务如何排队、取消、重试、回调

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容

按“状态机 + 消息队列 + 回调投递器”来设计。

前端提交视频生成请求后，先落任务表，任务状态从 CREATED 进入 QUEUED，投递进入消息队列, 再由 Worker 消费执行，进入 RUNNING，最终收敛到 SUCCEEDED / FAILED / CANCELED / TIMEOUT。

排队：入队时按幂等键来去重，按优先级 + create_time 排序，再根据租户做配额限制，避免大客户把全局资源打满。

取消分两类：任务在排队中直接出队并置 CANCELED，任务已经在运行中走 CANCEL_REQUESTED，Worker 在阶段边界轮询取消标记，安全释放 GPU 和临时文件。

重试只针对可重试错误，比如超时、网络抖动，采用指数退避加抖动，并限制最大次数，超过阈值进死信队列。不可重试错误如参数非法、内容违规，直接失败。

回调采用“至少一次”投递，带签名和时间戳做鉴权，失败重试并入回调死信队列；回调接收方业务方按 `task_id/event_id` 做幂等消费。
