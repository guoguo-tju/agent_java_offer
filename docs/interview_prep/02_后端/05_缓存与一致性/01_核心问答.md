### 6. 缓存击穿如何处理？

击穿是热点 key 失效瞬间并发打到 DB, 治理核心是把“多线程重建”收敛成“单线程重建”，其余线程要么等、要么快速返回. 常用方案是互斥重建（分布式锁）:只有一个线程去查库并回填缓存，其他线程等待或快速失败重试、热点永不过期配合主动更新

### 7. 缓存穿透如何处理？

缓存穿透是指请求查询的数据本身不存在，导致每次都 miss 缓存并打到 DB，容易被恶意流量放大。

可举个例子：
查用户 user (line 99999999)，该 ID 不存在。若不处理，10000 并发会都打到 DB。加上三层后：非法 ID 先被拦，合法但不存在的 ID 多数被布隆挡住，少量漏过的请求查库后写入空值，后续直接命中空缓存，不再压 DB。

治理一般做三层防线：
  1.入口参数校验
  先拦非法请求，比如负数 ID、超长字符串、明显异常格式。这层成本最低，能先挡掉一批无效流量。
  2.布隆过滤器（Bloom Filter）把主键集合放进布隆过滤器。查询前先判定：若“肯定不存在”直接返回，不查 Redis/DB。布隆过滤器要跟随数据增量更新，避免“新数据刚写入却被误拦”.
  3.空值缓存（Null Cache）当 DB 查不到时，把空结果也写入缓存，设置短 TTL（如 30s~120s）。防止同一个不存在 key 被反复打 DB。空值 TTL 要短，避免数据刚创建时被旧空缓存影响太久

### 8. 布隆过滤器是什么?

布隆过滤器可以理解为“省内存的存在性判断器”。生产里 Bloom 通常放在“昂贵查询之前做第一层粗筛”，核心价值是用很小内存挡掉大量确定不存在的请求，典型就是缓存穿透治理和海量去重场景。

1. 它是什么

- 用来判断元素“可能存在 / 一定不存在”。  
- 结论特性：  
- 返回不存在：一定不存在。  
- 返回存在：可能存在（有误判）。

1. 实现原理

- 底层是一个位数组（bit array）+ 多个哈希函数。  
- 插入元素时：用 `k` 个哈希函数算出 `k` 个位置，把这些 bit 置为 1。  
- 查询元素时：再算这 `k` 个位置。  
- 只要有一个位是 0 => 一定不存在。  
- 全是 1 => 可能存在。

1. 为什么会误判

- 不同元素可能把同一批 bit 置为 1，导致“本不存在的元素查询时恰好全是 1”。

1. 优缺点

- 优点：极省内存，查询/写入都很快，适合海量 key 预过滤。  
- 缺点：有误判；标准布隆不支持精确删除（可用 Counting Bloom 改进）。

1. 常见场景

- 缓存穿透防护（先判 key 是否可能存在）。  
- 爬虫 URL 去重。  
- 黑名单/白名单预判。

### 9. 缓存雪崩如何处理？

缓存雪崩是“同一时刻大量缓存失效”或“Redis整体不可用”，导致请求集中回源 DB，引发级联故障。治理要分预防和止血两层。

预防层（减少同一时刻失效）

1. TTL 随机抖动：

- 不要都设 30 分钟，改成 `30min + rand(0~5min)`。  
- 作用：把失效时间打散，避免同秒回源峰值。

1. 热点永不过期 + 主动更新：

- 对核心 key 不设 TTL，由 MQ/配置变更主动刷新。  
- 作用：把“被动过期”改成“主动可控更新”。

1. 多级缓存：

- Redis 前再加本地缓存（如 Caffeine），顶住短时流量尖峰。  
- Redis 抖动时本地缓存可当缓冲层。

止血层（雪崩发生时保护 DB）

1. 限流与熔断：

- 对回源接口做 QPS 限流，超过阈值快速失败或排队。  
- DB/下游超时时触发熔断，避免线程池被打满。

1. 降级兜底：

- 返回默认值、旧值、本地快照，优先保证可用性。
- 核心场景可做“只读降级”，暂时关闭非关键写操作。

一句话：  
雪崩处理不是单点技巧，而是“打散过期 + 多级缓存 + 限流熔断 + 降级兜底”组合拳。

### 12. 分布式锁如何安全释放？

Java 推荐直接用 Redisson（可重入、续期、解锁安全、）.比手写 SET NX PX 更常用:
RLock 支持可重入：同一线程可以重复加锁，不会自己把自己锁死。
自动续期（Watchdog）：业务没执行完时自动延长过期时间，降低“锁提前失效”风险。
解锁安全：内部用 Lua 校验锁持有者，避免误删别人的锁。
等锁更高效：基于 Pub/Sub 通知，减少无脑轮询 Redis。
API 完整：公平锁、读写锁、联锁、多锁等开箱即用。

### 13. 为什么不把所有查询都塞到 Redis？

Redis 适合高频热点和低延迟读，不适合无限扩张成“全量数据库”。全塞进去会带来高内存成本、复杂一致性和失效治理问题，冷热分层和按访问特征选型更可持续。

## 补充原文：Redis / ZooKeeper 分布式锁与高并发优化

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容

### Redis 最普通的分布式锁

最常见的 Redis 分布式锁写法是：

```redis
SET key my_random_value NX PX 30000
```

含义：
- `NX`：只有 key 不存在时才能设置成功；
- `PX 30000`：设置 30 秒自动过期，防止死锁。

释放锁时不能直接 `DEL key`，而是要比对 value 后再删，通常用 Lua 做原子校验：

```lua
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end
```

这里要用 `random_value` 的原因是：
- 客户端 A 拿到锁后如果卡了很久，锁可能已经过期并被客户端 B 重新拿到；
- 如果 A 醒来后直接删 key，就会把 B 的锁误删掉；
- 所以必须用“值匹配 + Lua”保证只删除自己的锁。

但这种最原始实现也有明显问题：
- Redis 单实例会有单点风险；
- 主从异步复制下，如果主节点挂了而锁还没同步到从节点，新主节点可能再次把锁发给别人，产生双持锁问题。

### Redisson 的价值

Redisson 在工程上更常用，因为它补齐了很多手写锁的坑：
- 支持单机、哨兵、Cluster、主从等多种部署；
- 加锁解锁逻辑封装在 Lua 里，天然原子；
- 支持可重入；
- 有 watchdog 自动续期，避免业务没执行完锁就先过期；
- 提供更完整的锁模型和更成熟的工程实现。

### ZooKeeper 分布式锁

ZooKeeper 的锁思路和 Redis 不一样，常见做法是基于**临时顺序节点**：
1. 客户端在某个锁目录下创建临时顺序节点；
2. 如果自己是最小节点，说明拿到锁；
3. 如果不是最小节点，就监听排在自己前面的那个节点；
4. 前驱节点删除后，再尝试拿锁。

这种做法的优点：
- 客户端挂掉后，临时节点会自动删除，锁自动释放；
- 通过监听前驱节点而不是监听锁根节点，可以减少羊群效应；
- 更容易实现公平锁语义。

### Redis 锁 vs ZooKeeper 锁

两者各有取舍：
- Redis 锁性能高，实现轻，适合高频短锁，但底层复制一致性要格外注意；
- ZooKeeper 锁一致性更强、自动释放更自然，但性能和复杂度成本更高；
- 如果业务是高并发热点资源竞争，很多时候更好的方向不是“把锁做得更重”，而是先从业务上拆分资源、降低热点。

### 分布式锁抗高并发

如果一个热点资源每秒有上万请求去抢同一把锁，单把锁会成为瓶颈。常见优化思路是：
- **分段加锁**：把一个大资源拆成多个小分段，每个分段独立加锁；
- **合并扣减**：一个分段不够时，再去锁别的分段做组合扣减；
- **实在极端高并发时，尽量往无锁化演进**：例如直接在 Redis / Tair 这类 KV 里原子扣减，再通过 MQ 异步回写关系型数据库，而不是让所有请求都先抢一把重锁。

面试里可以一句话总结：
**分布式锁是兜底手段，不是高并发系统的首选主路径；能拆热点、能异步、能原子扣减，就尽量不要把吞吐压在一把锁上。**
