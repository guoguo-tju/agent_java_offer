## 分布式设计（8）

### 1. CAP 在后端怎么落地理解？

你这句总结很对。CAP 真正在后端落地时，要抓住一个关键点：

1. CAP只在“发生网络分区(P)”时才要求取舍  

平时网络正常时，很多系统既能高可用也能强一致。  

一旦分区发生，你必须选：  

- 要 `C`：宁可拒绝部分请求，也不返回旧数据  

- 要 `A`：宁可先返回，后续再收敛一致

2. 后端落地不是“系统级二选一”，而是“业务分级”  

同一个公司里通常同时有 CP 和 AP 服务。  

- 交易/资金/库存扣减：偏 `CP`（错一笔代价太高）  

- 内容流/推荐/点赞计数：偏 `AP`（允许短暂不一致）  

- 用户资料展示：常做折中（读AP、关键写CP）



### 2. 最终一致性常见模式？

1. Outbox 事件模式  
本地业务数据和“待发送事件”写在同一个数据库事务里。  
比如订单服务创建订单时，同时写一条 order_created 到 outbox 表。事务提交后，后台任务再把这条事件投递到 MQ。  
这样能解决“订单写成功了，但消息没发出去”的问题。  
它的关键价值是：把“数据库写入”和“事件产生”绑在一个本地事务里。
2. 可靠重试  
消费者收到消息后处理失败，不是直接放弃，而是按规则重试。  
常见做法是：固定重试、指数退避、延迟队列、最大重试次数、失败后进死信队列。  
这适合处理临时性故障，比如网络抖动、下游超时、锁冲突。  
前提是消费逻辑必须幂等，否则重试会把业务做重。
3. 补偿事务  
如果某一步已经成功、后续步骤失败，就执行“反向操作”把系统拉回合理状态。  
比如订单已创建，但库存扣减失败，那就把订单状态改成取消；  
或者支付成功后发券失败，就补发券，实在不行就退款。  
补偿事务不是回滚数据库，而是用业务语义“纠偏”。
4. 定时对账 / 巡检修复  
这是最终兜底。  
即使 MQ 丢了、重试漏了、补偿没跑到，也可以通过定时任务扫描异常状态，把脏数据补回来。  
比如每天扫“已支付但订单未完成”的记录，重新触发履约；  
或者扫“outbox 未投递成功”的事件，重新发送。  
它保证系统不是“出错就永远错下去”。  


如果面试官继续追问，你再补一句：  
“像积分、通知、履约这类链路我会优先用最终一致；资金扣减这类强一致要求高的场景，会更谨慎，尽量缩小异步范围。”

### 3. 为什么幂等是刚需？

网络抖动、超时重试、消息重复投递在分布式系统里是常态，不是异常。没有幂等就会出现重复扣款、重复发券等严重事故，所以必须在业务层做幂等保护。

### 4. 限流该放哪？

网关层适合做全局流量闸门和租户级限流，保护系统入口；

服务内限流用于热点接口、下游依赖和关键资源池。

两层结合才能既保系统又保核心业务。

### 5. 熔断价值是什么？

熔断的核心是“快速失败”，在下游异常时及时切断调用，避免调用方线程和连接被耗尽。

它能把局部故障隔离在边界内，为核心链路争取恢复窗口。

### 6. 隔离舱模式有什么用？

隔离舱是把不同业务或依赖放入独立线程池、连接池、队列，避免互相拖垮。这样某个非核心模块雪崩时，不会占满全局资源影响主交易链路。

### 7. 最小可观测性集合？

最小集合是指标（Metrics）、日志（Logs）、链路追踪（Traces）三件套，并统一请求 ID 贯穿上下游。没有关联 ID，排障和回放基本无法高效完成。

Metrics 负责回答‘系统现在健不健康’，比如 QPS、RT、错误率、CPU、线程池、MQ 堆积；  
Logs 负责回答‘某次请求到底发生了什么’，记录关键业务字段、错误堆栈和上下文；  
Traces 负责回答‘这次请求经过了哪些服务、卡在哪一跳’，适合看跨服务调用链。

工程上通常是 Prometheus/Grafana 做 Metrics，ELK 或 Loki 做 Logs，Jaeger/SkyWalking/Zipkin 做 Traces，核心不是工具名，而是三者通过同一个 requestId 串起来。

### 8. 面试中如何讲取舍？

先讲清约束条件（流量、时延、成本、稳定性），再给出候选方案和评估维度。最后说明为什么选当前方案、放弃了什么、如何兜底，这样最体现工程判断力。

## 7. 分布式事务与秒杀（6）

### 1. 问：2PC、3PC、TCC、Saga 怎么选？

答：2PC一致性强但锁持有长、性能差；3PC改进阻塞但实现复杂；TCC业务侵入高、适合资金等强一致；Saga适合长流程最终一致。选型看一致性等级、延迟目标、研发成本和可补偿能力。  
追问：为什么很多互联网业务最终选 Saga/Outbox 而不是 2PC？

### 2. 问：最终一致性的常见工程解法有哪些？

答：最常见是 Outbox + MQ + 幂等消费 + 定时对账补偿。核心思想是把跨服务事务拆成可重试、可追踪、可回放的本地事务链，避免分布式全局锁。要保障消息可靠投递和重复消费安全。  
追问：Outbox 表积压后会带来什么风险，如何治理？

### 3. 问：秒杀系统核心链路怎么设计？

答：入口先限流防刷，库存预热到缓存，扣减库存走 Redis+Lua 原子化，成功后异步下单到 MQ，订单服务再落库。主链路追求“快速失败 + 快速返回”，非核心动作全部异步化，避免阻塞。  
追问：秒杀下单成功后支付失败，库存和订单如何补偿？

### 4. 问：防超卖策略通常怎么做？

答：缓存层先做预扣库存，数据库层做最终校验与幂等兜底。库存扣减必须原子，常用 Lua 脚本。对超卖敏感业务可加排队令牌和乐观锁版本控制。核心是“多层防线”，不能只信单层缓存。  
追问：只在 Redis 扣库存不回写 DB 会有什么后果？

### 5. 问：削峰填谷为什么离不开 MQ？

答：MQ 把突发请求转成可控消费速率，隔离前台高峰和后台处理能力差异。生产端快速写入，消费端按能力平滑处理，并可通过优先级队列保障核心业务先执行，防止链路被低价值流量挤占。  
追问：积压严重时，优先扩消费者还是先降级，判断依据是什么？

### 6. 问：高峰故障时如何做限流、降级、回滚？

答：先入口限流和熔断止损，再关闭非核心功能，保主链路可用；对异常版本一键回滚，结合灰度分批恢复。全过程要有实时监控和开关中心，保障“可观测、可操作、可恢复”。  
追问：回滚后为何还可能持续报错，下一步怎么排查？

高频连问路径：

1. 事务选型 -> 一致性等级 -> 补偿策略
2. 秒杀链路 -> 防超卖 -> 异步下单
3. 高峰止损 -> 降级回滚 -> 复盘优化

### 分布式事务常见方案

1. **XA / 两阶段提交（2PC）**
- 事务管理器先询问每个参与方能否执行；
- 第一阶段预留资源，第二阶段统一提交或回滚；
- 一致性强，但严重依赖数据库层面的锁和协调，不适合高并发场景。

2. **TCC（Try / Confirm / Cancel）**
- Try：检测资源并预留；
- Confirm：真正执行业务；
- Cancel：失败时执行补偿回滚。
- 优点是可实现较强一致；缺点是业务侵入大、补偿逻辑复杂。
- 更适合资金类、对一致性要求很高、且每个环节执行时间较短的场景。

3. **本地消息表**
- A 系统在本地事务里同时写业务数据和消息表；
- 后台线程轮询消息表，再调用 B 系统；
- B 系统必须幂等。
- 缺点是高并发下消息表本身会成为数据库压力点。

4. **可靠消息最终一致性**
- A 先向 MQ 发送 prepared / half message；
- 本地事务成功后再发 confirm / commit；
- 如果 confirm 丢了，MQ 通过回查机制再次确认本地事务状态；
- B 消费消息时需要幂等、失败重试和补偿。
- 这种方案更适合订单、库存、履约这类允许最终一致的链路。

### 订单场景里的可靠消息最终一致性

典型流程：
1. 先发 half message；
2. MQ 落盘成功后再走核心订单链路；
3. 核心链路成功则发 commit，失败则发 rollback；
4. MQ 在等待期间可通过回查机制确认事务状态；
5. 消费者收到消息后执行下游逻辑，失败则重试或进入补偿。

### Seata / TCC 框架理解

Seata 的角色通常分为：
- `TC`：协调器，维护全局事务状态；
- `TM`：事务发起者，负责开始/提交/回滚全局事务；
- `RM`：资源管理器，管理各分支事务并向 TC 汇报。

它的核心流程是：
- TM 向 TC 申请一个全局事务 ID；
- 各分支事务把自己注册到这个全局事务里；
- 成功则统一提交，失败则由 TC 驱动各分支回滚。

面试里可以补一句：
- 强一致核心链路更适合 TCC；
- 大部分互联网业务更常用“可靠消息 + 幂等 + 补偿 + 对账”这套最终一致方案。

### 原文内容

### 接口幂等性

接口幂等的核心目的，是防止因为重试、超时、网络抖动等原因导致“同一业务请求被执行多次”，出现重复下单、重复扣费、重复发券。

常见做法要分场景：

1. **插入型请求**
- 最直接的方法是数据库唯一键；
- 例如用订单号、业务流水号、用户+业务主键作为唯一约束；
- 如果重复插入，直接被数据库挡住。
- 也可以在 Redis 里用“接口 + 请求参数”拼一个唯一 key，先判断是否处理过。

2. **更新型请求**
- 更新比插入更复杂，因为不只是“有没有处理过”，还要区分“正在处理”还是“已经处理完成”。
- 一种可讲的思路是把请求处理拆成多个阶段，在每个阶段配状态位：
  - 执行前检查 Redis 幂等 key；
  - 如果值是 `1`，说明已经成功执行，直接返回；
  - 如果值是 `0`，说明有同请求正在执行，可以等待或快速失败；
  - 执行过程中如果异常，不修改成功标记；
  - 执行成功后再把 key 改为完成态。
- 对一些强一致更新，也可以配合数据库独占锁，例如 `select ... for update nowait`，确保同一条数据同时只被一个请求修改。

3. **工程化设计**
- 幂等键要有明确生成规则；
- 幂等状态要设置合理 TTL，避免永久脏状态；
- 日志里要能区分“首次执行、重复命中、处理中重试”；
- 高风险写操作要把幂等设计成平台能力，而不是每个接口各自手写。

面试里可以总结成一句：
**幂等不是简单“查一下有没有”，而是要区分“未执行、执行中、已完成”三种状态，并根据插入型、更新型请求分别设计。**
