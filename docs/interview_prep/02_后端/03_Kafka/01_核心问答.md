## MQ/Kafka（8）

### 1. 至少一次与恰好一次怎么选？

恰好一次语义实现复杂、成本高，跨系统链路很难端到端保证。多数业务采用“至少一次投递 + 消费端幂等”，在可控复杂度下获得足够的可靠性。

### 2. 消费幂等如何做？

核心是为每条业务事件定义唯一幂等键（订单号、业务流水号等），消费前先查重。常见实现是去重表、唯一索引、redis分布式锁，确保重复消息不会重复生效。在线上先用 Redis SETNX 做高性能幂等，降低重复消费压力；对资金/发券等关键链路，再用数据库唯一键做最终兜底，避免 Redis 异常导致重复执行

### 3. 消息积压怎么处理？

先定位根因: 判断是生产激增还是消费变慢，再看是否有慢分区或下游阻塞。
先止血: 处理上用扩容消费者、按优先级分流、临时降级非核心业务.
再做回补，避免二次冲击: 积压清理不能“满速回放”，要限速回补。单独开回补 consumer group，配合令牌桶限速，保护 DB 和下游接口。
最后补齐监控和容量治理，确保同类问题不再反复出现

### 4. 为什么按业务 key 分区？

按orderId、userId业务 key 分区能保证同一 key 的消息进入同一分区，天然保持局部有序。与此同时，不同 key 可并行消费，既满足顺序要求又保留水平扩展能力。

### 5. MQ生产端可靠性怎么配？

我会先保证不丢，再控制重复和乱序，最后再调批量与压缩拿吞吐
Kafka 生产端通常要配置 `acks=all`、合理重试和超时，提升写入可靠性。再结合批量发送与压缩提升吞吐，同时设置最大 in-flight 等参数控制乱序和重复风险。

### 6. 死信队列价值？

死信队列用于隔离多次重试仍失败的“毒消息”(数据格式错、字段脏、下游永久失败、代码 bug)，防止主队列被反复阻塞。它还能保留问题现场，便于人工修复、补偿重放和规则迭代。

### 7. 端到端不丢消息怎么做？

生产端要确保发送成功确认(Kafka 常配 acks=all、enable.idempotence=true,生产端的幂等允许重复发送)，Broker 侧做持久化和多副本同步；
消费端严格执行“处理成功再提交ack位点”。同时配合幂等与消费失败补偿机制(消费失败走重试队列或延迟重试，超过阈值进 DLQ，不能无限阻塞主链路)，才能在重试场景下实现“逻辑不丢不重”。

### 8.Kafka消息去重保证
Kafka 的“去重保证”要分三层理解。
第一层是生产端去重：开启 enable.idempotence=true，并配合 acks=all、合理重试，Kafka 会用生产者ID和序号在分区内去重，防止网络重试造成重复写入。
第二层是 Kafka 内部端到端一致：如果是“消费一条再生产一条”的链路，可用事务（transactional.id）加 read_committed，做到 Kafka 到 Kafka 的 Exactly-Once。
第三层是最容易被忽略的：一旦消费后要写数据库、发券、扣库存，Kafka 本身不能替业务系统去重，必须做业务幂等。常见做法是给每条消息带全局唯一 eventId，消费端落幂等表或用唯一索引（如订单号唯一），重复消息直接丢弃；并坚持“处理成功再提交 offset”。

### 9.Kafka为什么高性能
主要是在持久化的那个性能做了一些优化。
1.顺序写磁盘：避免随机 I/O，吞吐远高于随机写。
2.Page Cache：大量读写走操作系统缓存，减少磁盘真实访问。
3.零拷贝（sendfile/transferTo）：数据从内核缓存直接到网卡，减少用户态拷贝和 CPU 开销。
4.批量处理：生产端批量发送、Broker 批量落盘、消费端批量拉取，显著降低系统调用次数。

### Kafka 的高可用

Kafka 是分布式的，topic 会被拆成多个 partition，分布在不同机器上。每个 partition 会有一个 leader 和多个 follower：
- 生产者只向 leader 写；
- follower 主动从 leader 拉数据；
- leader 宕机后，follower 中会重新选出新的 leader，继续提供服务。

写入流程可以这么理解：
1. 生产者把消息发给 leader；
2. leader 先写本地磁盘；
3. follower 从 leader 同步数据；
4. 当满足副本确认条件后，leader 才给生产者返回 ack。

如果要尽量避免 broker 故障导致的数据丢失，常见配置是：
- `replication.factor > 1`
- `min.insync.replicas > 1`
- `acks=all`
- `retries` 设为足够大

这套配置的核心思想是：**至少要有多个副本真正跟上，并且所有必要副本确认后，才把消息视为写成功。**

### 重复消费与幂等

Kafka 消费端可能重复消费，本质原因是：
- 业务已经处理成功；
- 但 offset 还没来得及提交；
- 消费者重启后又会从旧 offset 再拉一次。

所以常见工程做法是：
- 关闭自动提交 offset；
- 处理成功后手动提交；
- 同时在业务层做幂等。

业务幂等常见做法：
- 用数据库唯一键防重；
- 或者用 Redis / 内存 Set 做去重判断。

### Kafka 如何保障不丢消息

1. **生产端**
- 要拿到 broker 的成功确认；
- 失败时要自动重试。

2. **Broker 端**
- 做持久化和多副本复制；
- leader 切换时只从满足同步条件的副本中选举。

3. **消费端**
- 处理成功再提交 offset；
- 失败时允许重试，但业务必须幂等。

### Kafka 如何保证顺序性

Kafka 的顺序性是“分区内有序，不是全局有序”。
常见做法是：
- 生产时按业务 key（如订单 ID）路由到同一个 partition；
- 消费时避免同一个 key 被多个线程并发打乱；
- 如果消费者内部多线程处理，可再按 key hash 到本地内存队列，由单线程顺序消费。

所以面试里可以一句话总结：
**Kafka 能保证同一 partition 内的消息顺序；要把业务顺序映射到 Kafka 顺序，关键是分区键设计和消费端串行处理。**
