## 5. 网络I/O与Netty（6）

### 1. 问：BIO、NIO、AIO 的核心区别是什么？

答：BIO 是同步阻塞，一连接一线程；NIO 是同步非阻塞，基于事件轮询处理多连接；AIO 是异步非阻塞，由系统完成 I/O 后回调。高并发网络服务通常采用 NIO/Netty，平衡性能与工程复杂度。  
追问：Java 生态里为什么 AIO 在业务系统里不如 NIO 常见？

### 2. 问：I/O 多路复用原理怎么讲？

答：多路复用通过一个线程监听多个连接的就绪事件（读/写/异常），只在事件到达时分发处理，避免线程被阻塞空耗。Linux 常见实现是 select/poll/epoll，其中 epoll 在高并发连接下更高效。  
追问：epoll 相比 select 的核心优化点是什么？

### 3. 问：Netty 如何体现 Reactor 模型？

答：Netty 通常采用主从 Reactor：Boss 线程负责 accept，新连接分配给 Worker 线程处理读写事件。业务处理可再下沉到业务线程池，避免 I/O 线程被阻塞。这样实现了连接管理和业务计算解耦。  
追问：如果在 I/O 线程里做重计算，会出现什么线上表现？

### 4. 问：粘包/半包问题如何处理？

答：TCP 是字节流，不保留消息边界，可能出现粘包半包。解决方式是自定义协议边界：固定长度、分隔符、长度字段。Netty 常配合 LengthFieldBasedFrameDecoder 等解码器确保消息按帧还原。  
追问：长度字段被恶意构造时，如何防止内存攻击？

### 5. 问：零拷贝和 mmap 有什么价值？

答：核心是减少用户态与内核态之间的数据拷贝和上下文切换。sendfile/mmap 能显著提升大文件传输和高吞吐场景性能。Kafka、Nginx 等都依赖这些能力，但要注意页缓存行为和内存占用。  
追问：零拷贝能否完全避免 CPU 参与，为什么？

### 6. 问：Netty 堆外内存泄漏怎么排查？

答：先看 direct memory 指标和 Full GC 行为，再启用泄漏检测（ResourceLeakDetector）定位未释放 ByteBuf。排查重点是引用计数对象是否成对 release，以及异常分支是否遗漏释放逻辑。  
追问：线上开启最高级泄漏检测的代价是什么？

高频连问路径：

1. BIO/NIO/AIO -> 多路复用 -> Reactor
2. TCP字节流 -> 粘包半包 -> 编解码
3. 零拷贝 -> 堆外内存 -> 泄漏排查

## 8. OS网络排障与发布治理（6）

### 1. 问：线上 CPU 飙高怎么排查？

答：先用 top 定位高负载进程，再用 top -Hp 定位热点线程，线程ID转16进制后用 jstack 对应堆栈，确认是死循环、锁竞争还是频繁GC。最后结合发布记录和流量变化判断是代码回归还是流量异常。  
追问：如果热点线程都在 WAITING，CPU 为什么还可能高？

### 2. 问：内存与 GC 现场怎么快速定位？

答：先看 GC 日志和 jstat 判断是频繁 Young GC 还是 Full GC，再用 jmap/jcmd 导出堆信息分析大对象和泄漏链。区分是堆内、元空间还是堆外内存问题，定位后再做参数和代码双向修复。  
追问：什么时候该优先抓 heap dump，什么时候先看 NMT？

### 3. 问：网络基础高频题如何简洁回答？

答：三次握手是为确认双方收发能力，四次挥手是全双工独立关闭；HTTP/1.1 解决长连接，HTTP/2 引入多路复用与头压缩，HTTP/3 基于 QUIC 减少队头阻塞。回答要围绕“为什么比上一代更好”。  
追问：为什么 HTTP/2 多路复用下仍可能出现延迟抖动？

### 4. 问：蓝绿、金丝雀、Beta 有什么区别？

答：蓝绿是两套完整环境切换，回滚快但成本高；金丝雀是小流量逐步放量，风险最可控；Beta偏目标人群试用验证。生产上常用“金丝雀 + 指标门禁 + 自动回滚”组合，而不是单一策略。  
追问：大促期间为什么不建议做大比例金丝雀？

### 5. 问：灰度发布的风险控制关键点是什么？

答：关键是“分批、观测、门禁、回滚”。先按用户/区域分批放量，监控错误率、RT、资源和业务指标；设置自动暂停阈值；异常时一键回切。灰度策略必须和配置版本绑定，避免策略漂移。  
追问：灰度看板里你最看重哪3个核心指标？

### 6. 问：变更三板斧怎么落地到工程体系？

答：可灰度是分批放量策略，可监控是发布窗口指标守护，可回滚是快速恢复能力。三者需要统一变更流水线承载，做到“发布前有准入、发布中有门禁、发布后有复盘”，形成闭环治理。  
追问：如果监控阈值设太保守，会造成什么工程副作用？

高频连问路径：

1. CPU高 -> 线程热点 -> 堆栈定位
2. GC异常 -> 内存分类 -> 现场抓取
3. 发布策略 -> 灰度门禁 -> 自动回滚

### 原文内容

### Netty 与 Reactor

Reactor 模型本质是事件驱动分发：
- Reactor 线程负责监听和分发 I/O 事件；
- Handler 负责真正处理读写逻辑。

Netty 常见是主从 Reactor：
- `Boss/MainReactor` 负责接收连接；
- `Worker/SubReactor` 负责连接上的读写事件；
- 非 I/O 的业务逻辑再交给业务线程池，避免 I/O 线程阻塞。

### 堆外内存（off-heap）

使用 `ByteBuffer.allocateDirect()` 可以申请堆外内存。优势是：
- 网络 I/O 或文件 I/O 时，数据本来就会进入堆外区域；
- 如果直接使用堆外内存，就可以减少一次从堆内到堆外的拷贝。

回收要点：
- 可通过 `-XX:MaxDirectMemorySize` 限制最大堆外内存；
- DirectByteBuffer 对象失去引用后，在 GC 时可能回收并释放堆外内存；
- 堆外内存如果持续泄漏，也会触发 OOM。

### mmap 与 sendfile / 零拷贝

普通 I/O 读取磁盘文件再发网络，通常涉及：
- 用户态/内核态多次切换；
- 内核缓冲区 -> 用户缓冲区 -> Socket 缓冲区的多次拷贝。

`mmap` 的核心是把内核缓冲区和用户缓冲区建立映射，减少一次数据拷贝。

`sendfile` 的核心是：
- 数据从磁盘到内核缓冲区；
- 再直接从内核缓冲区交给网卡发送；
- 省掉“先拷到用户态再写回内核”的过程。

所以 Kafka、Tomcat 这类高吞吐系统会大量利用零拷贝来降低上下文切换和 CPU 拷贝成本。
