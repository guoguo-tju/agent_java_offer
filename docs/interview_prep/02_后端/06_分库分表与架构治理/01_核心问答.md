### 12. MySQL的分库分表、读写分离

**核心区别**
1. `分库分表`：解决“单机存储/写入瓶颈”。  
2. `读写分离`：解决“读取压力过大”。  
3. 通常先做读写分离，再做分库分表。

**分库分表**
1. 垂直拆分：按业务域拆库（用户库、订单库）。  
2. 水平拆分：同一表按分片键拆（如 `user_id % 64`）。  
3. 关键点：分片键选得好不好决定热点和扩容难度。  
4. 常见问题：跨分片 `join/count/page`、分布式事务、全局ID。  
5. 工程做法：业务层聚合/冗余字段/汇总表，ID用雪花或号段。  
6. 扩容迁移：双写 -> 回填 -> 校验 -> 灰度切流 -> 下线旧分片。

**读写分离**
1. 架构：主库写、从库读（通过中间件/代理路由）。  
2. 风险：主从延迟导致“刚写就读不到”。  
3. 兜底：写后短时间强制读主、延迟监控、超阈值切主读。  
4. 还要做：从库负载均衡、故障自动切换、GTID/复制健康检查。

## 4. 分库分表与数据治理（6）

### 1. 问：为什么要分库分表？什么时候不该做？

答：当单库在容量、QPS、写锁竞争或维护窗口上到瓶颈时才做分库分表。若业务规模小、查询复杂且强事务多，过早拆分会显著增加复杂度。判断标准是“收益是否大于数据治理成本”。  
追问：分库分表前你会先做哪三件低成本优化？

### 2. 问：垂直拆分和水平拆分怎么选？

答：垂直拆分按业务域解耦，解决的是职责和团队边界问题；水平拆分按数据量扩展，解决的是容量和吞吐问题。常见路径是先垂直再水平，避免把单体问题直接放大成分布式问题。  
追问：仅做垂直拆分为什么往往解决不了热点写入问题？

### 3. 问：分片键怎么选才能避免热点？

答：分片键要兼顾写入均匀和查询命中。优先选择高基数、稳定且查询常带条件的字段；避免时间戳这类天然热点键。可用“业务键 + 随机扰动”降低热点，并结合热分片迁移机制。  
追问：如果业务天然按商户聚集，热点商户怎么做专项治理？

### 4. 问：跨分片 join、分页、排序、count 怎么处理？

答：原则是尽量避免跨分片实时计算。常见做法是业务侧聚合、异步宽表、搜索引擎或OLAP承接复杂查询。分页走“先路由再聚合”，深分页用游标。count 用近实时汇总表，不做全量实时扫。  
追问：跨分片查询延迟高时，先改 SQL 还是先改架构？

### 5. 问：分库分表后全局 ID 怎么生成？

答：常见方案是雪花算法、号段模式、数据库序列服务。雪花适合高并发且去中心化，号段适合稳定批量发号。要重点处理时钟回拨、重复ID、趋势递增和可观测性，避免主键冲突事故。  
追问：雪花算法出现时钟回拨时有哪些可落地兜底方案？

### 6. 问：不停机迁移到分库分表怎么做？

答：通常走“双写 + 回填 + 校验 + 切流 + 回滚”五步。先建立新链路并双写，历史数据离线回填，做全量与增量校验后灰度切读切写，保留回滚开关。关键是可观测和幂等补偿链路完备。  
追问：双写阶段如何避免主从数据长期漂移？

高频连问路径：

1. 拆分时机 -> 拆分方式 -> 成本评估
2. 分片键 -> 热点治理 -> 扩容迁移
3. 跨分片查询 -> 聚合策略 -> 数据一致性

### Snowflake 时钟回拨问题

Snowflake 这类趋势递增 ID 方案强依赖机器时钟。如果机器和时间服务器校时后发生时钟回拨，就可能出现：
- 发号重复；
- 或者服务为了避免重复而短时间不可用。

一种工程化兜底思路是：
- 在内存里维护过去一段时间内（例如 1 小时）每一毫秒、每台机器已经发出的最大 ID；
- 如果检测到时钟回拨，就定位回拨到了之前的哪一毫秒；
- 然后在那一毫秒对应的最大序列号基础上继续自增，而不是直接重新从 0 开始发。

这样做的本质是：
**把“时间倒退”问题，转化成“同一毫秒内继续用更大的序列号发号”，从而降低重复 ID 风险。**

当然，工程上还可以结合：
- 小幅回拨时短暂等待；
- 大幅回拨时切换机器号 / 直接熔断告警；
- 结合监控观察时钟漂移和发号失败率。
