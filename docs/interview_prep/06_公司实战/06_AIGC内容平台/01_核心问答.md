## 1. AIGC内容平台

### 资料说明

### 原文内容

你可以重点准备下面这套“爱诗版”题库：

讲一个你做过的复杂系统，怎么拆服务、做幂等、做降级。

Redis 缓存击穿、雪崩、双写一致性怎么处理。

- 双写一致性建议：**先更新 DB，再删除缓存**（不是更新缓存）。

MySQL 索引、回表、MVCC、事务隔离在真实业务里怎么落地。

- **事务隔离怎么选（先定一致性级别）**
  - 读多写少、报表类：常用 READ COMMITTED，减少锁冲突。
  - 交易/库存/余额：常用 REPEATABLE READ（MySQL 默认），必要时配合行锁与唯一约束。
  - RR 能保证同一事务内快照读一致，避免读来读去结果变动。InnoDB 在 RR 下对 for update / update 走当前读+加锁，能抑制并发写冲突。
  - **快照读 = 稳定看历史；当前读 = 锁住看现在并准备改。**



Kafka 如何保证至少一次、顺序、去重。

跨地域多机房部署，如何做流量切换和容灾。

海外/弱网环境下，接口超时、重试、熔断怎么设计。

SSE、WebSocket、长轮询在 AI 流式返回里怎么选。

大模型返回很慢或失败时，前后端怎么协同降级。

异步任务/视频生成任务如何排队、取消、重试、回调。

按“状态机 + 消息队列 + 回调投递器”来设计。

前端提交视频生成请求后，先落任务表，任务状态从 CREATED 进入 QUEUED，投递进入消息队列, 再由 Worker 消费执行，进入 RUNNING，最终收敛到 SUCCEEDED / FAILED / CANCELED / TIMEOUT。

排队 入队时按幂等键来去重, 按 优先级 + create_time 排序, 再根据租户做配额限制，避免大客户把全局资源打满。

取消分两类：任务再排队中直接出队并置 CANCELED，任务已经在运行中走 CANCEL_REQUESTED，Worker在阶段边界轮询取消标记，安全释放GPU和临时文件。

重试只针对可重试错误，比如超时、网络抖动，采用指数退避加抖动，并限制最大次数，超过阈值进死信队列。不可重试错误如参数非法、内容违规，直接失败。

回调采用“至少一次”投递，带签名和时间戳做鉴权，失败重试并入回调死信队列；回调接收方业务方按 task_id/event_id 做幂等消费。

如果做一个 Agent 创作助手，工作流怎么编排、状态怎么保存、怎么观测。

工作流可以用推荐图或者状态机，大致包括这么几个节点。刚开始进来先是意图的识别，然后就是任务的规划。然后这就是分段执分段执行任务，分段执行过程中可以进行检索编排。然后再进行任务的审审校核对。然后就进行任务，如果核对不通过，可能要根据沈教的建议，再做一一版修改。之后是人工审核节点。然后是做任务的发布。

状态怎么保存？状态可以理解为就是记忆。这里分短期记忆和长期记忆。短期记忆就是在一次上下文会话过程中需要使用的信息。它对，它可以保存到内存里，也可以保存到 Redis 里。如果是长期记忆的话，比如说用户的特用户的特征、行为方式，或者是专业知识类似的东西可以考虑存到向量检索，或者是买次口这样的关系型数据库里。对，如果是一些日志行为类的，可以像参考龙虾，存到本地的 MD 文件里。

怎么观测？一般可观测的最小三件套，一个是 log 日志，一个是 metric 指标，然后是 trace 链路追踪这一套。然后具体的观测指标，一个是首先就是系统的稳定性、系统的延迟、P99、P95、CPU 之类的。然后还有一点就是成本的监控，就是 token token 的一个监控。然后是图片排的话，涉及到一些就是执行步骤，执行步数的一些监控。啊，其次还有是业务方向的监控，比如说是业务的成功率，超超时率、失败失败率、失败的错误码的分布情况。其次还有监控整个 Agent 的一个鲁棒性，就是在工具调用失败这个情况下，它是怎么弄呢？然后还观测 Agent 的一个主动性，就是它观察它需要人工介入的一个情况，以及观察它它的一个推理的过程，人工观察它的推理的过程是否符合逻辑，以及最后它如果它是按计划执行，它的执行的过程是否严格的遵守它的计划。

算法:
公开能看到的有“麻将算法实现”“会议室 II”等，说明它不是纯聊天式面试，通常会带至少一道编码题

要。下面这版我按 **“某AIGC内容平台 Java / Agent / 服务端岗”** 来给你整理，优先基于公开能查到的爱诗面经和它当前公开岗位/产品特征来倒推。公开面经里高频出现的是 **项目深挖、Redis/MySQL/Kafka、高可用与跨地域部署、手写题**；而公司公开产品和岗位又明显带有 **视频生成、Agent 创作助手、API 平台、服务端/AIGC 后端** 的特点，所以面试大概率会是 **传统后端基础 + 场景设计 + 一点 AI 应用工程化** 的混合题型。

## 一、项目深挖题

1. **讲一个你做过的最复杂、最能体现价值的项目。**
  答题抓手：业务背景、核心指标、你的职责、技术难点、结果提升。别平铺直叙，要能讲出“为什么难”。
2. **这个项目里，你做的部分如果拿掉，会发生什么？**
  答题抓手：证明你不是打杂。讲你负责链路的不可替代性，比如核心接口、核心模块、核心优化。
3. **项目里最难的一次故障是什么？你怎么定位和止损？**
  答题抓手：现象、排查路径、根因、止血、复盘、长期治理。要体现工程负责人思维。
4. **如果现在让你重做一次这个项目，你会怎么改架构？**
  答题抓手：说出当时的局限，再给出升级方案，比如异步化、缓存分层、配置中心、观测补齐。
5. **你在项目里做过最有技术含量的优化是什么？**
  答题抓手：最好讲性能、稳定性、成本、吞吐四选一，给出前后对比数据。
6. **项目里做过哪些 trade-off？为什么没选别的方案？**
  答题抓手：体现成熟度。比如强一致 vs 最终一致、同步 vs 异步、自研 vs 现成组件。

## 二、Java / MySQL / Redis / MQ 基础设施题

1. **MySQL 索引为什么能加速？什么情况下会失效？**
  答题抓手：B+ 树、回表、覆盖索引、最左匹配、范围查询、函数操作导致失效。
2. **MVCC 是什么？你怎么解释 Read View？**
  答题抓手：版本链、undo log、快照读、当前读。不要只背定义，最好结合你业务里的事务场景说。
3. **事务隔离级别怎么选？你真实工作里用到过哪些？**
  答题抓手：不要只说四个级别，讲清在库存、营销、账户这种场景里怎么选。
4. **Redis 常用数据结构分别适合什么场景？**
  答题抓手：String、Hash、List、Set、ZSet、Bitmap、HyperLogLog。最好带真实业务例子。
5. **Redis 缓存击穿、穿透、雪崩分别怎么处理？**
  答题抓手：互斥锁、逻辑过期、布隆过滤器、随机过期时间、热点隔离、降级兜底。
6. **Redis 和 MySQL 双写一致性怎么保证？**
  答题抓手：延迟双删、binlog 订阅、消息队列异步刷新、读写路径隔离。要敢讲“不可能绝对强一致”。
7. **Redis 哨兵和 Cluster 分别解决什么问题？**
  答题抓手：高可用 vs 横向扩展；主从切换、槽位分片、客户端路由。
8. **Kafka 怎么保证消息不丢、不重、不乱？**
  答题抓手：生产幂等、acks、ISR、消费者提交 offset、业务幂等、分区有序。
9. **消息队列消费失败怎么处理？**
  答题抓手：重试、死信队列、补偿任务、人工介入、幂等键设计。
10. **你会怎么设计一个支持优雅停机的消费者？**
  答题抓手：停止拉取、消费完手头消息、提交 offset、释放资源、优雅关闭线程池。这个方向在公开面经里出现过。

## 三、高可用 / 系统设计题

1. **如果服务部署在多地域机房，怎么做容灾切流？**
  答题抓手：DNS/GSLB、入口层切流、配置中心、灰度、流量染色、降级开关。公开面经里有“新加坡机房挂了怎么切”的类似题。
2. **跨地域调用延迟高、抖动大，你怎么保证链路稳定？**
  答题抓手：就近接入、异步化、超时重试、熔断、批处理、缓存前置。
3. **高并发接口突然被打爆，你的止血顺序是什么？**
  答题抓手：限流、降级、缓存、扩容、隔离、熔断。要有“先保核心链路”的意识。
4. **如何设计一个高可用的异步任务平台？**
  答题抓手：任务状态机、重试策略、优先级队列、幂等、取消、回调、失败补偿。
5. **配置更新如果不能靠 Redis pub/sub 强一致分发，你会怎么做？**
  答题抓手：配置中心长轮询、版本号、灰度发布、本地缓存、强校验。这个思路在公开 Go 面经里有类似影子。
6. **如果一个外部依赖服务很慢，但又不能完全去掉，怎么设计调用链？**
  答题抓手：超时、重试、隔离线程池、熔断、缓存快照、兜底数据。
7. **如何做接口幂等？不同业务类型有何不同？**
  答题抓手：token、防重表、唯一索引、状态机、消息去重。最好用支付、任务提交、批处理举例。
8. **怎么设计可观测性，才能快速定位线上问题？**
  答题抓手：日志、指标、trace、告警、链路 ID、错误码分层、关键业务埋点。

## 四、AI / Agent / 视频生成后端题

1. **如果让你做一个“视频生成任务平台”，后端核心模块怎么拆？**
  答题抓手：任务接入层、参数校验、队列调度、模型服务编排、素材存储、回调通知、计费额度、审计日志。爱诗公开产品和岗位都明显会碰到这类问题。
2. **视频生成往往耗时很长，前端如何实时看到进度？**
  答题抓手：任务状态机 + 轮询 / SSE / WebSocket；阶段进度、失败原因、取消、重试。公开面经里 AI 流式交互题不少。
3. **SSE、WebSocket、长轮询，做 AI 流式返回时你怎么选？**
  答题抓手：单向推流优先 SSE，双向实时交互用 WebSocket，兼容差时再考虑轮询。顺手讲代理层、断线重连、背压。
4. **如果做一个 Agent 创作助手，你怎么设计工作流？**
  答题抓手：用户意图解析、提示词优化、素材收集、镜头规划、模型选择、任务编排、结果评估、人工回改。你做过 Agent，要把 LangGraph/状态图经验讲进去。
5. **多模型协同生成时，怎么做失败重试和成本控制？**
  答题抓手：路由策略、fallback、预算控制、质量分层、缓存中间结果、按场景选模型。
6. **怎么评估一个 AI 创作 Agent 做得好不好？**
  答题抓手：任务完成率、平均生成时长、失败率、用户二次编辑率、满意度、成本、复用率。别只说准确率。

## 六、你回答时最容易踩的坑

第一，**只讲技术点，不讲业务价值**。
爱诗这种公司不太吃“我会八股”，更想知道你能不能把系统做成产品能力。

第二，**把 AI 讲得太虚**。
别空谈“大模型很强”。你要落到：任务编排、状态机、异步调度、流式返回、失败补偿、成本控制。

第三，**项目深挖讲不出取舍**。
成熟工程师一定要能说出“为什么不用另一个方案”。

第四，**没有把 Java 经验翻译成 AI 经验**。
你原来做搜索、营销、规则、工作流，这些都可以转译成 Agent 的记忆、规划、工具链、编排和评测。
