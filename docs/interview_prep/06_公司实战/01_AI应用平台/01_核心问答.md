### 你今晚最该准备的 8 个题眼

**1）Agent 架构怎么设计**
一定要能讲清：Planner / Executor / Tool Router / Memory / Observation / Evaluator / Retry / Fallback 这几层怎么分工；什么时候单 Agent，什么时候 Multi-Agent；多 Agent 的收益不是“更酷”，而是 **上下文隔离、专业化和并发**。这点和 某AI应用平台 官方对 Sub-agent 的说法完全一致。([新浪财经][6])

**2）RAG / 知识库 / memory**
至少要能顺下来：文档清洗 → chunk 策略 → 索引 → recall → rerank → prompt assembly → citation / grounding。然后能回答：为什么要 multi-query、是否要二次筛选、embedding 检索和规则知识库有什么差异、错知识/过期知识怎么治理。公开面经里这些都被直接问到了。([牛客网][2])

**3）大代码库里如何让 AI 理解项目**
这题很可能会打到你。你要准备一套像样答案：目录图谱、模块摘要、rules/约束、关键调用链、核心 domain object、增量索引、变更影响分析、只给局部上下文而不是整仓塞满。公开面经已经出现了“几十万行代码项目里怎么让 AI 快速理解结构”。([牛客网][3])

**4）AI 出码为什么会错，怎么提高留用率**
你要能讲“看起来对但逻辑错”的根因：上下文缺失、需求不清、约束没喂、检索命中不准、工具执行反馈不够、评测闭环缺失。然后讲治理动作：任务切小、先影响分析、规则库、检索、计划先行、双 Agent review、单测/集测、灰度/双跑。面经里直接问到了出码留用率、复杂业务质量下降原因、错知识误导。([牛客网][3])

**5）MCP / tools / workflow orchestration**
某AI应用平台 官方文档和岗位要求都提到了 MCP 和大量工具集成。你至少要能回答：MCP 解决什么问题、和传统 tool calling 的关系、权限/鉴权怎么做、工具结果如何标准化、失败重试和超时如何处理。([agent.minimaxi.com][1])

**6）评测与质量监控**
不是只会说 accuracy。你要讲离线和在线两套：任务成功率、步骤成功率、工具调用成功率、引用正确率、用户采纳率、人工复核率、延迟、token 成本、回退率、重试率。某AI应用平台 公开面经里有“如何建立 AI 应用质量监控体系”，官方也明显把“靠谱”和“交付结果”放在核心位置。([牛客网][8])

**7）灰度、双跑、线上稳定性**
公开面经里已经问到“灰度切流 + 在线双跑”。这是二面很喜欢的工程味问题。你要能讲：新旧链路如何并行、结果怎么比对、日志和埋点怎么做、怎么降级回滚、如何避免影响真实用户。([牛客网][2])

**8）基础功不能空**
Transformer、HNSW、Redis、缓存一致性、Linux/Git、网络基础、一道中等算法题，这些不能完全不备。某AI应用平台 的 AI 岗公开面经里这些东西反复出现。([牛客网][2])

你可以优先把下面 12 题过一遍：

1. 你做的 Agent 系统里，**规划、执行、工具调用、memory** 分别怎么设计？
2. **Single Agent 和 Multi-Agent** 你怎么选？什么场景必须拆？
3. **Sub-agent** 的收益到底是什么，除了“并行”还有什么？
4. **上下文隔离**

每个子代理只看自己需要的信息，减少主上下文污染、幻觉和 prompt 变长问题。

1. **职责专业化**

可以按角色拆分（检索、执行、审校、风控），每个子代理用不同提示词/工具/模型，更稳。

1. **故障隔离**

某个子代理跑偏或失败，不会拖垮整条链路；可以局部重试、局部回滚。

1. **权限最小化**

给不同子代理不同权限（只读/可写/高危工具禁用），安全边界更清晰。

1. **成本与性能分层**

简单任务给小模型，关键步骤给强模型，实现“该省省、该强强”。

1. 知识库/RAG 里，**chunk、召回、rerank、multi-query** 你怎么做？
2. **规则知识库** 和 **embedding memory** 的差别是什么？

`规则知识库` 和 `embedding memory` 的差别可以记成：**硬规则 vs 软记忆**。

- 规则库：稳定、可审计、可追责（适合合规/风控）。  
- embedding：覆盖广、能处理长尾，但可能召回偏差。

 **适用场景**  

- 规则库：风控边界、权限、计费、交易限制。  
- embedding：FAQ、经验复用、个性化上下文、知识问答。

**工程实践**  

  **规则库做“硬约束和最终裁决”，embedding 做“语义检索和补充上下文”。**

1. 如果知识库内容**过期或错误**，怎么避免误导模型？

先说入库，所有知识都必须带版本号、生效/失效时间、来源等级和责任人，过期内容自动降权或下线；像费率、风控、政策这类高风险知识要人工审核后才能进入高可信索引。

第二层是检索，先做 metadata 硬过滤，只查有效时间内、可信来源的数据，再做向量+关键词混检和重排序，把相似度、新鲜度、可信度一起打分，避免旧文档因语义相近被误召回。

第三层是生成，要求模型“有证据才回答”，必须带引用；证据不足或冲突时直接拒答或提示人工确认。

最后是线上止损，快速封禁问题文档ID或来源, 持续监控过期命中率、引用覆盖率、冲突率、幻觉率，指标异常就一键切高可信索引并回滚快照。

1. 在一个**几十万行代码仓库**里，你怎么让 AI 快速理解项目结构？

我不会让 AI 一上来就读几十万行代码，这样又慢又乱。

第一步先把项目画成目录/地图：哪是入口、哪是核心业务、哪是数据库、哪是外部接口。先让 AI 知道“这个系统大概怎么跑”。

第二步再做“按需找文件”，比如要改登录，就只给它登录相关的 5 到 10 个文件，不会把全仓库都塞进去。

第三步是先看主流程，比如一个请求怎么从 Controller 走到 Service，再到数据库，这样 AI 很快能抓住主干，不会在细节里迷路。

第四步是边看边验证，AI 说某个类负责某功能，我会让它用搜索和调用关系再确认一下. 最后把确认过的结论沉淀成小文档，比如“模块说明、关键入口、常见坑”，下次再问就更快。

核心思路就一句话：**先搭框架，再看细节；渐进式披露相关内容；每一步都要验证以及沉淀摘要。** 

1. 为什么你们不是让 AI **直接从 PRD 出码**，而是“方案驱动”？
2. AI 出码“**看起来对但逻辑错**”时，你怎么发现、怎么兜底？

第一步写代码时引入spec规范约束, 主要是给 AI 做一些硬性的不能做的约束。引入一些正确的范例，以及团队的代码规约。 测试时需求拆成可验证的用例，至少覆盖三类：正常场景、边界场景和异常场景，比如空参数、重复请求、并发冲突、超时重试这些最容易出问题的点。

第二步我会做小流量灰度，不只看接口 200，还看真实业务指标，比如成功率、错误率、延迟、资金和库存是否异常，确认逻辑在真实流量下也成立。

第三步是兜底机制，新逻辑必须挂 Feature Flag，发现异常可以一键关闭并切回旧逻辑；写操作加幂等键，避免重复扣费或重复下单；关键链路保留审计日志和 request_id，出问题可以快速回放定位。

1. **MCP / tool calling** 如何做权限控制、失败重试和结果标准化？

我们做 MCP / tool calling，不让模型直接“自由调用”，而是经过一个 Tool Gateway。

第一层是权限控制：按用户、Agent、场景做 RBAC/ABAC，默认最小权限，只开放白名单工具；高危工具（下单、转账、删数据）必须二次确认，参数也做 schema 校验和额度限制。

第二层是失败重试：先做错误分类，只有超时、限流、临时 5xx 这类可恢复错误才重试；业务错误、参数错误、权限错误不重试。重试采用指数退避+抖动，设最大重试次数和总超时，配熔断与降级，防止雪崩。写操作必须幂等键，避免重试造成重复副作用。

第三层是结果标准化：所有工具返回统一 envelope，例如 `code/message/data/retryable/trace_id`，并把外部错误码映射成平台标准错误码；模型只消费标准字段，不直接依赖第三方原始响应。这样做的价值是：权限可控、重试可预期、结果可编排，也方便监控和审计。  

**模型只负责决策，工具网关负责安全与可靠性。**

1. Agent 应用怎么做**评测和线上监控**？

面试就按“**离线评测 + 线上监控 + 回归闭环**”讲。

**可背版（300-500字）**  

离线侧先做分层评测，第一层看任务成功率（是否完成目标），第二层看过程质量（工具调用是否正确、是否越权、是否死循环），第三层看安全与鲁棒性（注入攻击、脏数据、长尾问题）。数据集要覆盖真实用户样本、边界样本和故障样本，并保留一套黄金集做版本对比。

线上侧我会盯四类指标：效果指标（成功率、人工接管率、用户满意度）、效率指标（P50/P95延迟、步骤数、token成本）、稳定性指标（工具失败率、重试率、超时率、回环率）、安全指标（越权调用、拒答命中、敏感操作拦截）。同时全链路打 trace_id，把一次任务里“计划-调用-结果-最终答复”串起来，便于定位问题。治理上要有灰度发布、实时告警和一键降级，异常时先止损再排查。

最后把线上 bad case 回流到离线评测集，持续做 A/B 和回归测试。

# 面试会议纪要深度分析：核心问题与通过率预判

## 一、面试核心问题拆解

本次面试为**AI Agent研发岗技术初面**，核心考察两大能力：高并发中台系统设计功底、Agent领域技术认知与实战思考，面试者暴露的问题按严重程度分级如下：

### （一）系统设计模块（中台岗位核心考察项）

#### 1. 致命硬伤（直接影响通过率的核心逻辑错误）

- **核心业务场景理解完全偏差**：针对1万TPS的邀请码生成峰值，面试者提出「异步生成+让用户等待1-2分钟，通过短信/邮件通知」的方案，完全违背了中台服务的核心定位——给业务方提供**低延迟、强实时、高可用**的标准化接口。邀请码生成是C端用户强交互场景，用户点击生成需毫秒级返回结果，该方案直接否定了中台服务的核心SLA，暴露了对业务场景的底层认知缺失。
- **方案逻辑闭环缺失**：邀请码生成的核心前提是全局唯一性，面试者初始方案未考虑「64位雪花ID转8位base36字符串，大空间转小空间必然存在冲突」的问题，被面试官点破后才补充DB校验兜底，说明架构设计缺乏前置性的风险预判与闭环思维。
- **预生成方案逻辑矛盾**：针对DB写入压力提出的「按日活用户预生成邀请码」方案，存在核心逻辑漏洞——无法预判哪些用户会生成邀请码，预生成绑定uid会造成海量资源浪费，未考虑业内成熟的「预生成未绑定的空闲邀请码池，用户申领时实时绑定uid」的方案，高并发实战经验不足。

#### 2. 明显短板（岗位必备能力的缺失）

- **分库分表复杂场景无解决方案**：针对「按uid分片后，被邀请人反向查邀请人」的高频场景，面试者先表示「没接触过」，后提出的「大数据离线表」方案完全无法满足实时查询需求，未提及业内成熟的二级索引表、宽表实时同步、ES检索等方案，说明仅掌握分库分表的基础用法，未处理过复杂业务的分片查询场景。
- **数据一致性方案严重不足**：Redis与DB异步同步的方案，仅提及「削峰填谷」，未设计对账机制、故障重试、幂等保障、兜底恢复等核心环节，对于涉及资金的返佣场景，仅一句「异步丢MQ」带过，未提及事务消息、账务对账、防重防刷等金融级方案，架构设计的严谨性不足。
- **数值空间计算能力薄弱**：未提前计算8位字母+数字（base36）的空间容量（36^8=2.8万亿），对雪花ID的数值范围无清晰认知，被面试官追问冲突问题时无法给出精准的技术论证，只能被动补兜底方案，基础技术功底不扎实。

#### 3. 细节疏漏（印象减分项）

- 审题不严谨，面试官明确要求「8位字母+数字的邀请码」，面试者初始表述为「7-8位」，错误提出base64编码，未注意到纯字母+数字仅支持base36，第一印象失分。
- 中间件选型无场景适配，仅泛泛提及Kafka/RocketMQ，未区分「账务返佣场景需RocketMQ事务消息保障」「高吞吐日志场景适配Kafka」，对中间件的选型逻辑仅停留在「知道名字」，未理解场景适配性。

### （二）Agent技术模块（岗位核心方向考察项）

#### 1. 核心概念理解停留在表面，未触及本质

- **MCP与Skill的对比认知偏差**：仅泛泛表述「MCP是对外协议，Skill是内部约定，二者互补」，未讲透核心本质——MCP是标准化的工具接入协议，解决的是工具跨runtime、动态发现、标准化调用的问题；Skill是工具能力的封装单元，可基于MCP实现，二者是层与层的关系。针对面试官「接口+说明文档能否替代MCP」的反问，未提及MCP的核心价值（服务发现、权限控制、流式交互、标准化入参出参校验），说明仅了解概念，无实战落地经验。
- **Skill与RAG的核心差异理解错误**：仅表述「RAG重，Skill轻」，未戳中本质区别——Skill是**可执行的动作单元**，解决工具调用的上下文污染问题；RAG是**知识检索单元**，解决模型私有知识接入的问题，二者核心目标完全不同，回答未体现对两个技术方向的深度理解。

#### 2. 行业动态无独立深度思考，仅停留在信息复述

- 针对Claude Code源码泄露的核心问题，仅复述了网上流传的「上下文管理、安全审核、sub agent」等表层信息，未结合自身研发经验，给出对Agent工程化的深度思考、设计取舍的理解，或对国内Agent研发的借鉴意义，说明仅被动接收行业信息，无主动的深度研究与独立思考。
- 针对Agent Team与Sub Agent的场景差异，仅泛泛提及「跨多领域复杂项目」，未给出具体的落地场景，无法回答「写代码场景下，Agent Team能做什么Sub Agent做不了的」核心问题，对多智能体协同的实战场景理解极浅。

#### 3. 核心模块认知不足，无成熟的优化思路

- 针对小龙虾Memory系统的设计，仅吐槽「费TOKEN」，未理解其用MD文件存储的核心优势（本地部署的隐私性、可移植性、离线可用性、用户可编辑性），也未提及业内成熟的记忆系统优化方案（分层记忆、向量检索、记忆摘要、遗忘机制），对Agent核心的记忆模块认知不足。

### （三）面试沟通环节减分项

反问环节仅询问了团队业务方向与规模，未提出与岗位、技术相关的深度问题（如团队当前在Agent方向的核心技术挑战、技术栈选型、研发流程等），体现出对岗位的求知欲与准备度不足。
