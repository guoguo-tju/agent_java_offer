## 1. 交易Agent与风控平台

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容



### 你最该准备的 12 道题

**1. 如果让你设计 交易型Agent产品 的后端架构，你会怎么拆？**
你要能讲出：`Query理解 -> 任务路由 -> 检索/RAG -> 工具调用 -> 风控校验 -> 执行 -> 结果解释 -> 观测与评测`。
面试官想听的不是“我会用 LangChain/Dify”，而是**怎么把分析和执行分层、怎么设护栏、怎么追踪一次用户请求到真实下单的全链路**。这类题是被 交易型Agent产品 的官方产品形态直接“暗示”出来的。

**2. Agent 为什么不能只做成固定 workflow？什么时候该用 planning？**
你要能区分：
固定流程适合低风险、强约束任务；
Agent/planning 适合不确定查询、需要多轮推理和多工具决策；
而交易场景里真正高风险的动作，常常不能完全交给自由规划，必须加**硬规则/风控闸门**。
这题是典型的 “AI 工程能力 + 业务安全感” 题。结合现在后端/AI 工程岗位的高频题，Agent 的 planning、memory、tools、action 本身就是常见考点。

**3. 如何做多工具调用？工具失败怎么办？**
你要准备：工具 schema、超时、熔断、重试、fallback、幂等、权限隔离、结果校验。
在 某交易平台 这种产品里，工具不只是搜索或问答，还包括**行情、新闻、持仓、风控、下单**。一旦工具失败，不能只说“返回错误”，要说明**如何降级成建议模式而不是执行模式**。

**4. 如何做 RAG，文档切片怎么设计，如何评估效果？**
这是现在 AI 工程方向的通用高频题。你至少要能回答：chunk 大小、overlap、metadata、hybrid retrieval、MMR 去重、重排序、离线评估与线上指标。牛客整理的 2026 Java/AI 工程题里，把 RAG 架构、检索优化、长上下文处理和效果评估都列成高频点。

**5. 多轮对话怎么维护上下文？记忆机制怎么做？**
你要能讲出：短期记忆（会话窗口）、长期记忆（向量库/用户画像）、事件记忆（重要操作）、摘要压缩、失效策略。
如果你回答成“我把历史消息全塞回去”，会比较弱。Agent 面试现在已经会追问 memory 的实际工程取舍。

**6. 怎么防 Agent 幻觉、误导性分析和 Prompt 注入？**
这题在交易场景尤其要命。你应该准备：
模型输出结构化约束、只允许白名单工具、引用证据、敏感动作二次确认、风控规则优先、注入隔离、工具输出校验、策略与执行分层。
牛客的 AI 工程整理里把**幻觉、Prompt 注入攻击、自动化评估体系**都列成高频追问。

**7. 为什么交易系统里会优先用 WebSocket？REST 怎么配合？**
一个稳的回答应该是：
行情、订单状态、仓位变化适合 WebSocket 的低延迟实时推送；
查询历史、配置类接口、回补丢包适合 REST；
系统设计上经常是 **WebSocket 收实时流 + REST 做补数/对账**。
某交易平台 官方文档本身就把 WebSocket 当成核心接口，并支持通过 WebSocket 下单。

**8. Order book 的 snapshot + 增量更新怎么保证正确？**
这是非常像交易所面试题的点。你要能讲：
先拿 snapshot，再合并增量；
价格档同价替换、数量为 0 删除、无同价则按 bid 降序 / ask 升序插入；
定期或每次计算 checksum，对不上就触发重拉快照。
某交易平台 官方文档甚至直接给了 **CRC32** 的构造方式。

**9. 为什么下单要设计 clientOid？怎么做幂等？**
这一题几乎就是送分题，因为 某交易平台 文档都写明了：`clientOid` 是客户端自定义订单 ID，官方明确提醒它是客户端与服务端之间**唯一的幂等保证字段**，不传会有重复下单风险。
你要会顺着讲：
请求去重、重试安全、网络超时后的查询补偿、订单状态最终一致性。 

**10. 交易场景里如何做风控？**
你至少要会拆成这几层：
输入风控（参数校验、黑白名单、用户状态）
策略风控（仓位、杠杆、限价/市价约束）
执行风控（频控、订单上限、重复单）
结果风控（成交回执、对账、告警）
官方文档里能直接看到交易参数校验、持仓模式差异、订单上限等规则，所以这块非常可能被问。

**11. 如果模型服务抖动，怎么降级？**
标准答案别只说“熔断”。你最好说成：
先降级到纯行情/规则引擎；
保留查询和分析，不开放自动执行；
缓存最近一次结构化建议；
对外返回明确的“不确定”状态；
并打通 tracing、成本、时延、成功率监控。
这类问题在 AI 工程面里也很高频，尤其首字延迟、失败率、AB 实验和自动化评估。

**12. 你如何评估一个 Agent 做得好不好？**
这题一定别只答“看用户满意度”。
建议你至少讲 4 类指标：
任务完成率、工具调用成功率、执行成功率、时延/成本；
再加上业务指标，比如留存、复访、转化，或者误触发率、人工兜底率。
因为 交易型Agent产品 官方强调的是从**分析到执行的全链路闭环**，所以评估指标也不能只看聊天质量。

### 如果岗位偏链上 / Wallet / Onchain Agent，再加这组题

某链上钱包产品 官方还有一条 **某链上钱包产品 Skill** 产品线，主打“**为 AI Agent 而生**”、**0 Gas 交易**、**一键跨链**、**x402 自主支付**、**8+ 主流公链**、**Order Mode**，还给了 `swapQuote / swapSend / orderQuote / orderCreate / orderSubmit` 的代码示例。
如果你的 JD 里出现 **Wallet / Onchain / Skill / x402 / cross-chain / no gas** 这些词，就要额外准备：
“Gasless 交易怎么实现？”
“跨链订单状态机怎么设计？”
“为什么用 order mode 而不是多 API 串联？”
“AI Agent 的自主支付权限怎么控？”
“签名、订单提交、状态追踪怎么做幂等和补偿？”

### 我对你这场面试的实际判断

如果你面的是交易平台侧的 Agent 开发岗位，我会判断它不像纯大模型平台岗，而更像 **AI Agent 工程 + 交易执行系统 + 风控/实时数据** 的混合岗。
也就是说，**Java 后端、并发、MySQL/Redis/MQ** 这些基础不能丢；同时 **RAG、tool calling、memory、幻觉治理、评测、Prompt/上下文管理** 也要能讲；最后你还得补上 **order book、WebSocket、幂等、对账、风控** 这些交易系统味道很重的内容。这个判断主要来自公开产品形态、接口设计和行业常见技术栈。

你今晚如果时间不多，我建议你按这个顺序刷：
**先背 12 题里的 7/8/9/10**，因为这些最像交易平台场景题；
然后补 **2/3/6/12**，这是 Agent 岗最容易拉开差距的部分；
最后再复习 **Java 并发、MySQL 索引、Redis 分布式锁、MQ 幂等** 这些后端基本盘。公开的 Java/AI 工程高频题汇总里，这些仍然是必考区。

如果你手上有具体 JD，可以继续按岗位描述定制一版“最可能被问的 20 题 + 每题怎么答”。


## 2. 延伸追问与场景化回答

### 资料说明

- 本节内容来自个人面试资料的重组整理与公开资料归纳。
### 原文内容

**1. 先判断你做的是不是真 Agent，而不是套壳工作流**
他很可能会问：
“你这个系统为什么一定要用 Agent，不是 workflow 就够了吗？”
“你的规划、执行、观察、反思是怎么拆的？”
“MCP、tool calling、function calling、workflow orchestration 到底有什么区别？”
“多 Agent 什么时候值得上，什么时候反而更差？”
这类问题很像最近公开 Agent 面试高频题，也和交易平台在 Agent Hub / MCP 方向上的投入一致。

**2. 深挖你项目的系统架构**
这类 usually 是二面的核心。你很可能会被追问：
“你的 Agent 整体架构图是什么？”
“状态怎么存？短期记忆、长期记忆怎么分？”
“任务是同步还是异步？失败重试怎么做？”
“工具调用失败、超时、返回脏数据时怎么 fallback？”
“怎么避免无限循环、上下文污染、重复执行？”
公开 JD 里通常会强调 API/微服务化、容器化、监控告警、灰度发布、弹性扩缩和线上稳定性。

**3. RAG 和知识系统，几乎一定会被问**
高概率问题包括：
“你的检索链路怎么设计，query rewrite / recall / rerank / answer generation 各怎么做？”
“为什么选这个 embedding 模型和向量库？”
“你怎么处理时效性、脏数据、权限隔离？”
“怎么评估 RAG 真的有效，不是看起来有效？”
这类岗位通常会把 RAG、向量数据库、Embeddings、评测和持续学习写得比较明确；公开 Agent 面经里也把 RAG 列成高频。

**4. 模型与评测：技术总监会看你有没有“量化思维”**
很可能会问：
“你怎么评估 Agent 成功率？”
“线上看哪些指标？”
“prompt、模型、工具、检索，到底是谁导致了失败？”
“有没有离线评测集？线上 A/B 怎么做？”
“什么时候考虑 LoRA / QLoRA，什么时候只做 prompt + RAG 就够？”
因为这类 AI 岗通常会要求做 **线下评测集、线上 A/B、效果与成本度量、LoRA/QLoRA、DPO/ORPO、合成数据增强**。

**5. 安全和合规，这一块在交易平台场景里很像“必问题”**
这是我觉得你最要重点准备的。公开安全研究与行业实践里，普遍会点名 **Prompt Injection、插件/Skill 供应链风险、API Key 与账户权限滥用、自动执行导致误操作和权限扩大** 等问题。

所以技术总监很可能会直接问：
“如果 Agent 可以下单，你怎么做权限边界？”
“Prompt injection 怎么防？”
“外部工具或 Skill 被污染怎么办？”
“API Key 怎么最小权限化？”
“是否需要 human-in-the-loop，在哪一层拦？”
“日志里如何脱敏？审计链怎么做？”
你回答这类题时，最好带上 **read-only 默认、动作分级授权、额度限制、风控前置校验、双确认、审计日志、回放、沙箱执行、密钥隔离** 这些词。因为这和交易平台公开安全材料的方向高度一致。

**6. 会看你有没有交易所业务 sense，不只是技术 sense**
这类岗位说明里经常会提到：要懂 **交易所业务数据流、用户旅程、风控与合规、ROI 导向**，还偏好有 **AI x Crypto、交易、投研、链上探索** 经验的人。

所以他可能会问：
“Agent 在交易所里最先该落在哪个场景，客服、投研、风控还是交易助手？”
“这个 Agent 给平台带来的核心指标是什么，转化率、留存、响应时延、拦截率、人工替代率？”
“如果你只能做一个 MVP，你先做哪个能力，为什么？”
“你怎么看 AI 在现货 / 合约 / 风控 / 合规里的边界？”
这类题不是考你背行业知识，而是看你能不能把技术和业务结果连起来。

**7. 还会有一小部分工程基础/八股/算法穿插**
公开 Agent 面经里，除了 Agent 本身，还常见 Redis、HTTPS、拓扑排序、训练流程这类基础题。总监不一定细抠，但可能突然穿插 1-2 个来验证底子。



我给你押一组**最像交易平台二面现场会出现的题**：

1. 你做过的 Agent 项目里，哪一部分最能证明它不是普通 chatbot？
2. 为什么这个场景适合 Agent，而不是固定 workflow？
3. 你的 tool schema 怎么设计，如何约束模型少乱调工具？
4. 你的 RAG 是怎么做召回、重排、引用和 freshness 控制的？
5. 线上失败最多的是哪类 case，怎么定位是模型问题还是系统问题？
6. 你如何评估 task success rate、tool success rate、latency、cost 和人工接管率？



1. 如果让你给一个交易平台做一个“可真实交易”的 Agent，你如何设计权限和风控边界？

可以把这题讲成“**九个词对应三层防线**”，面试会很清晰：

1. **权限层（先防误用）**  

`read-only 默认`：新 Agent 默认只能查行情和持仓，不能下单。  

`动作分级授权`：查阅、模拟下单、真实下单、调杠杆、撤单分别授权。  

`密钥隔离`：按环境/策略/账户拆分 API Key，生产密钥放 KMS/HSM，不落盘，定期轮转。以及子账户隔离

2. **交易前风控层（先校验再执行）**  

`额度限制`：单笔限额、单日限额、单策略限额、最大仓位、最大杠杆。  

`风控前置校验`：在下单网关校验价格偏离、滑点、频率、风险敞口，不通过直接拒单。  

`双确认`：高风险动作（大额开仓/高杠杆/新币对）要求二次确认，可人工审批或双签。

3. **运行与追责层（可追溯可止损）**  

`审计日志`：完整记录“输入-推理摘要-风控结果-最终指令-交易回执”。  

`回放`：按 request_id 回放一次决策链，用于事故复盘和策略评估。  

`沙箱执行`：新策略先模拟盘/影子模式，稳定后再灰度真实资金。



1. prompt injection 在交易场景里最危险的后果是什么，怎么防？

最危险的后果一句话：**让 Agent“绕过风控去执行高风险真实下单”，造成快速亏损甚至爆仓，而且是自动连续发生**。

1. **执行权隔离**：LLM 只产出“建议”，最终下单必须经过确定性风控引擎。  

2. **上下文隔离**：外部内容（新闻、社媒、RAG）一律标记`untrusted`，永不当系统指令。  

3. **工具白名单 + 参数校验**：只允许固定动作；杠杆、数量、币对、方向都做硬校验。  

4. **高危双确认**：大额/高杠杆/新策略必须人工确认或双签。  

5. **运行时止损**：异常行为检测（频次突增、风险暴露异常）+ 一键熔断`read-only`。



1. Skill / plugin 供应链被污染时，系统层怎么止损？

核心是要先保证资金和安全和边界，然后再恢复服务，宁可短时降级，也不能带毒运行。

首先把线上的大模型的权限只给到只读。然后去关闭对第三方这个工具的调用，暂停高位的任务队列。后续定位影响面，以及是最终导致的是哪个插件的原因。然后把这个插件做回滚到安全的版本，然后逐步的恢复恢复线上的服务。



1. 为什么交易平台要做 Agent Hub / MCP，而不是继续让大家自己调 API？

交易平台做 Agent Hub / MCP，主要是解决“大家各自调 API”带来的三大问题：接入慢、质量不稳、风险不可控。  
第一，开发效率。直接调 API 每家都要重复做鉴权、重试、限流、参数校验和工具封装；Hub + MCP 用统一协议和工具描述，接入周期从“周级”降到“天级”。  
第二，风险治理。交易场景高风险，平台必须把权限分级、额度限制、风控前置校验、审计日志、熔断降级做成系统能力；如果全靠各家自己实现，安全和合规口径会碎片化。  
第三，生态和商业化。Hub 能做插件发现、版本管理、灰度发布、计量计费和质量评分，形成开发者生态网络效应。  
所以本质上是：**底层保留原生 API 给高阶团队，上一层提供 Agent Hub/MCP 给规模化生态**，既保灵活性又保平台可控性。

1. 如果只有 2 个工程师 + 6 周时间，你会给一个交易平台做什么 MVP？

链上赚钱agent, 结合X402协议, 参与链上活动, 打土狗,做PVP, 预测市场, 一周需求调研。一周做整个工工具下游依赖数据链的打通。后一到两周做开发。再后面2周做测试、审计、稳定性建设以及小流量灰度与实盘。

1. 你过去做过哪个线上事故/失败案例，事后怎么复盘和改架构？
