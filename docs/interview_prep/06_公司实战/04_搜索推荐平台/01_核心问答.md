2. **RAG 检索准确率**怎么评估。

首先一个是对检索性能的评估，这边主要的目标是评估它，我们这个检索流程能否找的对和找找的全。评估首先需要构造一个有标注的数据集。然后呢，通过离线或在线的一些核心指标，

**第一阶段：检索性能评估 (Retrieval Evaluation)** 这个阶段的目标是评估我们的检索器（Retriever）能否“**找得对、找得全**”。评估需要一个包含（问题，相关文档ID）的标注数据集。主要指标:

1. **上下文精确率 (Context Precision):** 衡量检索到的文档中有多少是真正与问题相关的。它反映了**检索结果的信噪比**。Precision@K：TopK 中相关片段占比
2. **上下文召回率 (Context Recall):** 衡量所有相关的文档中，有多少被我们的检索器成功找回来了。它反映了**信息查找的全面性**。Recall@K：所有正确证据被召回了多少
3. Hit@K**:** TopK 里是否至少命中一个正确证据
  **第二阶段：生成性能评估 (Generation Evaluation)** 这个阶段的目标是评估LLM在给定上下文后，能否生成“**忠实、准确、有用**”的答案。

- **核心指标（通常需要LLM-as-a-Judge或人工评估）：**
  1. **可溯源性/忠诚度 (Faithfulness / Groundedness):**
    - **评估问题：** 生成的答案是否完全基于所提供的上下文？是否存在捏造或幻觉？
  2. **答案相关性 (Answer Relevancy):**
    - **评估问题：** 生成的答案是否直接、清晰地回答了用户的原始问题？
  3. **答案正确性 (Answer Correctness):**
    - **评估问题：** 答案中的信息是否事实准确？（这是一个更严格的指标，因为有时即使忠于原文，原文也可能是错的）
- **自动化评估框架：**
  - 像 **RAGAS**, **ARES**, **TruLens** 这样的开源框架，它们使用LLM-as-a-Judge的思想，将上述的Faithfulness, Relevancy等指标自动化计算出来，极大地提高了评估效率。例如，RAGAS会生成问题、答案，并自动检查答案是否忠实于上下文。

第二个是在生成时会有一些提升生成的效果的措施。一个是在 prompt 里明确指基于给定证据回答。还有一个是强制模型，强制著名引用的来源。然后就是生成后用一些评估手段来正确的评估它生成的质量，包括生成的整个忠忠诚度，包括正确性或者相关性这些指标。然后还要对生成的这个工程指标，包括幻觉率、引用率，这些做一个长期的监控。

3. 向量检索“**慢和贵**”怎么优化。

- **延迟-慢：** RAG的“检索+生成”两步天然比直接调用LLM要慢。在实时交互场景下，检索和LLM生成的延迟都必须被极致优化。
- **成本-贵：**
  - **计算成本：** 大规模文档的嵌入、向量数据库的运行、LLM的API调用，都是持续的成本支出。
  - **存储成本：** 向量索引本身会占用大量的存储空间，尤其是高维度的嵌入。
- 优化思路: 就是整个把整个阶段拆成 Embedding 嵌入、 向量检索/存储、重排，这几大阶段，然后先找到最大的一个瓶颈。
- 如果是 embedding 阶段太贵的话，可以换一个更便宜的模型。
- 如果是 AN 阶段是在线阶段，如果觉得太慢的话，可以通过调 AN 的一些召回参数，在召回率和实验之间找到一个平衡点。
- 向量存储的成本的话，可以通过一些压缩存储来降低内存和机器的成本。
- 召回的时候混合检索，比如说 BM25加向量检索，减少纯向量误召回，能降低后续的重排成本。

面试中没答上来的点:

1.针对excel表格, rag召回质量不好, 有什么好的改进方法?

- 不要按纯文本切块，要做“表格结构化切块”：sheet -> 表 -> 行 -> 单元格。
- 每行都补齐表头语义，比如把行序列化成“日期=xxx, 客户=xxx, 金额=xxx”。
- 做混合检索：关键词检索（BM25）+ 向量检索 + 元数据过滤（sheet、字段、日期）。
- TopK 后加 rerank，减少“看起来相关但答非所问”的片段。

2.Excel 表格里，它单个单元格的字段内容太多了，已经超过了切分长度了，这可以可以怎么处理呢？

- 对“单元格内部”再分段（句子/段落级），不是硬截断。
- 建两级索引：行级摘要索引 + 单元格明细索引。
- 检索先命中摘要，再回源拉明细 chunk。
- 超长字段可存“摘要向量 + 原文指针”，避免一次塞满上下文

3.RAG 离线构建的时候，它那个要实时性的要求比较高，每次那个文档一改动，可能就需要全量的做 embedding 替换向量数据库，这个成本比较高，怎么解呢？

- 用 chunk 级哈希做变更检测，只重算变更 chunk。
- 向量库做 upsert/delete by chunk_id，不动未变部分。
- 索引更新异步化（消息队列），读写解耦。
- 做版本化索引，支持灰度切换，避免重建时服务抖动。

4.我是想做增量更新的话，可可以有有可行性吗？因为它每次增量更新完，它的向量的整个排序都变化了，又得重新更新一遍，怎么解呢？

- 通常不用。向量检索排序是查询时实时算相似度，不是全库预排序。
- 实践上用“Base 索引 + Delta 增量索引”，查询时合并结果。
- 只在低峰期做周期性 merge/compact，提升检索效率。
- 只有在大规模分布变化时，才做全量重训或重建。

5.但是用户可能有涉及到各类各式各样的问题，比如说是日期，用户想看，比如说过去某一天的一个当天的一个文档记录。但是呢，他这个文档上传 Word 的时候他就，那更新时间就不准了。那如果我那我只能去解析文档里的信息来提取出来这个日期了。但是这样的话，可能就要把全站所有的文档，可能有很大很大的文档都得全都提取，这样成本太高了。针对这个痛点可以怎么解呢？

- 建“业务时间元数据”字段，不只依赖文件更新时间。
- 只对新增和变更文档做抽取；历史库按热度异步回填。
- 查询时先用日期索引粗筛，再对 TopN 候选做深度解析。
- 抽取结果持久化并缓存，避免重复算。

6.那如果这次建了元数据的需求是按日期搜索，下次日期就按作者或者其他新新的字段搜索，那是不是还得重重建索引呢？这有什么好的解法吗？

**一般不需要重建向量索引**。  
我们只需要把检索拆成两层：

1. **语义层（向量索引）**
  - 负责“内容相关性”，尽量稳定。
  - 不因“按日期/按作者过滤”而变。
2. **结构层（元数据索引）**
  - 负责“字段过滤和排序”，比如 date/author/tag/region。
  - 新字段上线时扩 schema + 回填该字段即可。

实际查询流程：

- 先按元数据筛候选（date=xxx、author=yyy）
- 再在候选集做向量召回 + 重排  
这样既快又准。
