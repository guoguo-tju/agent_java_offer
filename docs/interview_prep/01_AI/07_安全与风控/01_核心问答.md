#### 母题6：安全对齐与高风险动作控制（含HITL）

主问题：如何确保 Agent 安全、可控、可追责？

口述答案：
安全要分层做：模型层做对齐训练，系统层做最小权限与工具白名单，执行层做沙箱隔离与资源限额，业务层对高风险动作强制人工审批。再配合红队测试、注入攻击测试、全链路审计日志和告警阈值。核心原则是“先可控再智能”：即使模型偶发漂移，也不能直接触达不可逆动作。

常见追问：

1. 如何设计高风险动作审批流？
2. 红队测试怎么选样本与评估口径？
3. 安全与体验冲突时如何平衡？

原题答案补充（来自原54题）：

##### 原题10：如何确保一个 Agent 的行为是安全、可控且符合人类意图的？在 Agent 的设计中，有哪些保障对齐方法？

答：
要做分层安全：

模型层做对齐训练（如 RLHF/DPO），对基础LLM进行微调，使其遵循“有用、诚实、无害”的原则，这是所有安全措施的基石

系统层做最小权限和工具白名单, 只给Agent完成其任务所必需的最少的工具和权限

执行层做沙箱隔离与资源限额，将Agent的执行环境与宿主系统隔离, 让Agent生成的代码或命令在一个受控的沙箱（如Docker容器、虚拟机）中执行。这样即使Agent被劫持或产生恶意代码，其破坏范围也被限制在沙箱内部，不会影响到外部系统.

业务层做高风险动作的人类确认（HITL）。在执行如“删除文件”、“发送邮件”、“执行金融交易”等敏感操作前，Agent必须生成一个执行计划，并暂停等待人类用户的明确批准.

再配合护栏规则、红队测试和行为审计日志。

##### 原题33：什么是红队测试？它在发现 LLM 和 Agent 的安全漏洞与偏见方面扮演着什么角色？

答：
它的价值是发现常规测试覆盖不到的高风险边界情况，尤其对 Agent 很关键，因为 Agent 具备执行能力。红队测试是用攻击者视角主动找Agent系统弱点, 以评估和提升其安全性和鲁棒性。常见手法有越权请求、提示注入、越狱诱导、工具滥用。实践上红队结果应回流到规则护栏、权限策略和训练数据，形成持续加固闭环。

##### 原题51：Human-in-the-loop 在 LangGraph 怎么落地？

答：
我会在关键节点使用 interrupt 或审批节点，让系统在“高风险动作前”暂停，等待人工确认后继续。典型场景是资金操作、批量发送、生产变更。这样保留自动化效率，同时把不可逆动作的最终控制权交给人。

### 原文内容

### 4) Understanding Manus sandbox - your cloud computer

- 原文链接：
  - [https://manus.im/blog/manus-sandbox](https://manus.im/blog/manus-sandbox)
- 核心思想：
  - 将 Agent 执行环境放在沙箱中，形成可隔离、可复现、可回放的“云电脑”执行面。
  - 沙箱让工具调用不直接触达核心系统，降低高权限动作风险。
  - 文章强调“思考能力”必须和“安全执行环境”配套，才能上线生产场景。
- 文章概述（约500~1000字）：
  - 这篇文章的重点是把 Agent 的“执行面”讲清楚：模型会推理并不等于系统可以安全上线，真正落地还需要一个可控的运行环境。Manus 把这个运行环境定义为 sandbox（云电脑），核心价值是隔离、可复现、可治理。隔离意味着 Agent 的浏览器操作、代码执行、文件读写都在受控环境中完成，不直接接触用户本机或核心生产系统；可复现意味着每次任务过程可追踪、可回放，便于故障排查和合规审计；可治理意味着可以施加权限边界、资源配额、网络策略和操作白名单。文章强调很多 Agent 事故并非“模型胡说”，而是执行环节缺乏约束，例如重复提交、高权限误操作、异常重试放大。沙箱机制相当于给 Agent 套上工程安全带，把高风险动作从“默认放行”改成“可验证后执行”。从面试表达看，你可以把这篇文章提炼为三层：安全层（权限/隔离）、稳定层（重试/回滚/断点恢复）、观测层（日志/轨迹/审计）。再补一句“Agent 的生产门槛不是会不会回答，而是能不能安全执行”，就能把你和只谈模型能力的候选人拉开差距。
- 面试可能问的点：
  - 问：为什么 Agent 项目必须做执行沙箱？
  答：因为 Agent 一旦具备网页操作、代码执行和外部调用能力，风险从“答错”升级为“误执行”。沙箱把执行面与核心系统隔离，能限制破坏范围，避免高权限误操作直达生产环境。没有沙箱，很多企业场景即使效果好也不敢上线。
  - 问：沙箱里如何做权限控制、资源限制和审计回放？
  答：我会做最小权限白名单（可访问域名、可用工具、可写目录），再加 CPU/内存/时长限额和网络出口策略。所有关键动作记录为结构化事件日志，绑定任务ID和操作者，支持按步骤回放，便于定位问题、追责和合规审计。
  - 问：沙箱模式对稳定性、排障和合规有什么收益？
  答：稳定性上可通过隔离和限额避免故障扩散；排障上有可回放轨迹，能快速复现实例并修复；合规上可形成“谁在何时做了什么”的审计链。对企业来说，这三点比单次模型效果更关键，决定系统是否可持续运营。
