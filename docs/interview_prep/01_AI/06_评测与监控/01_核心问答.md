#### 母题14：评估体系与线上监控（LLM/Agent）

主问题：如何设计一套可落地的 LLM/Agent 评估体系？

口述答案：
评估要“离线 + 在线 + 人评”三位一体。离线看能力覆盖（事实性、推理、安全、格式）；在线看业务指标（成功率、转化、时延、成本、稳定性）；人评校准自动评估偏差。对 Agent 还要看过程指标，如步数、工具成功率、重试率、中断率。上线后通过持续监控与坏例回流形成闭环，避免模型漂移无人感知。

常见追问：

1. 为什么 BLEU/ROUGE 评 LLM 不够？
2. LLM-as-a-Judge 的偏差怎么控制？
3. 线上成功率下降如何快速归因？

原题答案补充（来自原54题）：

##### 原题26：为什么传统的 NLP 评估指标（如 BLEU, ROUGE）对于评估现代 LLM 的生成质量来说，存在很大的局限性？

答：
BLEU/ROUGE 主要比较词面重合，适合翻译或摘要早期任务，但对现代 LLM 明显不够。它们难衡量语义等价、事实准确、推理正确和安全性，还会惩罚“措辞不同但内容正确”的回答。面试可强调：现在评估要从词匹配升级到“语义+事实+推理+对齐”的多维体系。

##### 原题27：请介绍几个目前行业内广泛使用的 LLM 综合性基准测试，并说明它们各自的侧重点。（例如：MMLU, Big-Bench, HumanEval）

答：
MMLU 主要看跨学科知识与解题能力，覆盖多门学科；Big-Bench 更偏能力边界与复杂任务泛化；HumanEval 聚焦代码生成，通常用 pass@k 评估。可补充 GSM8K 看数学推理。面试里建议说“公开基准用于横向比较，但最终还要叠加业务数据集做纵向验证”，这样更工程化。

##### 原题28：什么是“LLM-as-a-Judge”？使用 LLM 来评估另一个 LLM 的输出，有哪些优点和潜在的偏见？

答：
LLM-as-a-Judge 是用强模型按预设 Rubric 去评估候选模型输出，优点是快、可规模化、口径相对统一。风险在于偏见：位置偏好、冗长偏好、风格偏好，以及裁判模型自身知识错误。实践上一般配“自动评审+人工抽检”，并做双盲与顺序打乱，减少系统性偏差。

##### 原题29：如何设计一个评估方案来衡量 LLM 的特定能力，比如“事实性/幻觉水平”、“推理能力”或“安全性”？

答：

**1.衡量“事实性/幻觉水平”：**

- **就是基于一个知识库进行问答。然后看这个模型所回答的是否能从这个知识库中找到答案进行匹配，不是自己瞎编的。另外就是设计一些能够以对抗性的问题，能够诱导模型产生幻觉的问题。比如说询问不存在的人或者事件的信息，然后看一下模型的回答。**

**2. 衡量“推理能力”：**

- 那推理能力就是让模型做一些数，数学题或者逻辑推理的题，或者需需要自行设计一些推理的任务。然后呢，一评估的时候，一个是看结果的评估，就看最后这个回答的是否是准确。另外一个是过程的评估，就是看模型的思维链。要主要看它的推理步骤是否合乎逻辑，是否正确和推理过程。

**3. 衡量“安全性”：**

- 安全性就是模模型是否会拒绝一些回答有害、不道德或者危险的一些用户请求。所以就要找一些对抗性的提示词数据集，然后包含了一些精心设计的引导模型绕过安全护栏的危险问题。另外一个思路就是专门的红蓝攻防，由人类专家主动的构造性的做一些攻击性的提示。评判方法就是看这个模型的一个拒绝率。模型看拒绝回，要拒绝回答这个有害问题的比例。以及误伤率，就是模型会错误的拒绝回答一个正常安全的一个问题。还有就是人工评估介入审核。

##### 原题30：评估一个 Agent 为什么比评估一个基础 LLM 更加困难和复杂？评估的维度有哪些不同？

答：
基础 LLM 多是单轮“输入-输出”评估，而 Agent 是多步交互系统，过程会影响结果，状态也持续变化，所以复杂度高很多。除了结果正确性，还要评估步骤效率、工具调用质量、异常恢复能力、成本和时延。对LLM的评估更像是“**产品质量检测**”，而对Agent的评估更像是“**路况复杂的真实驾驶测试**”，不仅要看是否到达终点，更要看驾驶过程中的效率、安全性和应对突发状况的能力。

**评估维度的不同：**


| **评估维度**    | **基础 LLM**                                                                        | **Agent**                                                                                                                                                     |
| ----------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **核心评估对象**  | **单个回答的质量** (Quality of a single response)                                        | **整个任务完成过程** (The entire task completion process)                                                                                                             |
| **主要维度**    | - **准确性 (Accuracy)** - **流畅性 (Fluency)** - **相关性 (Relevance)** - **安全性 (Safety)** | - **任务成功率 (Task Success Rate):** 能否最终完成目标？ - **效率 (Efficiency):** 完成任务花了多少资源？（见下文） - **鲁棒性 (Robustness):** 能否处理异常和错误？ - **自主性 (Autonomy):** 在没有人类干预的情况下能走多远？  |
| **新增的过程维度** | (无)                                                                               | - **成本 (Cost):** LLM调用次数、API费用、Token消耗。 - **延迟 (Latency):** 完成任务的总时间。 - **步骤数 (Number of Steps):** 任务分解和执行的步数。 - **纠错能力 (Error Recovery):** 从工具报错或错误状态中恢复的能力。 |
| **评估方法**    | 静态数据集上的基准测试 (MMLU, HumanEval)                                                     | **交互式环境**中的基准测试 (WebArena, AgentBench)                                                                                                                        |


##### 原题31：你了解哪些专门用于评估 Agent 能力的基准测试？这些基准通常如何构建测试环境和任务？

答：
常见有 WebArena、AgentBench、GAIA。它们一般构建可复现沙箱环境（网页、终端、数据库、工具接口），给高层任务目标，让 Agent 自主拆解执行，最后通过脚本检查终态是否达成目标。相比静态题库，这类基准更贴近真实任务流程，能评估规划、工具使用和鲁棒性，而不仅是最终文本质量。

随着Agent研究的兴起，一系列专门用于评估Agent能力的基准测试被开发出来，它们的核心特点是提供**可控的、可复现的交互式环境**。

**几个知名的Agent能力基准测试：**

1. **WebArena:**
  - **专注领域：** **网页浏览与操作**。是构建了一个高度逼真的独立的网页环境磨难器，环境模拟器复刻了包括多个真实网站，如电商论坛、开发写作工具的功能，让 A 智能在这个独立的环境里完成复杂的任务。比如说是在电商网站上找到一个满足满足要求的商品并加入购物车。然后评估方式是最终在网页状态会会程序化的判断，比如说判断购物车里是否有正确的商品。
2. **AgentBench:**
  - **专注领域：** **通用Agent能力的综合评估**。Agent Bench 是一个全面，比较全面的包含了八种不同的环境来评估 Agent 的能力。比如说是操作系统环境，在 Linux 终端操作文件，执行命令。比如说在数据库环境，对一个 SQL 数据库进行查询。比如说知识图谱环境，在知识图谱中进行多条推理。比如说一些简单的文字与与冒险游戏。
3. **GAIA (General AI Assistants):**
  - **专注领域：** **模拟人类使用真实工具完成复杂任务**。就是完成，模拟人类使用真实工具完成一个比较极具挑战性的任务。一般这种任务需要多种工具并用才能解决，比如说用网页搜索、代码、文件操作才能一并解决。”

**这些基准通常如何构建测试环境和任务？**

测试环境的构建一般是构建一个可控隔离的沙箱环境, 不会让 A 生态来访问真实的互联网环境。一般是用 Docker 容器来封装一个包含浏览器终端文件系统的独立环境。对于网页浏览，本地会托管一个网站的静态版本，使用或使用外部后台来模拟响应 Agent 的请求。评估任务的话，通常会以一个高层次的目标的形式给出，不是给具体的操作步骤和命令。这样可以涵盖要测试的 Agent 的综合的能力，比如说信息检索工具的使用、推理规划和记忆等。这个任务通常也会附带一个明确的可程序化验证的成功标准。评估的话，就是你根据上面的设置一个评估标准，去整一个脚本，去来自动的检查这个沙箱环境的一些最终的任务状态。

##### 原题32：在评估一个 Agent 的任务完成情况时，除了最终结果的正确性，还有哪些过程指标是值得关注的？（例如：效率、成本、鲁棒性）

答：
过程指标建议至少看六项：步骤数、总时延、Token/费用、工具调用成功率、重试率、异常恢复时长。原因是很多系统“能做对但代价太高”，线上不可用。这些过程指标可直接指导优化方向，比如减少无效循环、提升重排质量或改进工具参数校验，比只看准确率更有工程价值。

**值得关注的关键过程指标包括：**

**1. 效率 (Efficiency):**

- 成本方向，你要考虑 token 的调用量以及工具 API 的调用费用。
- 步骤数，就是 Agent 在执行、思考、行动的循环总次数。通常更少的步骤完成任务被 Agent 认为规划能力更强。
- 计算的耗时，总耗时和计算的时间。

**2. 鲁棒性 (Robustness):**

- 错误处理能力, 就是当工具类返回错误或者网页加载失败，或者遇到其他预预期之外的情况时，A 智能能否识别并采取纠正措施，比如尝试不同的工具或者修正参数或者重新规划。错误与处理能力。
- 抗干扰能力，就是在环境中加入一些噪声或者误导性的信息来看，评估现在的成功率下降多少。

**3. 自主性(Autonomy & Alignment):**

- 一个需要人类干预的次数，需要人类干预次数越少，说明这智能更自主，需要人类帮助次数越少。
- 行为的可解释性，看Agent 的思考过程是否清晰和合乎逻辑。
- 计划的遵循程度，如果agent提前制定一个计划，看他在在执行过程当中，多大程度遵循了自己的计划。

##### 原题34：在进行人工评估时，如何设计合理的评估准则和流程，以保证评估结果的客观性和一致性？

答：
要先把维度原子化，比如准确性、完整性、安全性、简洁性，并给每档分数明确定义和示例。流程上做盲评、多评审、冲突仲裁和一致性统计（如 Kappa），避免主观漂移。对模型对比任务，成对比较通常比绝对打分更稳定。总结句：人工评估不是“凭感觉”，而是标准化实验流程。

##### 原题35：如何持续监控和评估一个已经部署上线的 LLM 应用或 Agent 服务的表现，以应对可能出现的性能衰退或行为漂移？

答：

一个完整的监控评估体系应包含以下几个层面：

1. **采集和监控（Automated Monitoring）：**

- 采集和监控指标，包括一次请求完整的所有的交互的数据，包括用户的输入、模型的中间思考过程、最终的输出以及调用的工具、延迟以及 token 的消耗等等。
- 很重要的一点就是用户的反馈，在产品的界面中嵌入一些用户反馈机制，比如说顶踩的按钮/打分和一些意见反馈等等。
- 还有一些间接指标: 输出指标中回答的长度、代码块的比例、JSON 格式错误率、拒绝回答率等，以及过程指标，agent的执行过程中的平均步数、工具调用频率、工具调用失败率这些。

**2. 审核与分析（Review and Analysis）：**

- **自动化质量评估：**一个是用程序定期的抽样，从生产流量中随机抽取一小部分的样本，来做评估。还有一个思路就是用一个更强大的裁判的大模型，根据一套评估的准则，以及是否有害，是否跑题，对抽样样本进行自动打分。还有就将抽样样本与内户维护的一个高质量的一个黄金评估机进行对比，看模型在这些关键问题上的表现是否稳定。
- **定期人工审计：** 定期组织运营或评估团队，对生产环境中的随机样本、用户反馈的坏案例、以及自动化监控发现的异常案例进行深入的人工分析。

**3. 反馈闭环与模型迭代（Feedback Loop and Model Iteration）：**

- 将生产环境中发现的有价值的案例，特别是失败案例和用户不喜欢的案例，清洗标注后，持续的加入到评估集和微调数据集中，定期的进行模型的再训练和微调，以适应新的数据分布和用户需求，然后再上线新版本的 时候，注意用 AB 测试框架，小流量验证新版本是否优于旧版本，确定每次迭代都是正向的。

通过建立这样一个“**采集/监控 -> 分析 -> 迭代**”的闭环，我们可以主动地管理和维护线上服务的质量，而不是被动地等待用户投诉。

##### 原题54：LangChain / LangGraph 项目上线后看哪些指标？

答：
我会分四类看：任务成功率（业务结果）、链路稳定性（错误率/重试率/中断率）、性能与成本（p95 时延、token 成本）、质量指标（幻觉率、格式通过率、人工接管率）。没有这套指标，就无法判断是模型问题还是编排问题。

追问：如果成功率下降，你会按什么顺序定位是 Prompt、检索、工具还是图编排的问题？

我按图编排→工具→检索→Prompt 的顺序做漏斗定位，先看编排流程收敛和可用性，再看语义质量，用固定回放集和分层指标做归因，

1. **图编排层（先查）**

看 `loop_ratestep_count_p95timeout_rateretry_rate`。

如果步数暴涨、回环增多、超时增多，优先判定是路由/终止条件问题。

1. **工具层**

看工具 `success_rate5xx/429schema校验失败率p95延迟`。

如果调用失败或慢，是工具可用性/契约问题。

1. **检索层**

看 `hit@k/recall@k`、空召回率、过滤后命中率、索引新鲜度。

如果召回质量掉了，答案成功率会同步下降，即使 Prompt 不变。

1. **Prompt 层（最后查）**

在“同一检索结果 + 同一工具返回”下做 Prompt A/B。

若前面都正常但输出格式、拒答边界、推理稳定性变差，才定位为 Prompt 回归。

**症状到根因的速判**

- 步数/超时突然上升：图编排
- 工具报错和慢调用上升：工具
- 空召回或错召回上升：检索
- 链路都健康但答案质量掉：Prompt

### 26. 原题：agent好坏是如何评价的？
**口述答案（约300字）**：
Agent评估我会分“效果、效率、稳定性、安全性”四个维度。效果看任务完成率、答案正确率、引用准确率；效率看首token时延、端到端时延、单任务token和外部API成本；稳定性看工具调用成功率、重试率、超时率、回环率；安全性看越权调用、敏感操作拦截率和审计完整性。评测方法上采用“离线集+线上流量”双轨：离线做可重复回归，线上看真实用户反馈与业务指标。最重要的是坏例回流机制，把失败case结构化记录并分类到Prompt、检索、工具、编排四层。面试里我会强调：评估不是单一准确率，而是以业务目标为核心的综合评分体系。

### 22. 原题：意图识别准确率大概能到多少？怎么算出来的？
**口述答案（约300字）**：
准确率我不会给拍脑袋数字，必须先定义口径。通常我会按离线和在线两套指标：离线看标注测试集上的Accuracy、F1、混淆矩阵；在线看业务口径的任务完成率、误路由率和人工接管率。比如“准确率95%”要明确是Top1还是TopK、是否包含低置信度拒答样本。计算流程是：先构建分层数据集（高频、长尾、对抗样本），再做固定版本评测；上线后通过埋点还原“预测意图-真实落地结果”，做按周回归。对面试官我会强调：单看准确率不够，要看错误代价。如果一个低频误判会触发高风险动作，就要提高阈值并强制二次确认，宁可多澄清一次也不能错执行业务。
